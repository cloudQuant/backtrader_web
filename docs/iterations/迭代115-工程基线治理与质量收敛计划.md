# 迭代115：工程基线治理与质量收敛计划

## 1. 迭代目标

本迭代用于收敛最近一轮项目分析后仍然存在的工程问题，目标不是新增业务功能，而是让“代码、文档、测试、仓库状态”重新保持一致。

本计划专门面向**初级工程师**设计，要求任务边界清晰、实现路径明确、每个故事都可以独立提交和验收。

## 2. 背景说明

截至 **2026-03-08**，项目已经完成了一轮基线治理，当前已确认的事实包括：

1. 本地开发基线已经统一为：
   - 前端开发端口：`3000`
   - 后端 API 端口：`8000`
   - 首页视图文件：`src/frontend/src/views/DashboardPage.vue`
2. 认证限流已真实接入认证接口，且 `request.state.user_id` 已开始参与日志上下文。
3. 前端单测和后端关键回归测试可以通过，但仍有测试噪音、文档漂移和仓库生成物污染问题没有彻底收口。

当前剩余问题主要分为四类：

1. 活跃文档仍有旧端口、旧页面名、旧访问地址。
2. 前端测试基建不统一，Element Plus stub 不完整，导致 warning 过多。
3. 仓库仍跟踪 coverage、E2E 报告等生成物，影响 diff、搜索和 Code Review。
4. 限流、请求上下文等新接入能力缺少更精确的回归测试。

## 3. 适用对象

- 工作年限 0 到 2 年的初级工程师
- 可以在一次 Code Review 指导下完成单个故事
- 能够独立运行 `pytest`、`npm run test`、`rg`、`git status`

## 4. 本次迭代不包含的内容

以下事项保留给后续由中高级工程师主导的专项重构，本次不要扩展：

1. 将回测/优化等长任务从 Web 进程内状态迁移到独立 worker / queue
2. 将 repository 自管理事务重构为 request-scoped session 或 unit-of-work
3. 引入 Alembic 并改造数据库初始化流程
4. 重写整套权限、审计或部署架构

## 5. 开发前准备

开始任何故事前，先保证本地环境与 CI 基线一致：

1. Node 使用 `20.x`
2. Python 使用 `3.10.x`
3. 前端目录：`src/frontend/`
4. 后端目录：`src/backend/`

建议先执行：

```bash
cat .nvmrc
cd src/frontend && npm ci
cd src/backend && pip install -e ".[dev,backtrader]"
```

## 6. 交付物

本迭代完成后，仓库中应至少新增或更新以下成果：

1. 一组与当前开发基线一致的活跃文档
2. 一套更稳定的前端测试辅助设施
3. 一套防止生成物再次入库的忽略规则与 CI 守卫
4. 一组覆盖认证限流与请求上下文的回归测试

## 7. 故事拆解

### Story 115.1：修正文档与运行基线漂移

**目标**

把当前仍然面向开发者或用户的“活跃文档”统一到真实运行基线，避免新同学按旧端口、旧页面名或旧 API 地址启动项目。

**优先修改文件**

1. `docs/INSTALLATION.md`
2. `docs/API.md`
3. `docs/TECHNICAL_DOCS.md`
4. `docs/USER_GUIDE.md`
5. `docs/TROUBLESHOOTING.md`
6. `docs/INDEX.md`

**只读排查，不要在本故事中改写的文件**

1. `docs/iterations/*.md`
2. `docs/AGILE_DEVELOPMENT.md`
3. `docs/source-tree-analysis.md`
4. `docs/project-scan-report.json`

这些文件带有明显历史快照属性，可以记录旧事实，但不能继续被当作当前开发入口文档。

**实现步骤**

1. 搜索活跃文档中的旧端口和旧页面名：

```bash
rg -n "localhost:5173|localhost:8001|Dashboard\\.vue" docs/INSTALLATION.md docs/API.md docs/TECHNICAL_DOCS.md docs/USER_GUIDE.md docs/TROUBLESHOOTING.md docs/INDEX.md
```

2. 将前端访问地址统一为 `http://localhost:3000`
3. 将后端 API/Swagger 地址统一为 `http://localhost:8000`
4. 将页面文件名描述统一为 `DashboardPage.vue`
5. 如果文档中提到默认管理员账号密码，明确标注“仅开发环境默认值，不可作为生产环境配置建议”
6. 更新 `docs/INDEX.md` 中的文档入口说明，避免继续指向过期事实

**验收标准**

1. 活跃文档中不再出现 `localhost:5173`
2. 活跃文档中不再出现 `localhost:8001`
3. 活跃文档中不再出现 `Dashboard.vue`
4. 文档对开发环境默认账号密码的表述明确区分“开发默认值”和“生产配置要求”

**自测命令**

```bash
rg -n "localhost:5173|localhost:8001|Dashboard\\.vue" docs/INSTALLATION.md docs/API.md docs/TECHNICAL_DOCS.md docs/USER_GUIDE.md docs/TROUBLESHOOTING.md docs/INDEX.md
```

预期结果：无输出。

**预计工时**

0.5 到 1 人天

### Story 115.2：收敛前端测试噪音，补齐共享测试基建

**目标**

让前端测试不仅“能通过”，还要“输出干净、便于继续扩展”。本故事重点是把重复 stub 和常见 Vue warning 收敛到统一测试基建中。

**重点文件**

1. `src/frontend/src/test/stubs.ts`
2. `src/frontend/src/test/setup.ts`
3. 新增 `src/frontend/src/test/mountWithPlugins.ts` 或同级共享辅助文件
4. `src/frontend/src/test/views/LoginPage.test.ts`
5. `src/frontend/src/test/views/RegisterPage.test.ts`
6. `src/frontend/src/test/views/SettingsPage.test.ts`
7. `src/frontend/src/test/views/Dashboard.test.ts`
8. `src/frontend/src/test/components/common/AppLayout.test.ts`

**已知典型问题**

1. `el-checkbox` 等组件未统一 stub
2. `v-loading` 指令未统一 mock
3. 局部测试文件自己拼装一套 stubs，重复度高
4. 某些 stub 的 prop 类型过于随意，容易触发 `Invalid prop` warning

**实现步骤**

1. 扩充共享 `elStubs`，优先补齐以下常见组件：
   - `el-checkbox`
   - `el-checkbox-group`
   - `el-avatar`
   - `el-container`
   - `el-aside`
   - `el-header`
   - `el-main`
2. 在全局测试初始化中补一个统一的 `v-loading` 指令 mock
3. 抽出共享挂载辅助函数，统一注入：
   - `Pinia`
   - `Router`
   - `elStubs`
   - 常用全局 mock
4. 选 3 到 5 个噪音最高的测试文件，改造成使用共享挂载工具
5. 删除这些测试文件中重复、可被共享工具替代的本地 stub
6. 保留真正与业务断言有关的 mock，不要为“减少 warning”而删除有效断言

**验收标准**

1. `npm run test -- --run` 通过
2. 运行测试时，不再出现以下已知 warning：
   - `Failed to resolve component: el-checkbox`
   - `Failed to resolve directive: loading`
   - `Invalid prop: type check failed for prop "size"`
3. 新增页面测试时，可以直接复用共享挂载工具，而不是每个文件重新写一套 `global.stubs`

**自测命令**

```bash
cd src/frontend
npm run test -- --run
```

如需检查 warning，可将输出重定向到临时文件后再搜索关键字。

**预计工时**

1 到 1.5 人天

### Story 115.3：清理仓库生成物并增加防回流机制

**目标**

将测试报告、coverage 报告、E2E 产物从主分支开发流程中剥离，减少脏 diff、误提交和搜索噪音。

**重点文件**

1. `.gitignore`
2. `.github/workflows/ci.yml`
3. 可新增 `scripts/check-generated-artifacts.sh`

**优先清理的生成物路径**

1. `src/backend/coverage/`
2. `src/backend/coverage.json`
3. `src/frontend/coverage/`
4. `src/frontend/e2e-results/`
5. `src/frontend/playwright-report/`
6. `src/frontend/test-results/`

**实现步骤**

1. 先确认上面路径是否全部属于可再生成内容
2. 在 `.gitignore` 中补齐缺失的忽略规则
3. 将已经被 Git 跟踪的生成物从索引中移除
4. 增加一个脚本或 CI 步骤，检查上述路径是否再次被提交
5. 将该检查接入 `ci.yml`，建议放在测试任务之前，尽早失败

**建议命令**

```bash
git ls-files | rg "^(src/backend/coverage/|src/backend/coverage\\.json|src/frontend/coverage/|src/frontend/e2e-results/|src/frontend/playwright-report/|src/frontend/test-results/)"
```

如果需要从 Git 索引移除但保留本地文件，可使用：

```bash
git rm -r --cached src/backend/coverage src/frontend/coverage src/frontend/e2e-results src/frontend/playwright-report src/frontend/test-results
git rm --cached src/backend/coverage.json
```

**验收标准**

1. `.gitignore` 覆盖上述生成物路径
2. `git ls-files` 不再列出这些生成物
3. CI 中存在一条明确的“生成物守卫”检查
4. 以后再次误提交这些目录时，CI 能直接失败并给出可读错误

**预计工时**

0.5 到 1 人天

### Story 115.4：补齐认证限流和请求上下文回归测试

**目标**

把最近新增的认证限流和 `request.state.user_id` 行为补齐成明确的自动化回归测试，避免后续重构把这些行为悄悄破坏。

**重点文件**

1. `src/backend/tests/test_rate_limiting.py`
2. `src/backend/tests/test_deps.py`
3. `src/backend/tests/test_logging_middleware.py`
4. `src/backend/tests/conftest.py`

**实现步骤**

1. 在 `test_rate_limiting.py` 中新增“真正打到限流阈值”的测试：
   - 注册接口：验证超过阈值后返回 `429`
   - 登录接口：验证超过阈值后返回 `429`
2. 为避免测试互相污染，继续复用当前已存在的 `limiter.reset()` 机制
3. 在 `test_deps.py` 中新增单元测试：
   - `get_current_user()` 成功时会把 `request.state.user_id` 写入请求上下文
   - `get_current_user_optional()` 成功时也会写入相同字段
4. 在 `test_logging_middleware.py` 中补一个轻量回归测试，验证日志上下文能够读取到 `user_id`
5. 保持测试粒度清晰：
   - 限流行为测 API
   - `request.state` 写入测依赖函数
   - 日志字段测中间件

**验收标准**

1. 至少一个认证写接口和一个认证读/登录接口存在真实 `429` 回归测试
2. `request.state.user_id` 写入行为有直接断言
3. 日志中间件相关测试覆盖 `user_id` 字段透传
4. 相关测试可以单独运行并稳定通过

**自测命令**

```bash
cd src/backend
pytest -q tests/test_rate_limiting.py tests/test_deps.py tests/test_logging_middleware.py
```

**预计工时**

0.5 到 1 人天

## 8. 推荐执行顺序

建议按下面顺序推进，每个故事单独一个 PR：

1. `Story 115.1` 文档对齐
2. `Story 115.2` 前端测试基建
3. `Story 115.4` 后端回归测试
4. `Story 115.3` 仓库生成物清理与 CI 守卫

这样安排的原因：

1. 文档修正风险最小，适合作为热身任务
2. 前端测试基建完成后，后续页面测试会更省力
3. 后端测试补齐是纯增量工作，适合在中间完成
4. 仓库清理会造成较多文件变动，放到最后更容易评审

## 9. PR 规范

每个故事独立提交，不要把四个故事混成一个 PR。

**分支命名建议**

1. `iteration115/story-115-1-doc-baseline`
2. `iteration115/story-115-2-frontend-test-foundation`
3. `iteration115/story-115-3-generated-artifact-guard`
4. `iteration115/story-115-4-auth-regression-tests`

**提交信息建议**

1. `docs(iteration115): align active docs with current dev baseline`
2. `test(frontend): consolidate shared stubs and mount helpers`
3. `ci(repo): block generated artifacts from being committed`
4. `test(auth): add regression coverage for rate limit and request context`

## 10. 交付检查清单

提交 PR 前，逐项确认：

- [ ] 只修改了当前故事需要的文件
- [ ] 本地运行了该故事要求的自测命令
- [ ] PR 描述中写明了“改了什么、怎么验收、还有什么风险”
- [ ] 没有顺手引入超出本次范围的大重构
- [ ] 没有修改历史迭代文档来伪造“当前事实”

## 11. 评审人重点关注项

Code Review 时，评审人重点看下面四件事：

1. 文档是否只修“活跃文档”，而不是把历史记录重写成当前事实
2. 测试基建是否真正减少了重复代码，而不是简单复制更多 stub
3. 生成物清理是否只移除可再生成文件，没有误删源代码或手工文档
4. 限流测试是否稳定，不依赖测试执行顺序

## 12. 迭代完成定义

当以下条件同时满足时，迭代115可视为完成：

1. 四个故事全部合并
2. 活跃文档与当前运行基线一致
3. 前端测试输出明显更干净，新增页面测试有统一挂载入口
4. 生成物不再进入版本库，CI 能阻止回流
5. 认证限流和请求上下文有明确的自动化回归保护

## 13. 总工时预估

总计约 **3 到 4.5 人天**，适合拆成一周内完成的短迭代。

建议由 1 名初级工程师负责实施，1 名中高级工程师负责 Review 和必要的边界把控。
