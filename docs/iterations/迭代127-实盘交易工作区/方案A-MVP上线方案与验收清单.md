# 迭代127 - 方案A：MVP上线方案与验收清单

> **文档状态**: 产品版  
> **最后更新**: 2026-04-13  
> **适用目标**: 首批上线 / 业务验收 / 演示评审

## 1. 文档目标

本方案用于定义迭代127的**首批上线版本（MVP）**，确保“策略交易工作区”在不牺牲实盘安全和回退能力的前提下，完成统一入口的首次交付。

本方案不追求一次性替换全部旧功能，而是围绕以下原则组织首批交付：

- **先闭环** — 先打通“建工作区、建单元、启停交易、看状态”的核心路径
- **先人工、后自动** — 首批上线以人工可控为主，自动交易不作为上线门槛
- **先灰度、后替换** — 新旧入口并存，待稳定后再淘汰旧页面
- **先安全、后效率** — 实盘操作优先保证校验、确认和可回退

---

## 2. MVP定位

### 2.1 产品目标

将现有“模拟交易”和“实盘交易”两个分散入口，整合为统一的“策略交易”工作区入口，并复用策略研究的工作区架构，让用户能够在同一工作区内管理交易型策略单元。

### 2.2 首批上线要解决的核心问题

- 用户能否在统一入口中创建交易工作区
- 用户能否把研究中的策略单元顺畅迁移到交易场景
- 用户能否在同一列表中管理模拟单元和实盘单元
- 用户能否在交易中稳定看到关键运行状态
- 用户能否在实盘操作前获得足够的安全保护

### 2.3 本次上线成功定义

满足以下条件即可视为 MVP 达标：

- 新“策略交易”入口可正常访问
- 可以创建 `trading` 类型工作区
- 可以新建或导入交易单元
- 可以手动启动/停止模拟和实盘单元
- 可以查看状态、仓位、盈亏、价格等关键字段
- 实盘启动具备网关校验和二次确认
- 页面刷新后可恢复可识别状态
- 旧入口保留，可作为回退路径

## 3. MVP范围

### 3.1 本次必须交付

- 新增侧边栏“策略交易”入口
- 基于工作区架构支持 `workspace_type='trading'`
- 策略单元支持 `trading_mode`、`gateway_config`
- 交易工作区列表页和详情页
- 交易型单元创建、编辑、删除、导入、导出
- 模拟单元/实盘单元的手动启动和停止
- 交易运行状态展示
- 实盘启动前的网关校验与二次确认
- 研究工作区与交易工作区之间的导入导出互通

### 3.2 本次明确不作为上线门槛

- 自动交易调度
- 锁定交易 / 锁定运行 / 解锁
- 头寸管理器独立对话框
- WebSocket 实时推送
- 旧模拟交易/实盘交易页面删除
- 全量工具栏按钮一次性平铺上线

### 3.3 工具栏交付策略

首批上线采用“高频直出、低频收纳”的交付方式：

#### P0 直出按钮

- 全选
- 启动策略单元
- 停止策略单元
- 新建策略单元
- 删除策略单元
- 导入策略单元
- 导出策略单元
- 数据源设置
- 策略单元设置
- 公式应用设置

#### P1/P2 收纳按钮

- 启动自动交易
- 关闭自动交易
- 锁定交易
- 锁定运行
- 解锁
- 头寸管理器
- 打开K线
- 打开组合报告
- 新建优化任务
- 定时优化设置
- 交易日统计选项
- 联动分组

## 4. 核心用户路径

### 4.1 路径一：创建交易工作区

1. 用户进入“策略交易”入口
2. 创建工作区
3. 系统按交易类型保存为 `trading` 工作区
4. 用户进入详情页开始配置单元

**验收重点：**

- 页面样式与策略研究一致
- 列表只展示交易工作区
- 不影响原策略研究工作区

### 4.2 路径二：研究转交易

1. 用户在策略研究中导出策略单元
2. 用户在策略交易中导入单元
3. 系统补齐交易默认字段
4. 用户补充交易模式、网关等必要配置
5. 用户准备启动交易

**验收重点：**

- 研究导出的 JSON 可直接导入
- 不需要人工修改文件格式
- 不兼容字段有明确处理策略

### 4.3 路径三：人工启动交易

1. 用户选择一个或多个策略单元
2. 用户点击启动
3. 若存在实盘单元，系统先执行网关检查
4. 系统弹出二次确认
5. 启动成功后进入运行状态

**验收重点：**

- 模拟单元与实盘单元必须清晰区分
- 实盘启动必须强提醒
- 失败时有明确反馈

### 4.4 路径四：盘中状态查看

1. 用户在工作区列表中查看多个单元
2. 页面按轮询刷新核心状态
3. 用户可快速识别运行中、已停止、异常状态
4. 用户可根据状态执行停止或排查动作

**验收重点：**

- 关键字段稳定刷新
- 列过多时布局仍可用
- 刷新页面后可恢复主要状态

## 5. 产品规则与安全护栏

### 5.1 实盘操作护栏

- 实盘单元必须配置网关预设
- 实盘单元在列表中必须有明显视觉标识
- 启动实盘前必须二次确认
- 网关不可达或参数缺失时，禁止启动
- 首批上线不支持默认静默启动实盘

### 5.2 状态展示护栏

- 运行状态优先级高于扩展信息
- 异常状态必须在列表中直接可见
- 若实时数据不可达，至少展示最近快照时间
- 不允许“页面看起来正常但实际不可交易”的模糊状态

### 5.3 灰度与回退护栏

- 首批上线保留旧“模拟交易”“实盘交易”入口
- 新入口建议带 Beta 标识
- 新入口出现阻断问题时，可以临时回退到旧入口
- 旧入口删除必须在新入口验收通过之后执行

## 6. 验收清单

## 6.1 页面与导航验收

- [ ] 侧边栏出现“策略交易”入口
- [ ] 新入口可进入交易工作区列表页
- [ ] 页面主视觉风格与策略研究一致
- [ ] 旧模拟交易/实盘交易入口仍可访问

## 6.2 数据与模型验收

- [ ] 可创建 `trading` 类型工作区
- [ ] 可创建模拟单元
- [ ] 可创建实盘单元
- [ ] 实盘单元未配置网关时不可启动
- [ ] 研究工作区数据不受影响

## 6.3 核心交易闭环验收

- [ ] 模拟单元可手动启动
- [ ] 模拟单元可手动停止
- [ ] 实盘单元可手动启动
- [ ] 实盘单元可手动停止
- [ ] 状态从 `idle -> running -> idle` 正常切换
- [ ] 运行中可看到头寸、盈亏、最新价等关键字段
- [ ] 页面刷新后仍能识别主要运行状态

## 6.4 研究互通验收

- [ ] 策略研究导出文件可导入策略交易
- [ ] 导入后系统自动补齐默认交易字段
- [ ] 策略交易导出后可回流策略研究，且不会导致导入失败

## 6.5 安全与异常验收

- [ ] 启动实盘时出现明确确认弹窗
- [ ] 网关校验失败时给出明确错误提示
- [ ] 运行状态拉取失败时有可理解提示
- [ ] 快照过期或状态异常时有可识别提示

## 6.6 回归验收

- [ ] 策略研究工作区列表和详情功能正常
- [ ] 策略研究中的单元管理正常
- [ ] 原有导入导出能力正常
- [ ] 原有模拟交易/实盘交易页面不被本次改动破坏

## 7. 上线与发布策略

### 7.1 发布建议

采用“灰度可回退”方式发布：

- 第一步：上线新“策略交易”入口
- 第二步：保留旧入口作为业务回退通道
- 第三步：邀请核心用户试用新入口
- 第四步：完成验收后，再决定是否进入旧入口替换阶段

### 7.2 发布前检查项

- 后端迁移脚本可执行
- 前端路由与菜单可访问
- 模拟/实盘单元启停链路已联调
- 至少完成一轮研究→交易导入演示
- 至少完成一轮实盘安全校验演示

### 7.3 回退策略

若出现以下情况，建议临时回退到旧入口：

- 新入口无法创建或加载交易工作区
- 启停交易存在阻断性异常
- 实盘安全校验异常导致误启动风险
- 交易状态展示大量失真

回退方式：

- 保留旧菜单入口作为临时主入口
- 暂停推广新入口
- 修复后再恢复灰度验证

## 8. 演示脚本建议

### 8.1 标准演示流程

1. 打开“策略交易”入口
2. 创建一个交易工作区
3. 从策略研究导入一个策略单元
4. 新建一个模拟单元和一个实盘单元
5. 对实盘单元补齐网关配置
6. 启动模拟单元，查看状态变化
7. 启动实盘单元，演示校验和确认
8. 展示运行状态字段
9. 停止单元，展示状态回落
10. 刷新页面，展示快照恢复

### 8.2 演示通过标准

- 不出现明显流程中断
- 不需要人工修改导入文件结构
- 实盘启动的安全链路完整可见
- 核心字段展示清晰，产品逻辑可讲通

---

## 9. 结论

方案A的核心不是“做完所有功能”，而是确保迭代127可以以**更低风险、更高确定性**上线。

对于本项目，最正确的首批交付目标是：

**先交付一个可用、可演示、可回退、可保护实盘安全的策略交易工作区 MVP。**
