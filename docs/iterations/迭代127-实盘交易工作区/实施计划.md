# 迭代127 - 策略交易工作区 - 实施计划

> **文档状态**: 优化版  
> **最后更新**: 2026-04-13

## 1. 文档目标

本计划基于初始需求，结合当前项目架构（迭代124工作区体系、现有模拟/实盘交易服务），从**产品交付、上线风险、用户可用性**三个视角对原方案进行优化，形成一版更适合研发执行和阶段验收的迭代方案。核心策略仍然是**最大化复用策略研究代码**，通过 `workspace_type` 扩展实现策略交易功能，而非从零构建。

### 技术经理修订摘要

结合代码基线审查，迭代127按以下技术口径落地：

1. **统一工作区壳，拆分交易单元页签实现**：列表页、详情页、创建工作区弹窗、公共设置弹窗继续复用；交易单元表格单独实现 `TradingWorkspaceUnitsTab`，避免把研究工作区 990 行表格继续做条件分支，降低回归风险。
2. **统一工作区 API，运行时复用现有交易 manager**：不新增独立 `TradingRuntimeManager`，而是在 `workspace_service.py` 中按 `workspace_type` 分流，交易工作区复用现有 `live_trading_manager.py` 做启停和状态查询。
3. **P0 只做人工可控闭环**：首批只交付“创建工作区/创建单元/手动启动停止/查看状态与头寸”，自动交易、WebSocket、旧页面下线全部留在后续阶段。
4. **实盘与模拟共用一条启动链路**：`trading_mode=paper/live` 只影响参数注入和网关校验，`paper` 模式显式关闭网关推断，`live` 模式必须有网关预设。
5. **研究运行时与交易运行时彻底隔离**：交易单元 CRUD 不再同步研究工作区的本地回测运行目录，避免交易单元依赖 `strategies/<strategy_id>/strategy_*.py` 的研究模板结构。
6. **旧入口保留为兜底回退路径**：`SimulatePage` / `LiveTradingPage` 在本迭代不删除，仅新增“策略交易”入口并完成主链路验证。

### 本计划重点解决的问题

1. **从“功能堆叠”改为“闭环交付”** — 围绕“创建工作区 → 导入/新建单元 → 手动启停 → 查看状态”构建 MVP
2. **从“技术分解”改为“用户路径分解”** — 先保证核心交易路径可用，再叠加自动化和增强能力
3. **从“一步替换”改为“灰度替换”** — 旧模拟交易/实盘交易页面不在 P0 删除，待新入口验证稳定后再下线
4. **从“功能全上”改为“风险分层”** — 自动交易、旧页面清理、WebSocket 等高风险事项不进入首批上线门槛
5. **补充业务验收口径** — 明确业务成功标准、安全护栏、回滚策略和阶段退出条件

## 2. 实施原则

### 2.1 复用优先

- Workspace/StrategyUnit 模型扩展而非新建
- 工作区列表/详情/单元管理 UI 组件直接复用，通过 props 差异化
- 数据源/单元设置/策略参数/优化配置等对话框完全复用
- 导入导出格式兼容策略研究，实现互通

### 2.2 先闭环再丰富

先实现核心交易链路：
- 创建交易工作区
- 添加策略单元（指定交易模式 + 网关）
- 启动/停止交易
- 实时查看头寸/盈亏/状态

再逐步补充：
- 自动交易调度
- 锁定交易/运行
- 头寸管理器
- 交易日统计
- WebSocket 实时推送

### 2.3 向下兼容

- 现有策略研究功能完全不受影响
- Workspace 表新增字段有默认值
- 旧模拟/实盘页面在验证完成前保留

### 2.4 渐进式替换

- 阶段1-4：新旧并存，侧边栏同时保留
- 阶段5：验证通过后，删除旧页面和路由

### 2.5 产品成功定义

本迭代完成后，用户至少应能稳定完成以下动作：

- 在统一的“策略交易”入口中创建交易工作区
- 从策略研究导入策略单元，或直接新建策略单元
- 明确区分模拟交易和实盘交易单元，并完成手动启动/停止
- 在同一列表中持续查看状态、头寸、盈亏、最新价等核心信息
- 在页面刷新或短时异常后，仍能恢复到可识别的运行状态

同时，本迭代**不以“功能数量最多”为成功标准**，而以以下标准为主：

- 核心交易路径可演示、可测试、可回归
- 实盘操作具备必要的确认和防误触保护
- 新旧能力可并存，出现问题时可以快速回退到旧入口
- 策略研究与策略交易之间的导入导出链路可用

### 3.4 本迭代优先保障的核心用户场景

#### 场景A：研究转交易

1. 用户在策略研究中导出策略单元
2. 在策略交易工作区中导入
3. 系统自动补齐交易默认字段
4. 用户完成必要配置后启动交易

这是最关键的业务连接场景，优先级高于高级配置能力。

#### 场景B：盘中人工盯盘

1. 用户进入交易工作区
2. 查看多个单元的运行状态、持仓、盈亏和价格变化
3. 手动启动或停止部分策略单元
4. 遇到异常时能快速识别是模拟单元还是实盘单元

这是首批上线最主要的使用场景，也是表格字段设计和工具栏设计的依据。

#### 场景C：实盘谨慎操作

1. 用户选择实盘单元执行启动
2. 系统校验网关预设和连接状态
3. 对实盘启动进行二次确认
4. 启动成功后在列表中清晰展示“实盘中”状态

这是产品侧必须补强的安全场景，优先级不低于自动交易。

## 4. 范围拆分

### 4.1 P0：本迭代必须交付

- 工作区模型扩展（workspace_type='trading'）
- 策略单元交易扩展（trading_mode, gateway_config）
- 策略交易工作区列表页（复用 WorkspaceListPage）
- 策略交易工作区详情页（复用 WorkspaceDetailPage）
- 策略单元管理（新增 TradingWorkspaceUnitsTab，复用原工作区壳与公共弹窗）
- 创建单元时选择交易模式和网关
- 启动/停止策略单元交易
- 实时交易状态展示（头寸/盈亏/价格/杠杆等全部列）
- 实盘启动前的网关校验与二次确认
- 侧边栏新增"策略交易"入口（建议带 Beta 标识或保留旧入口）
- 策略研究↔策略交易 JSON 导入导出互通

> **P0 定义优化**：P0 只承诺交付“人工可控的交易工作区闭环”，不把自动交易、旧页面删除、WebSocket 替换等高风险事项纳入首批上线门槛。

> **技术补充口径**：P0 的“实时状态”以轮询 + `trading_snapshot` 持久化为准，不把推送式更新作为上线前置条件。

### 4.2 P1：本迭代建议交付

- 自动交易调度配置（迁入工作区）
- 锁定交易/锁定运行/解锁
- 头寸管理器对话框
- 交易日汇总数据
- WebSocket 实时推送（替代轮询）
- 工具栏高级能力归位（打开K线、组合报告、优化入口等）

> **当前补充说明**：本轮已将“启动自动交易 / 关闭自动交易 / 自动交易配置”接入策略交易工作区工具栏，底层仍复用现有 `auto_trading` 调度服务，因此它属于增强交付，不改变 P0 的上线门槛定义。

> **当前落地状态补充**：交易工作区已补齐头寸管理器、交易日统计选项、定时优化设置、联动分组入口；旧 `simulate/live-trading` 前端入口已下线，保留兼容跳转到统一的 `/trading` 路径。

### 4.3 P2：可顺延到后续迭代

- 旧页面删除（SimulatePage/LiveTradingPage 等）
- 旧路由删除与菜单完全替换
- 定时优化设置
- 交易日统计选项对话框
- 联动分组
- 历史交易对账
- 多账户资金管理
- 风控规则引擎

### 4.4 范围裁剪原则

- **凡是会直接影响实盘安全的能力，必须先做保护措施，再做批量化和自动化**
- **凡是现有旧页面可继续承担的能力，不强制在本迭代一次性迁完**
- **凡是主要用于增强体验而不影响闭环的能力，默认降到 P1/P2**
- **顶部工具栏以“可用优先”而非“一次摆满全部按钮”为原则**，P0 先保证常用按钮直出，其余高级能力可收纳到“更多操作”

## 5. 分阶段实施计划

### 阶段0：产品校准与上线口径冻结（0.5 天）

#### 目标

冻结 MVP 范围、关键安全约束和灰度策略，避免研发过程中范围回涨。

#### 必做任务

1. 确认 P0 只覆盖“人工可控闭环”，自动交易降级为 P1
2. 确认顶部工具栏分层：常用动作直出，高级动作允许收纳
3. 确认实盘操作的产品护栏：模式标识、网关校验、二次确认
4. 确认旧页面保留策略与下线条件
5. 确认研究→交易导入是优先路径之一

#### 输出物

- P0/P1/P2 清单冻结
- 首批上线验收清单
- 灰度发布与回滚原则

#### 验收标准

- 产品、研发、测试对首批交付边界无歧义
- 没有高风险能力混入 P0

### 阶段3：手动交易闭环（核心里程碑，3-4 天）

#### 目标

打通本迭代最重要的首批可上线闭环：手动启动、手动停止、稳定查看状态。

#### 后端任务

1. **新建 `app/services/trading_runtime_manager.py`** — 内存交易运行时状态管理器
2. **新建 `app/services/trading_workspace_service.py`** — 交易工作区编排服务
3. **扩展 `app/api/workspace_api.py`** — 新增交易路由：
   - `POST /workspace/{id}/trading/start`
   - `POST /workspace/{id}/trading/stop`
   - `GET /workspace/{id}/trading/status`
4. **集成回调** — LiveTradingService/PaperTradingService 的回调更新 TradingRuntimeManager
5. **定期快照** — 后台任务每 5 分钟持久化 `trading_snapshot`
6. **实盘安全校验** — 启动前校验网关预设、连接状态和必要参数

#### 前端任务

7. **扩展 `WorkspaceUnitsTab.vue`** — 支持交易专属列和手动启停动作
8. **扩展 `stores/workspace.ts`** — 新增 `fetchTradingStatus()`, `startTrading()`, `stopTrading()` 等 action
9. **交易状态轮询** — 在交易工作区以 2 秒间隔轮询状态
10. **实盘二次确认** — 启动实盘单元时给出明确确认

#### 验收标准

- 用户可以手动启动模拟单元
- 用户可以手动启动实盘单元，且启动前有必要校验与确认
- 状态从 `idle → running → idle` 闭环正确
- 运行中可稳定看到头寸、盈亏、最新价等核心字段
- 页面刷新后能通过快照恢复到可识别状态

> **阶段3 完成即达到首批上线门槛。** 若时间或稳定性不足，不阻塞后续 P1 能力顺延。

### 阶段4：互通体验与盘中效率增强（2 天）

#### 目标

补齐用户从策略研究迁移到策略交易的体验，并提升盘中使用效率。

#### 后端/前端任务

1. **导入导出兼容** — 确保策略研究导出的 JSON 可导入策略交易，自动补齐默认交易字段
2. **反向兼容** — 策略交易导出后可回流到策略研究，交易字段按兼容规则忽略
3. **工具栏整理** — 常用按钮保留直出，高级动作归并到“更多操作”
4. **状态字段优化** — 明确状态、详情、交易日等列的展示优先级与默认显隐
5. **异常提示** — 网关失败、状态同步失败、快照过期等错误提示更明确

#### 验收标准

- 研究→交易导入成功率可达演示级稳定
- 工具栏不会因为按钮过多而严重影响主流程使用
- 常见失败场景能得到可理解的反馈

## 8. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LiveTradingService/PaperTradingService 集成复杂度高 | 启停逻辑出错 | 先跑通单个单元启停，再做批量；阶段3完成即可形成 MVP |
| 实盘误触发启动 | 真实资金风险 | 实盘模式显著标识 + 启动前校验 + 二次确认 |
| TradingRuntimeManager 内存状态丢失（进程重启） | 状态恢复延迟 | 定期持久化快照 + 启动时从 snapshot 恢复 |
| 交易列太多导致表格横向溢出 | UI体验差 | 采用固定列、横向滚动、默认显隐和工具栏分层 |
| 顶部按钮过多影响可用性 | 学习成本高、误操作率高 | P0 聚焦高频动作，其余收纳到“更多操作” |
| 策略研究功能回归问题 | 影响现有用户 | workspace_type 默认 'research'，阶段1完成后立即回归测试 |
| 自动交易调度跨时区问题 | 启停时间不准 | 降为 P1，统一使用服务器本地时间并明确时区说明 |
| 网关连接超时/失败 | 实盘启动失败 | 启动前校验网关可达性，失败给出明确错误 |
| 旧页面过早删除 | 一旦新入口不稳定缺乏回退 | 旧页面顺延到阶段6 删除，采用灰度替换 |

## 10. 建议的研发执行顺序

1. 先完成字段和基础接口扩展（阶段1）
2. 再搭好新入口和工作区主流程（阶段2）
3. 再打通手动交易闭环（阶段3）— **核心里程碑**
4. 接着补策略研究互通、状态体验和工具栏整理（阶段4）
5. 最后按稳定性决定是否继续推进自动交易增强（阶段5）和旧入口下线（阶段6）

每个阶段结束后系统都处于**可运行、可验证**状态。

### 10.1 首批上线退出条件

满足以下条件即可进入首批上线/验收：

- 新“策略交易”入口可用
- 可创建交易工作区并完成至少一条“研究→交易”导入链路
- 可手动启动/停止模拟与实盘单元
- 实盘启动具备必要的确认和失败反馈
- 页面刷新后仍可识别主要运行状态
- 策略研究主流程回归通过

## 11. 结论

迭代127定义为：

**以“策略交易工作区”统一模拟交易和实盘交易，最大化复用策略研究架构，优先交付人工可控、可灰度上线的交易工作区闭环。**

优化后的设计核心是**先闭环、后增强，先灰度、后替换**：

- 先交付统一入口、手动启停、状态监控、研究互通这四个核心能力
- 自动交易、锁定、头寸管理、旧页面删除放在后续阶段推进
- 保证实盘安全护栏和回退路径优先于功能完备度

旧模拟/实盘页面继续采用**渐进式替换**策略，新功能经过业务验收和稳定性验证后再安全下线。
