# 迭代119：认证事务边界与测试信号清洁计划

## 1. 迭代目标

本迭代聚焦两个高收益但尚未完全收口的问题：

1. 让认证链路中的多步写操作真正落在同一事务边界内，避免 refresh token 轮换和修改密码出现“半成功”状态
2. 清理当前测试输出中的已知 warning 和噪音，让测试结果更适合作为持续验收信号

本迭代不追求大规模架构翻新，而是在当前基线已经恢复可运行的前提下，继续提升代码一致性和质量闸门的可信度。

## 2. 背景

截至 **2026-03-08**，项目已经完成迭代118的主要收口工作：

1. 后端入口、环境校验、数据库初始化路径和仓库生成物治理已经恢复到可验收状态
2. `session_provider.py` 和 `sql_repository.py` 已可用，但真正接入同一事务边界的业务链路还偏少
3. 认证 refresh token 相关逻辑仍存在多次独立 commit 的历史包袱
4. 测试虽然能通过，但仍存在一些典型噪音：
   - `pytest-asyncio` fixture scope deprecation warning
   - 默认管理员密码触发的重复 Pydantic warning
   - 前端图表测试触发的 Vue prop warning / jsdom navigation warning

这些问题不会立即阻断开发，但会持续降低回归测试的信号质量，也会让后续事务重构缺少可信的起点。

## 3. 本次迭代范围

### 包含

1. 认证服务中的 refresh token 轮换事务收口
2. 修改密码与 token 全量失效的事务收口
3. 对应的回滚测试
4. 后端与前端测试 warning 清理
5. 迭代文档和索引更新

### 不包含

2. 将回测长任务完全迁出 Web 进程
3. 引入 Alembic 迁移体系
4. 重做前端测试框架或 E2E 架构

## 4. 故事拆解

### Story 119.1：让认证 refresh token 链路真正具备事务一致性

**问题**

当前认证服务中的以下流程本质上是多步写操作：

1. `login_with_refresh`
2. `refresh_tokens`
3. `change_password`
4. `revoke_all_user_tokens`

如果这些步骤分散在多个独立 commit 中执行，一旦中途抛异常，就可能留下部分状态已经落库、部分状态没有落库的中间态。

**目标**

1. `refresh_tokens` 在同一事务里完成“校验旧 token -> 撤销旧 token -> 写入新 token”
2. `change_password` 在同一事务里完成“校验旧密码 -> 修改密码哈希 -> 撤销全部 refresh token”
3. 对意外异常补回滚测试，证明不会留下半成功状态

**涉及文件**

1. `src/backend/app/services/auth_service.py`
2. `src/backend/tests/test_refresh_token.py`
3. 如有必要，补充 `docs/REQUEST_SCOPED_SESSION.md`

**验收标准**

1. refresh token 轮换失败时，旧 token 不会被错误标记为 revoked
2. 修改密码过程中撤销 token 失败时，密码更新也会回滚
3. 现有 refresh token / auth 测试保持通过

---

### Story 119.2：把测试结果从“能通过”推进到“输出干净”

**问题**

当前测试输出里仍残留一些已知噪音：

1. `pytest-asyncio` 关于 `asyncio_default_fixture_loop_scope` 的 deprecation warning
2. 测试环境默认管理员密码触发的重复 warning
3. 前端图表测试中对组件契约和下载行为 mock 不完整，导致额外 warning

**目标**

1. 后端 pytest 默认不再输出无关的 fixture loop scope 警告
2. 通用测试环境不再因为默认管理员密码产生批量 warning
3. 前端图表测试在符合组件契约的前提下运行，不再制造无意义 stderr

**涉及文件**

1. `src/backend/pytest.ini`
2. `src/backend/tests/conftest.py`
3. `src/frontend/src/test/components/charts/KlineChart.test.ts`
4. `src/frontend/src/test/components/charts/TradeSignalChart.test.ts`

**验收标准**

1. 相关测试运行时 warning 数量明显下降
2. 测试断言仍覆盖原先的关键行为，而不是通过删测试来“消音”

---

### Story 119.3：为回测执行器抽离做准备

**问题**

`BacktestExecutionManager` 已经收口了状态持久化，但 `BacktestService` 仍然直接承担执行入口和进程内取消控制。这一层还缺少一个明确的执行器边界。

**目标**

1. 在文档和代码职责上明确区分“状态管理器”和“执行器”
2. 为后续把执行器迁到独立 runner / worker 做准备

**本迭代预期**

本故事先做梳理，不强制在 119 内完成全部代码拆分。
