## Phase 1: 安全加固 (P0 - 紧急)

### 任务 1.1: 修复沙箱安全漏洞

**文件**: `src/backend/app/utils/sandbox.py`

**问题**: 当前沙箱使用字符串匹配检测危险代码，存在多种绕过方式：
- `importlib.import_module()` 绕过 import 检测
- `getattr(obj, 'eval')` 绕过函数名检测
- `dict(globals())` / `vars()` 绕过 `globals()[` 检测
- `__init__` 等合法双下划线代码被误杀

**修复方案**:
- [ ] 将字符串匹配替换为 AST（抽象语法树）解析，使用 `ast.NodeVisitor` 遍历代码节点
- [ ] 添加执行超时机制（`signal.alarm` 或 `threading.Timer`）
- [ ] 建立 AST 白名单机制：只允许预定义的 `ast.Import`、`ast.ImportFrom` 目标模块
- [ ] 添加 `importlib` 调用检测
- [ ] 为沙箱执行添加内存限制（通过 `resource` 模块）

### 任务 1.2: 实盘交易凭证加密

**文件**: `src/backend/app/services/live_trading_service.py`

**问题**: API key 和 secret 以明文存储在内存 `self.tasks` 字典中（第 63 行、165-175 行）

**修复方案**:
- [ ] 引入运行时凭证加密：使用 `cryptography.fernet` 对敏感字段加密存储
- [ ] 添加凭证 TTL 过期机制，使用后及时清理
- [ ] 在日志中对凭证信息脱敏（替换为 `***`）
- [ ] 添加凭证字段的 `__repr__` 覆盖，防止意外打印

### 任务 1.3: 修复日志敏感信息泄露

**文件**: `src/backend/app/middleware/logging.py`

**问题**: 请求查询参数直接记录，可能泄露 token/密码（第 89-90 行）

**修复方案**:
- [ ] 添加敏感参数过滤列表：`token`, `password`, `secret`, `key`, `api_key`
- [ ] 过滤 `Authorization`, `Cookie` 等敏感请求头
- [ ] 实现 `X-Forwarded-For` 头解析以获取真实客户端 IP（第 72 行）

## Phase 2: 线程安全与资源管理 (P1)

### 任务 2.1: 修复实盘交易服务线程安全

**文件**: `src/backend/app/services/live_trading_service.py`

**问题**: `self.tasks` 和 `self.cerebro_instances` 字典在多线程间共享但无锁保护（第 53-54 行）

**修复方案**:
- [ ] 使用 `threading.Lock()` 保护共享字典的读写
- [ ] 添加任务生命周期管理，任务完成/失败后自动从字典中清除
- [ ] 实现优雅关闭机制：`shutdown()` 方法停止所有运行中的交易并清理资源
- [ ] 考虑将任务状态持久化到数据库，支持多实例部署

### 任务 2.2: 修复优化服务竞态条件

**文件**: `src/backend/app/services/param_optimization_service.py`

**问题**: 任务状态的检查-更新操作不是原子的（第 482 行 check-then-act 模式）

**修复方案**:
- [ ] 使用数据库乐观锁（version 字段）保证状态更新的原子性
- [ ] 限制 `ProcessPoolExecutor` 池大小（当前无限制，第 332 行）
- [ ] 添加 Optuna study 的显式清理逻辑

### 任务 2.3: 前端资源泄漏修复

**文件**: `src/frontend/src/composables/useBacktestRuntime.ts`

**问题**:
- 心跳定时器在 WebSocket 异常关闭时可能未清理（第 112-117 行）

**修复方案**:
- [ ] 使用 `AbortController` 为轮询请求提供取消能力
- [ ] 在 `onUnmounted` 中确保所有定时器和连接彻底清理
- [ ] 添加 WebSocket 重连退避机制

## 执行顺序与依赖关系

```
Phase 1 (安全) ──→ Phase 2 (线程安全) ──→ Phase 3 (复杂度)
                                            ↓
Phase 4 (测试) ←──────────────────────────────
     ↓
Phase 5 (性能) ──→ Phase 6 (质量) ──→ Phase 7 (验证)
```

**关键原则**:
- Phase 1 安全修复必须最先完成
