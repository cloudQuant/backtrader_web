# 迭代124-测试可信度、遗留链路清理与基线同步计划

## 1. 背景

`backtrader_web` 在迭代123及其子计划之后，已经完成了一轮较重要的结构化收口：

- `123-A` 明确了任务执行模型与 CI 闸门的收口方向。
- `123-B` 已推动 `LiveTradingManager` 的内部职责拆分。
- `123-C` 已对实时链路和占位能力做了显式降级与边界澄清。

这意味着，项目当前最值得继续推进的内容，已经不再是“再写一份大而全的优化建议”，而是把**123 之后仍残留的真实缺口**继续收紧，尤其是那些会影响：

- 测试结果是否可信
- 维护者是否容易误判真实主链路
- 文档与项目事实是否一致
- 多进程/多实例语义是否清晰

本轮计划聚焦的是**后123阶段的残留收口**，重点不在扩新功能，而在于提升项目的可信度、可维护性和后续迭代效率。

## 3. 当前最值得优化的方向

### 3.1 P0：E2E 测试仍存在“通过但未真正验证”的静默成功问题

虽然项目已有 `docs/E2E_TEST_QUALITY_REVIEW.md` 对 E2E 现状做过审查，但当前 `tests/e2e/` 中仍能看到大量类似模式：

- `if locator.count() > 0:` 后才执行核心断言
- 依赖页面已有历史数据或既有实体，否则测试主体不会真正执行
- 某些测试即使没有验证关键元素，也可能直接通过

这类问题当前仍广泛存在于多个文件，例如：

- `tests/e2e/test_backtest_result.py`
- `tests/e2e/test_data_query.py`
- `tests/e2e/test_optimization.py`
- `tests/e2e/test_strategy.py`
- `tests/e2e/test_strategy_crud.py`

这会导致两个直接风险：

- CI 显示 E2E 通过，但关键路径实际上没有被验证。
- 后续重构时，团队会高估前端回归保护的强度。

**结论**：E2E 可信度应成为当前第一优先级改进项之一。

---

### 3.2 P0：实盘领域仍保留旧链路与新链路并存的遗留实现

当前主路由已经走 `app.api.live_trading_api` + `LiveTradingManager` 体系，但仓库中仍保留两套旧版实盘 API 文件：

- `src/backend/app/api/live_trading.py`
- `src/backend/app/api/live_trading_complete.py`

这两份文件都基于 `LiveTradingService` 的旧模型，并且接口形态、任务语义、依赖对象与当前主链路并不一致。与此同时：

- `src/backend/app/services/live_trading_service.py` 仍完整保留。
- `src/backend/app/services/monitoring_service.py` 仍直接实例化 `LiveTradingService()`。

这会带来典型的维护风险：

- 新维护者难以判断哪条才是当前主链路。
- 测试、文档、监控和真实运行语义之间可能逐步漂移。
- 后续继续优化实盘能力时，容易误改到已退场链路。

**结论**：需要明确“主链路 / 兼容链路 / 遗留链路”的分类，并对遗留实现做显式处置。

---

### 3.3 P0/P1：任务执行模型虽然已有收口，但仍未彻底摆脱进程内状态依赖

`123-A` 已经为任务模型收口建立方向，但从当前代码看，仍存在几个没有彻底解决的点：

- `backtest_service.py` 仍明确依赖 `BacktestExecutionRunner` 持有当前 API 进程内的执行句柄。
- `cancel_task()` 对运行中任务的取消，仍以“本进程持有本地句柄”为前提。
- `param_optimization_service.py` 仍保留全局 `_tasks` 与 `_tasks_lock`，同时混用 `ProcessPoolExecutor`、数据库状态和内存状态。
- `optimization_execution_manager.py` 虽然已落库，但 `create_task()` 初始化时直接把状态写成 `RUNNING`，与“先创建任务、再进入运行”的状态语义仍可进一步统一。

也就是说，当前状态并不是“完全没有进展”，而是处于：

- 持久化真相源已经存在
- 但进程内状态仍未退居为纯辅助缓存

**结论**：任务执行模型仍需做最后一轮“语义与实现边界”的收口，而不是继续长期维持混合模型。

---

### 3.4 P1：监控与告警服务仍是明显的单进程内存模型

`src/backend/app/services/monitoring_service.py` 当前保留如下实现特征：

- `_monitoring_tasks` 为进程内 `asyncio.Task` 注册表
- `_trigger_state` 为进程内状态
- 代码中明确备注：生产环境更适合使用持久化调度器

这说明监控告警能力当前的真实语义更接近：

- 单进程可用
- 多实例和重启恢复不可靠
- 运行状态不可跨进程协同

如果这部分能力仍被文档或产品界面表述为“完整告警系统”，就会形成新的能力错配。

**结论**：监控服务应被纳入“运行时模型收口”范围，至少明确其支持边界与演进策略。

---

### 3.5 P1：项目快照、索引与 API 事实仍有明显漂移

`docs/project-scan-report.json` 当前仍存在多个与现状不一致的问题，例如：

- `project_root` 指向旧目录 `/Users/yunjinqi/Documents/量化交易框架/backtrader_web`
- `api_inventory.live_trading` 未反映当前 gateway preset / health / manual connect 等接口
- `iteration_notes` 数量已过时

同时：

- `docs/iterations/README.md` 还停留在 123
- `docs/INDEX.md` 的迭代历史描述也还停留在 123

这类漂移会直接影响：

- 后续 AI 代理的项目理解
- 新成员的接手成本
- 文档引用与规划判断的准确性

**结论**：文档与项目事实同步仍然是应持续推进的主题，而且现在已经进入“需要重新生成基线”的阶段。

---

### 3.6 P1/P2：项目能力声明需要从“功能存在”升级为“能力边界真实可解释”

当前很多历史文档、README 或项目概述仍采用“模块已完成”的表达方式，例如：

- 实盘交易：完成
- 实时数据：完成
- 监控告警：完成

但从当前实现细看，项目更准确的状态应是：

- 主链路已具备可用版本
- 某些能力已明确降级
- 某些能力仍受单进程模型或遗留实现影响

如果项目继续沿用“全部完成”式叙述，而不建立能力矩阵，就容易造成误解。

**结论**：建议把“模块完成度”改为“能力矩阵 + 限制说明”的表达方式。

## 4. 迭代124目标

本轮建议聚焦以下四个目标：

- **目标1**：提升 E2E 测试的真实性与可回归性，减少静默通过。
- **目标2**：清理实盘领域遗留链路，明确唯一主链路与兼容/废弃边界。
- **目标3**：继续收紧任务执行与监控运行时模型，减少进程内状态成为事实源。
- **目标4**：重建项目基线快照与能力矩阵，让文档与代码事实同步。

## 5. 建议工作包

## 工作包A：E2E 测试可信度提升

### 目标

把当前“可能通过但没有真正验证”的前端 E2E 测试，提升为可解释、可复现、可回归的测试资产。

### 建议内容

- 盘点 `tests/e2e/` 中所有 `if count() > 0` / 条件性断言模式。
- 将“无数据时静默通过”改为以下两类之一：
  - 使用 fixture/API 预创建数据，确保断言路径真实执行。
  - 显式 `skip` 并说明原因，而不是隐式跳过。
- 为关键页面建立最小数据准备夹具：
  - 回测结果页
  - 优化页
  - 实盘页
  - 策略 CRUD 页
- 提升关键路径断言密度，避免只用 `len(content) > 100` 之类弱断言。

### 交付物

- E2E 问题模式清单
- 关键路径数据 fixture
- 改造后的 E2E 测试集
- 更新后的 E2E 质量复盘文档

### 验收标准

- 关键路径测试不再依赖已有历史数据才能成立。
- 测试跳过是显式可见的，而非静默通过。
- 关键页面至少具备 1 条真正执行主断言的稳定测试路径。

---

## 工作包B：实盘遗留链路识别与清理

### 目标

明确当前实盘领域的唯一主链路，并处置旧版 API / 旧版 service 残留。

### 建议内容

- 明确以下对象的状态分类：
  - 当前主链路
  - 兼容保留链路
  - 已遗留待下线链路
- 重点梳理：
  - `app/api/live_trading_api.py`
  - `app/api/live_trading.py`
  - `app/api/live_trading_complete.py`
  - `app/services/live_trading_manager.py`
  - `app/services/live_trading_service.py`
- 评估 `monitoring_service.py` 对 `LiveTradingService` 的依赖是否仍合理。
- 对不再使用的旧链路进行以下处理之一：
  - 从主可见入口移除
  - 标记为 legacy compatibility
  - 明确废弃计划与下线时点

### 交付物

- 实盘主链路/遗留链路对照表
- 遗留模块处置决策
- 必要的兼容说明或删除计划

### 验收标准

- 维护者能够清晰回答“当前真实主链路是哪条”。
- 不再存在 API 文件和 service 文件长期保留却职责身份不明的情况。
- 监控、测试、文档引用的实盘模型与真实主链路一致。

---

## 工作包C：任务与监控运行时模型继续收口

### 目标

把任务执行和监控运行的“持久化事实源”与“进程内运行时缓存”边界进一步说清并收口。

### 建议内容

- 对回测任务继续收紧：
  - 明确本地执行句柄与持久化状态的责任边界
  - 明确取消失败时的语义与返回
- 对参数优化任务继续收紧：
  - 评估 `_tasks` 还能否进一步退场或弱化
  - 收敛 DB 状态与运行时状态更新顺序
  - 审视 `create_task()` 初始状态是否应为 `pending`
- 对监控服务：
  - 明确其单进程语义
  - 决定是继续保持单进程模型并显式文档化，还是开始抽象持久化调度边界

### 交付物

- 任务与监控运行时模型说明
- 回测/优化/监控的边界对照表
- 对应契约测试或行为测试补充

### 验收标准

- 进程内状态不再被误认为唯一事实源。
- 回测/优化/监控三类能力的运行时边界有明确说明。
- 至少一个遗留混合语义点被真正收口。

## 8. 实施级任务清单

### T1：盘点 E2E 中的静默通过模式

- **目标**：列出所有依赖 `if count() > 0` 等条件性断言的测试。
- **输出**：E2E 风险清单，按页面和严重度分类。
- **完成定义**：可以明确回答“哪些 E2E 当前即使不验证主路径也会通过”。

### T2：为关键页面建立可复现的数据 fixture

- **目标**：减少 E2E 对历史数据或人工预置状态的依赖。
- **输出**：回测、优化、策略、实盘相关的最小数据准备夹具。
- **完成定义**：关键页面至少有一条不依赖历史库数据的稳定测试路径。

### T3：提高关键 E2E 断言质量

- **目标**：将弱断言和静默跳过改为显式、真实、有业务意义的验证。
- **输出**：关键 E2E 用例改造。
- **完成定义**：关键测试不再只验证“页面不为空”。

### T4：梳理实盘主链路与遗留链路边界

- **目标**：确认当前实盘 API 与 service 的真实主链路。
- **输出**：主链路/遗留链路映射表。
- **完成定义**：维护者能清楚知道哪些文件仍在主用，哪些只是遗留。

### T5：处置旧版实盘 API 与 `LiveTradingService`

- **目标**：结束旧版 live trading API / service 的悬置状态。
- **输出**：保留、废弃、迁移或下线决策。
- **完成定义**：遗留链路的身份不再模糊。

### T6：评估并收口监控服务的运行时边界

- **目标**：明确 `MonitoringService` 的单进程限制与后续演进边界。
- **输出**：监控运行时模型说明与必要代码调整。
- **完成定义**：监控能力的真实边界可以被清楚描述。

### T7：继续收口回测与优化任务状态源

- **目标**：减少 `_tasks`、本地 task runner 与 DB 真相源之间的混合语义。
- **输出**：任务状态边界收口方案与测试。
- **完成定义**：至少一个关键混合状态点被消除或显式文档化。

### T8：更新项目快照、索引和能力矩阵

- **目标**：让文档重新反映当前真实状态。
- **输出**：更新后的索引、快照、能力说明。
- **完成定义**：项目快照、迭代入口、API inventory 与当前代码一致。

### 推荐执行顺序

`T1 -> T2 -> T3 -> T4 -> T5 -> T6/T7 -> T8`

其中：

- **先让测试可信，再优化内部结构**
- **先明确主链路，再清理遗留实现**
- **先收口运行时边界，再更新能力声明**

## 10. 主要风险与控制措施

### 风险1：E2E 改造后短期内失败数上升

**控制措施**：

- 允许先暴露问题，再逐步补数据 fixture 与断言。
- 把“原来静默通过”的失败识别为质量信号，而不是单纯噪音。

### 风险2：遗留实盘链路清理误伤历史测试或兼容路径

**控制措施**：

- 先建立主链路/遗留链路映射表，再决定下线或保留。
- 若短期无法删除，至少要显式标注 legacy 身份。

### 风险3：任务与监控模型收口范围失控
