# 迭代129：当前项目执行面收口与体验一致性优化计划

## 1. 背景

本轮分析基于以下真实材料交叉核对后得出：

- `docs/ARCHITECTURE.md`
- `project-context.md`
- `docs/iterations/迭代126-当前项目优化审查与执行计划.md`
- `docs/iterations/迭代127-运行时契约统一与文档对齐计划.md`
- `docs/iterations/迭代128-当前项目优化候选清单与待确认执行计划.md`
- `src/backend/app/api/backtest_enhanced.py`
- `src/backend/app/websocket_manager.py`
- `src/backend/app/db/database.py`
- `src/backend/app/db/cache.py`
- `src/backend/tests/test_optimization_api.py`
- `src/frontend/src/views/BacktestPage.vue`
- `src/frontend/src/composables/useBacktestRuntime.ts`
- `src/frontend/src/stores/backtest.ts`
- `src/frontend/src/api/index.ts`

结论是：

当前项目已经不是“功能缺失型”问题，核心回测、参数优化、策略管理、模拟/实盘相关能力都已有比较完整的实现骨架。当前更值得投入的方向，已经从“继续铺功能”转向“收口执行面、统一权威入口、降低前端编排复杂度、补齐运行时测试与文档真相源”。

换句话说，项目下一阶段的主要收益，不在于再增加多少新页面，而在于让现有主链路更稳定、更可维护、更容易验证。

## 2. 当前状态判断

### 2.1 已具备较好基础的部分

1. 回测后端主链路已经具备较完整的任务化能力。
   - `backtest_enhanced.py` 已包含提交、状态查询、结果查询、取消、导出、WebSocket 推送等关键接口。

2. 参数优化后端测试基础明显强于此前阶段。
   - `src/backend/tests/test_optimization_api.py` 已对参数获取、任务提交、进度、结果、取消、路由存在性等做了较系统覆盖。

3. 数据库与缓存层已有基本运行基线。
   - `database.py` 已提供 schema/bootstrap 入口。
   - `cache.py` 已提供内存缓存和可选 Redis 缓存的双实现。

4. 前端已经把回测运行时从页面中抽出一层。
   - `useBacktestRuntime.ts` 已承担 WebSocket、轮询回退、终态收口等逻辑，说明项目已经开始向职责分层靠拢。

### 2.2 仍然值得优先处理的问题

| 问题 | 证据 | 影响 | 优先级 |
|------|------|------|------|
| 回测运行时权威契约仍未完全统一 | `backtest_enhanced.py` 既承担 REST 任务入口，也承担 WebSocket 运行时协议；`useBacktestRuntime.ts` 中还存在鉴权子协议、心跳、轮询回退等多套运行时分支 | 运行时边界不够清晰，联调、测试、文档同步成本高 | P0 |
| 参数优化入口仍存在双轨面 | `backtest_enhanced.py` 中保留了 deprecated 的优化入口，并继续桥接到 `optimization_api` | 外部接口面和内部权威实现不一致，容易继续形成兼容债务 | P0 |
| 回测页面编排职责仍然偏重 | `BacktestPage.vue` 同时负责策略切换、动态参数、任务提交、运行时状态、结果列表、消息提示 | 页面维护成本高，局部修改容易牵动整页 | P1 |
| 前端错误提示链路存在不一致 | `src/frontend/src/api/index.ts` 的响应拦截器已经统一弹错；`BacktestPage.vue` 中 `showRequestMessage()` 对 Axios 错误直接 `return`，导致局部 fallback/warning 语义丢失 | 用户提示可能重复，也可能静默，排障体验不稳定 | P1 |
| 测试与文档对“运行时真相”的覆盖仍不足 | 当前可见强测试主要集中在参数优化 API；尚未看到对 WebSocket 鉴权子协议、轮询回退、前端运行时终态切换的成体系验证 | 回归风险高，重构时缺少可信护栏 | P1 |
| 启动与缓存的可观测性仍偏弱 | `database.py` 的启动行为由环境开关决定，`cache.py` 的内存缓存缺少命中率/容量暴露 | 运行时排障主要依赖人工判断，难以量化性能和初始化状态 | P2 |

## 3. 本轮建议不优先推进的事项

以下事项目前不建议排在本轮最前面：

1. 强制拆分为更多独立服务。
   - 当前主要矛盾仍是单体内部职责边界与权威入口收口，不是部署形态本身。

2. 强制把 Redis 设为默认必选依赖。
   - 现阶段更需要先把缓存使用边界和观测能力补齐，而不是扩大基础设施依赖。

3. 大规模重做前端 UI 组件体系。
   - 当前前端主要问题是编排职责和错误处理一致性，不是视觉组件能力不足。

4. 继续扩大兼容层而不做权威入口治理。
   - 继续保留并扩展 deprecated 路径，会进一步放大未来清理成本。

## 4. 迭代目标

本轮迭代建议聚焦以下四个结果：

1. 明确回测与参数优化的权威运行时入口。
2. 降低 `BacktestPage.vue` 的编排复杂度。
3. 统一前端错误提示和运行时状态收口方式。
4. 补齐围绕运行时真相的测试与文档闭环。

## 5. 建议工作包

### WP1：回测运行时权威契约统一（P0）

#### 目标

把“提交任务、监听进度、获取终态、处理失败/取消、鉴权握手、轮询回退”的权威定义统一下来，避免前后端各自隐式维护一套运行时约定。

#### 范围

1. 明确回测任务的唯一权威入口集合：提交、状态、结果、取消、WebSocket。
2. 固化 WebSocket 事件模型。
   - 至少明确 `connected`、`task_created`、`progress`、`completed`、`failed`、`cancelled`、`pong` 的字段要求。
3. 明确 WebSocket 鉴权握手策略。
   - 当前前端通过子协议传 token，后端据此鉴权；需要把该约束从“代码事实”升级为“文档事实 + 类型事实”。
4. 明确轮询回退何时启动、何时停止、超时如何提示。
5. 后端补一份运行时契约文档或在现有文档中明确权威段落。

#### 交付物

- 一份权威运行时契约说明
- 前端与后端共享的事件字段约束
- 去歧义后的回测任务调用说明

#### 验收标准

- 文档中能明确回答“回测主链路的唯一入口是什么”。
- 前端不再需要依赖页面级隐式知识推断 WebSocket 事件语义。
- WebSocket 与轮询回退的行为边界可以被测试直接验证。

### WP2：参数优化能力收口与兼容层治理（P0）

#### 目标

把参数优化从“新旧双轨并存”推进到“单一权威入口 + 明确兼容策略”。

#### 范围

1. 明确 `optimization_api` 是否作为唯一权威优化 API。
2. 评估并收口 `backtest_enhanced.py` 中 deprecated 优化入口的处理方式：
   - 保留但仅做兼容转发，并给出移除时间窗；或
   - 在下一阶段直接移除并同步文档。
3. 统一优化任务的提交、进度、结果、取消语义。
4. 统一优化结果模型命名和文档入口，避免“同一能力多个说明口径”。

#### 交付物

- 参数优化权威 API 决策
- 兼容层保留/移除策略
- 文档中的唯一推荐调用路径

#### 验收标准

- 新接入方只需要看一份优化 API 文档即可完成接入。
- deprecated 入口不再被当作正式能力继续扩展。
- 后端测试命名、文档命名、实际接口面保持一致。

### WP3：前端回测页解耦与错误体验统一（P1）

#### 目标

将 `BacktestPage.vue` 从“超大编排页”收敛为“页面容器 + 明确职责组件/组合式函数”，同时把错误提示职责统一起来。

#### 范围

1. 重新划分页面职责，至少拆清以下边界：
   - 策略与参数表单
   - 回测任务运行时状态
   - 回测结果展示
   - 历史结果列表
2. 明确谁拥有 `loading`、`currentTaskId`、`progressInfo` 的最终状态控制权。
3. 修正 `showRequestMessage()` 与全局 Axios 响应拦截器的职责冲突。
4. 统一 warning/error 的展示规则，避免部分错误被全局兜底、部分错误由局部处理、部分错误直接沉默。

#### 交付物

- 更清晰的回测页分层结构
- 单一可信的消息提示策略
- 更容易单测的 composable / component 边界

#### 验收标准

- 页面文件主要承担组装职责，不再混合过多运行时细节。
- Axios 错误、业务失败、回退超时三类提示路径可以清晰区分。
- 局部 warning 场景不再被“检测到 Axios 错误后直接返回”吞掉。

### WP4：运行时测试与文档真相源补齐（P1）

#### 目标

把当前“代码里大概能跑”提升为“关键运行时行为可以被重复验证”。

#### 范围

1. 后端补充围绕回测运行时的测试：
   - WebSocket 鉴权失败
   - 正常连接与消息流
   - 失败/取消终态
   - 轮询回退依赖的状态接口行为
2. 前端补充 `useBacktestRuntime.ts` 的单测或组件级测试：
   - WebSocket 正常完成
   - WebSocket 失败后切换轮询
   - 超时提示
   - 取消流程
3. 增补至少一条 E2E 主链路：
   - 提交回测
   - 观察进度
   - 终态收口
4. 同步更新以下文档中的运行时真相：
   - `docs/ARCHITECTURE.md`
   - `docs/BACKTEST_GUIDE.md`
   - `docs/OPTIMIZATION.md`
   - `project-context.md`
   - 必要时包括 `docs/DATABASE_INIT.md`

#### 交付物

- 运行时核心测试集
- 更新后的权威文档
- 可以作为回归护栏的最小验收路径

#### 验收标准

- 至少一条回测主链路能通过自动化验证覆盖提交到完成全过程。
- 文档能够准确解释当前真实行为，而不是停留在历史设计描述。
- 新人仅根据文档就能判断主链路该从哪里接入、如何排查。

### WP5：启动基线与缓存可观测性补强（P2）

#### 目标

把数据库初始化和缓存状态从“黑盒能用”提升为“可解释、可观测、可排查”。

#### 范围

1. 明确数据库启动模式矩阵：
   - 仅校验连接
   - 自动建表
   - 默认管理员自举
2. 增加启动日志或状态暴露，帮助区分“未执行”“已跳过”“已成功初始化”。
3. 为缓存层补充基本观测点：
   - 命中/未命中
   - 当前条目数
   - 清理触发情况
4. 补充运维文档，明确何时建议启用 Redis。

#### 交付物

- 启动模式说明
- 基础缓存观测指标
- 运维排查说明

#### 验收标准

- 运维侧能明确判断启动时是否执行过 schema/bootstrap。
- 缓存是否有效、是否膨胀、是否需要 Redis，不再完全依赖猜测。

## 6. 建议执行顺序

建议按以下顺序推进，而不是并行大面积展开：

### 第一阶段：先收权威入口

1. WP1 回测运行时权威契约统一
2. WP2 参数优化能力收口与兼容层治理

原因：这两项决定后续前端拆分和文档修正应该围绕哪个真相源展开。

### 第二阶段：再降前端复杂度

3. WP3 前端回测页解耦与错误体验统一

原因：如果运行时契约还没收口，前端拆分很容易建立在不稳定接口语义之上。

### 第三阶段：补测试与文档闭环

4. WP4 运行时测试与文档真相源补齐

原因：这一阶段用于固化前两个阶段的结果，防止回归。

### 第四阶段：补可观测性与运维基线

5. WP5 启动基线与缓存可观测性补强

原因：这是中期收益项，重要但不应阻塞主链路收口。

## 7. 风险与控制点

### 风险1：兼容层收口影响现有调用方

控制建议：

- 对 deprecated 入口设置明确过渡期。
- 文档中明确“推荐入口”和“兼容入口”的区分。
- 在移除前补最小兼容测试。

### 风险2：前端消息提示治理时出现重复弹窗或静默

控制建议：

- 明确“全局响应拦截器”和“页面局部提示”的边界。
- 先梳理错误分类，再落具体代码改造。

### 风险3：运行时测试补齐成本高于预期

控制建议：

- 先覆盖终态切换和失败分支，不追求一次性覆盖全部中间态。
- 优先建立一条可信主路径，再逐步补边界用例。

### 风险4：文档更新再次滞后于代码

控制建议：

- 将“文档同步更新”纳入每个工作包的完成定义。
- 对权威入口、运行时事件、兼容层状态设置固定文档章节。

## 8. 完成定义

本轮迭代在以下条件同时满足时，可认为达成目标：

1. 回测和参数优化都有清晰的权威入口说明。
2. Deprecated 入口不再继续承担正式能力演进。
3. `BacktestPage.vue` 不再承担过重编排职责。
4. 前端错误提示行为具备一致规则，不再存在明显静默或重复提示路径。
5. 回测主链路至少有一条自动化验证路径覆盖提交到终态。
6. 架构文档、使用文档、项目上下文与真实实现保持一致。

## 9. 推荐的落地选择

如果下一步只打算启动一个工作包，建议优先选择：

1. **优先级最高**：WP1 回测运行时权威契约统一
2. **紧随其后**：WP2 参数优化能力收口与兼容层治理
3. **实现切入点最好**：WP3 前端回测页解耦与错误体验统一

推荐原因：

- WP1 决定主链路真相源。
- WP2 决定是否继续累积兼容债务。
- WP3 则能最快改善开发体验和后续维护成本。
