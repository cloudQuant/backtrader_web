# 前端编排评估报告

## 1. 当前架构

| 模块 | 文件 | 行数 | 职责 |
|------|------|------|------|
| Store | `stores/backtest.ts` | ~60 | REST API 数据获取、状态管理 |
| Composable | `composables/useBacktestRuntime.ts` | ~265 | WebSocket 运行时、轮询回退、进度跟踪 |
| Page | `views/BacktestPage.vue` | ~500 | UI 渲染、表单处理、用户交互 |

## 2. 职责边界分析

### Store 职责（清晰）
- `results[]`: 回测历史列表
- `currentResult`: 当前回测结果
- `loading`: 数据加载状态
- `runBacktest()`: 提交回测任务
- `fetchResult()`: 获取单个结果
- `fetchResults()`: 获取历史列表
- `deleteResult()`: 删除结果

### Composable 职责（清晰）
- `loading`: 运行时加载状态
- `currentTaskId`: 当前任务 ID
- `progressInfo`: 进度信息
- WebSocket 连接管理
- 轮询回退机制
- 心跳维护
- 终态检测与处理

### Page 职责（可接受）
- 策略选择表单
- 参数动态渲染
- 结果展示
- 历史列表
- 用户交互协调

## 3. 评估结论

**当前架构合理，无需大规模重构。**

### 优点
1. **职责分离清晰**: Store 管数据，Composable 管运行时，Page 管 UI
2. **Composable 可复用**: `useBacktestRuntime` 可被其他需要回测运行时的组件复用
3. **测试友好**: Composable 和 Store 可独立测试

### 潜在改进点
1. **表单逻辑**: 如果参数配置变得更复杂，可考虑抽离 `useBacktestForm`
2. **结果展示**: 如果结果展示组件膨胀，可拆分为独立组件

## 4. 建议

### 短期（无需行动）
- 保持当前架构
- 为 Composable 补充单元测试

### 中期（观察）
- 监控 Page 文件大小，超过 800 行时考虑拆分
- 如果参数配置逻辑复杂化，抽离 `useBacktestForm`

### 长期（可选）
- 考虑将结果展示拆分为 `BacktestResultPanel` 组件
- 考虑将历史列表拆分为 `BacktestHistoryList` 组件
