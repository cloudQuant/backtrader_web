# 迭代123-B-实盘与网关管理解耦计划

## 1. 目标

本子迭代聚焦 `src/backend/app/services/live_trading_manager.py` 的职责解耦，目标是降低核心服务复杂度、提高可测试性，并为后续接入更多 gateway / exchange 类型保留清晰扩展点。

---

## 2. 当前问题摘要

从当前实现看，`live_trading_manager.py` 已承担过多职责：

- 实盘实例 CRUD
- JSON 文件持久化
- 进程启停与 PID 存活检测
- 启动后的运行状态同步
- 手工 gateway 连接与断连
- gateway 健康信息汇总
- 预置网关参数与 preset 定义
- 实例与 gateway 关系维护

这会带来以下后果：

- 任何改动都容易影响多个功能面
- 文件过大，不利于阅读与评审
- 单元测试构造成本高
- 新增交易所、运行模式或连接方式时，复杂度会继续快速上升

---

## 3. 本子迭代范围

### 3.1 In Scope

- 对 `LiveTradingManager` 进行职责拆分设计
- 收敛实例持久化、进程监督、gateway 生命周期、preset 定义四类责任
- 梳理 JSON 存储与未来仓储抽象的演进边界
- 为关键能力补上更细粒度的测试入口

### 3.2 Out of Scope

- 不在本轮重做整个实盘 API
- 不在本轮替换所有 gateway runtime 实现
- 不在本轮切换到新的外部持久化基础设施
- 不在本轮引入新的交易所产品能力

---

## 4. 设计原则

- **优先拆职责，不先改对外接口**
- **先形成协调器 + 子服务结构，再迁移细节**
- **状态存储与运行控制分离**
- **gateway 生命周期与业务实例生命周期分离**
- **preset 定义从执行逻辑中剥离**

---

## 5. 建议目标结构

建议将现有 `LiveTradingManager` 拆成如下结构：

- **LiveTradingManager**
  - 仅作为协调器
  - 负责对外暴露统一入口
- **InstanceStore / InstanceRepository**
  - 负责实例读写
  - 屏蔽当前 JSON 文件细节
- **ProcessSupervisor**
  - 负责进程启动、停止、PID 存活探测、启动恢复
- **GatewayRegistry / GatewayLifecycleService**
  - 负责 gateway runtime 注册、连接、断连、健康状态查询
- **GatewayPresetService**
  - 负责 preset 与 editable_fields 定义

这样做的核心收益是：

- 主协调器更薄
- 子能力可以分别测试
- 后续迁移 JSON -> DB 时不必同时触碰进程控制逻辑

---

## 6. 工作包拆分

## 工作包B1：实例存储抽象

### 目标

把当前基于 JSON 的实例持久化细节从主管理器中抽离。

### 建议动作

- 收口 `_load_instances()` 与 `_save_instances()` 的使用入口
- 引入明确的存储接口或内部仓储类
- 保持当前 JSON 文件实现不变，但通过抽象隔离底层存储细节
- 明确实例结构的最小必需字段与可选字段

### 验收标准

- 主管理器不再直接散落读写 JSON 文件
- 后续迁移到 DB 时，只需替换存储层

---

## 工作包B2：进程监督职责拆分

### 目标

将实例运行状态与进程控制逻辑从业务协调中拆出。

### 建议动作

- 拆出 PID 存活检测、进程扫描、启动恢复、停止逻辑
- 统一“运行中 / 已停止 / 已丢失进程”判定规则
- 明确哪些状态来自实例记录，哪些来自进程探测
- 为启动恢复场景补充回归验证

### 验收标准

- 进程控制逻辑不再混杂在 gateway/preset/实例查询逻辑中
- 启动恢复与 PID 同步逻辑可独立测试

---

## 工作包B3：Gateway 生命周期与实例关系分离

### 目标

明确 gateway 是独立运行对象，实例只是消费者，而不是把所有状态都混在 manager 内部字典里。

### 建议动作

- 明确 gateway 的注册、连接、断连、引用计数、健康检查接口
- 将 `_gateways`、`_instance_gateways` 的访问收口到生命周期服务
- 对共享 gateway 与手工 gateway 分别定义状态语义
- 统一健康快照输出格式

### 验收标准

- gateway 运行状态与实例配置状态可分别理解和测试
- 新增 gateway 类型时，不必修改主协调器的大量分支

---

## 工作包B4：Preset 与配置元数据拆分

### 目标

将 preset 定义从执行管理逻辑中抽出，降低文件膨胀。

### 建议动作

- 把 `get_gateway_presets()` 等静态配置元数据迁移到独立服务或数据模块
- 统一 preset 的字段定义方式
- 让 UI 所依赖的 preset 输出结构更稳定
- 明确 preset 只是输入模板，不参与运行控制

### 验收标准

- 主管理器不再包含大量静态 preset 数据定义
- preset 变更不再需要进入运行控制模块修改

---

## 7. 存储演进建议

本轮不建议直接把 JSON 存储一次性迁移到数据库，但建议提前预留边界：

- 当前阶段：
  - 保留 JSON 文件存储
  - 增加抽象层
- 下一阶段：
  - 如果实例数量、运行节点数量或审计需求提升，再迁移到 DB

这样可以避免在本轮把“解耦”和“迁移存储”两个高风险动作叠加。

---

## 8. 风险与控制措施

### 风险1：拆分过程影响现有 API 行为

- **控制措施**：对外 API 不先改，只做内部职责迁移。

### 风险2：启动恢复逻辑在拆分后失真

- **控制措施**：先为 `_sync_status_on_boot` 与进程探测逻辑补行为测试，再迁移代码。

### 风险3：gateway 与实例关联关系在重构时被破坏

- **控制措施**：先建立关系模型与最小接口，再替换底层字典访问。

## 10. 建议实施顺序

1. 先抽实例存储层
2. 再拆进程监督逻辑
3. 再抽 gateway 生命周期服务
4. 最后迁移 preset 与静态配置定义
5. 收尾时精简主协调器

---

## 11. 实施级任务清单

### T1：梳理 `LiveTradingManager` 现有职责边界

- **目标**：列清当前 manager 内部所有责任区，明确哪些逻辑属于存储、进程、gateway、preset、实例编排。
- **输入**：`live_trading_manager.py` 全量实现。
- **输出**：职责映射表，标明函数/字段所属责任区。
- **依赖**：无。
- **完成定义**：能够明确指出每一段核心逻辑的归属与未来迁移目标。

### T2：抽离实例存储接口

- **目标**：把 `_load_instances()` 与 `_save_instances()` 从主管理器直连调用改成统一存储接口。
- **输入**：T1 职责映射表。
- **输出**：实例存储抽象与当前 JSON 实现的适配层。
- **依赖**：T1。
- **完成定义**：主管理器不再散落直接读写 JSON 文件。

### T3：定义实例数据模型与字段约束

- **目标**：明确实例记录的最小字段、可选字段和状态字段语义。
- **输入**：现有实例 JSON 结构与 API 返回结构。
- **输出**：实例字段约定文档或内部模型说明。
- **依赖**：T2。
- **完成定义**：后续迁移与测试都可以基于统一字段结构，不再靠隐式约定。

### T4：抽离进程监督逻辑

- **目标**：将 PID 探测、启动恢复、停止、外部进程扫描等逻辑从主管理器剥离。
- **输入**：T1 职责映射表、现有进程相关函数。
- **输出**：独立的进程监督组件或服务。
- **依赖**：T1。
- **完成定义**：进程控制逻辑不再与 gateway 生命周期、preset 输出混杂在同一层。

### T5：补齐进程监督行为测试

- **目标**：用测试锁定启动恢复、PID 存活判断、外部运行实例识别等关键路径。
- **输入**：T4 进程监督组件。
- **输出**：进程监督相关测试。
- **依赖**：T4。
- **完成定义**：进程状态同步与恢复行为有自动化回归保护。

### T6：抽离 gateway 生命周期服务

- **目标**：把 gateway 注册、连接、断连、引用计数、健康状态查询收口到独立服务。
- **输入**：`_gateways`、`_instance_gateways` 相关逻辑。
- **输出**：gateway lifecycle service 或等价抽象。
- **依赖**：T1。
- **完成定义**：新增 gateway 类型时，主协调器不再承担主要分支逻辑。

### T7：抽离 preset 与配置元数据

- **目标**：将 `get_gateway_presets()` 等静态元数据迁移到独立服务或数据模块。
- **输入**：现有 preset 定义。
- **输出**：preset service 或独立配置模块。
- **依赖**：T1。
- **完成定义**：preset 变更无需修改主运行控制逻辑。

### T8：收口主协调器职责

- **目标**：让 `LiveTradingManager` 回到统一入口与编排器角色。
- **输入**：T2、T4、T6、T7 的拆分结果。
- **输出**：精简后的主协调器。
- **依赖**：T2、T4、T6、T7。
- **完成定义**：主协调器主要负责调用子服务，而不是承载全部实现细节。

### T9：补充结构说明与迁移记录

- **目标**：让后续维护者理解新的模块结构与 JSON -> DB 的潜在迁移边界。
- **输入**：T8 最终结构。
- **输出**：模块职责说明、迁移说明、变更摘要。
- **依赖**：T8。
- **完成定义**：后续继续迭代时，不需要再次从超大文件反推设计意图。

### 推荐执行顺序

`T1 -> T2 -> T3 -> T4 -> T5 -> T6 -> T7 -> T8 -> T9`

其中：

- **先抽存储与进程，再动 gateway 生命周期**
- **先完成内部拆分，再统一主协调器**
