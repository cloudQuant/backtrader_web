# 迭代123-A-任务执行与CI闸门收口计划

## 1. 目标

本子迭代聚焦两个最影响当前交付可信度的核心问题：

- **任务执行模型不统一**
- **CI 质量门禁存在伪阻断**

目标不是一次引入新的任务平台，而是在当前架构下先把**语义收口、边界说清、验证补齐**，为后续的 `123-B` 和 `123-C` 提供更稳定的实施基础。

---

## 2. 当前问题摘要

### 2.1 任务执行模型问题

结合当前代码现状：

- `src/backend/app/services/backtest_service.py`
  - 任务状态已落库
  - 但执行仍由 API 进程本地 `runner` 调度
  - 取消语义仍依赖本地句柄
- `src/backend/app/services/param_optimization_service.py`
  - 混用进程内 `_tasks`、后台线程、`ProcessPoolExecutor` 和 DB 持久化
  - 运行时状态与持久化状态并非单一真相源
  - 重启恢复和多实例语义仍不够清晰

### 2.2 CI 门禁问题

从 `.github/workflows/ci.yml` 与 `nightly.yml` 可见：

- `bandit`、`safety`、集成测试使用了 `|| true`
- `pip-audit`、`npm audit` 也存在无条件放行

这会导致：

- 报告生成了，但失败不一定阻断合并
- 团队对质量门禁状态产生误判
- 技术债被“可见但不执行”地持续累积

---

## 3. 本子迭代范围

### 3.1 In Scope

- 统一回测与参数优化任务的状态语义
- 收敛任务状态的事实来源与运行时缓存边界
- 补齐任务取消、失败、完成、查询的契约说明
- 识别并调整 CI 中真正应阻断的质量项
- 为 blocker / advisory 建立清晰分层
- 补充必要的测试与文档说明

### 3.2 Out of Scope

- 不在本轮引入 Celery、RQ、Kafka 等新队列系统
- 不在本轮大改回测执行引擎本身
- 不在本轮重做全部安全扫描策略
- 不在本轮处理实盘/网关的大规模解耦

---

## 4. 设计原则

- **先统一语义，再调整实现**
- **先补契约测试，再收敛内部结构**
- **DB 是持久化真相源，进程内状态只做运行期辅助**
- **CI 先做分层，再做加严**
- **避免一次改动同时触碰过多运行路径**

---

## 5. 工作包拆分

## 工作包A1：统一任务生命周期语义

### 目标

为回测任务与参数优化任务建立一致的状态定义与状态流转说明。

### 建议动作

- 明确统一状态集合，例如：
  - `pending`
  - `running`
  - `completed`
  - `failed`
  - `cancelled`
- 明确每种状态的进入条件、退出条件、对外返回字段
- 明确进度计算规则与错误消息结构
- 明确取消成功、取消失败、已完成不可取消等边界语义

### 交付物

- 任务状态契约文档
- 回测/优化 API 返回语义对齐说明

### 验收标准

- 回测与优化任务的状态语义不再各说各话
- API/文档/测试对同一状态的解释一致

---

## 工作包A2：收敛任务状态事实来源

### 目标

明确哪些数据必须落库，哪些数据允许只保存在进程内。

### 建议动作

- 把持久化状态视为唯一长期事实源
- 将进程内 `_tasks` 降为运行期缓存，而非唯一状态来源
- 明确以下信息是否必须持久化：
  - 当前状态
  - 已完成数量
  - 失败数量
  - 错误信息
  - 结果摘要
  - 取消标记
- 补充重启后查询行为说明

### 交付物

- 任务状态来源说明
- 参数优化任务状态存储策略收敛方案

### 验收标准

- 任务查询结果不再依赖单一进程内字典才能成立
- 重启后任务状态行为有明确定义

---

## 工作包A3：任务契约测试补强

### 目标

让状态收口与取消语义有可回归的测试保护。

### 建议动作

- 增加或补强以下测试：
  - 创建任务后的初始状态
  - 状态从 `pending -> running -> completed`
  - 失败任务的错误信息输出
  - 取消任务后的查询结果
  - 查询不存在任务的行为
  - 多实例语义下的文档化限制验证
- 优先覆盖回测任务与参数优化任务的公共行为

### 交付物

- 后端契约测试
- 必要的错误响应测试

### 验收标准

- 核心任务生命周期场景均有测试覆盖
- 取消、失败、查询边界有明确断言

---

## 工作包A4：CI 门禁分层与加严

### 目标

把当前“看似检查、实际放行”的步骤改造成可解释的门禁策略。

### 建议动作

- 将 CI 检查分成两类：
  - **Blocking**：失败必须阻断合并
  - **Advisory**：失败可记录但不阻断
- 优先改造下列项目：
  - 单元测试
  - 集成测试
  - 文档链接校验
  - 依赖同步校验
  - 关键安全检查
- 减少或移除关键路径里的 `|| true`
- 在 CI summary 中显式区分：
  - 哪些项失败会阻断
  - 哪些项只是提示

### 交付物

- 更新后的 workflow 配置
- 门禁分层说明文档

### 验收标准

- 测试失败会真实导致 CI 失败
- summary 中能快速识别 blocker/advisory
- 团队能够清楚知道哪些质量项必须修

---

## 6. 依赖关系

本子迭代应优先于 `123-B` 和 `123-C`。

原因：

- 如果任务状态语义仍不清晰，后续做实盘/网关解耦时验证成本会很高
- 如果 CI 仍大量放行，后续两个子迭代即使改对了，也难以及时发现回归

---

## 7. 风险与控制措施

### 风险1：调整任务语义时引入回归

- **控制措施**：先补测试，再收敛实现；先改契约，再改内部结构。

### 风险2：CI 一次加严太多导致开发受阻

- **控制措施**：先区分 blocker/advisory，再逐步提升阻断项范围。

### 风险3：参数优化任务改造牵扯过深

- **控制措施**：本轮只收敛状态来源与对外语义，不贸然替换全部执行机制。

---

## 8. 验收证据

本子迭代完成时，至少应具备以下证据：

- **代码证据**
  - 与任务状态模型相关的服务/API/workflow 变更清单
- **测试证据**
  - 任务生命周期与取消语义相关测试通过
  - CI 门禁调整后的执行结果摘要
- **文档证据**
  - 状态语义说明
  - CI blocker/advisory 分层说明

---

## 9. 建议实施顺序

1. 先补任务状态与取消语义文档
2. 再补任务生命周期契约测试
3. 再收敛参数优化与回测任务的状态边界
4. 最后调整 CI 工作流与 summary 输出

---

## 10. 实施级任务清单

### T1：盘点回测与参数优化任务状态字段

- **目标**：列清回测任务与参数优化任务当前各自的状态字段、状态值、进度字段与错误字段。
- **输入**：`backtest_service.py`、`backtest_manager.py`、`backtest_runner.py`、`param_optimization_service.py`
- **输出**：一份差异清单，标出字段命名差异、状态值差异、取消语义差异。
- **依赖**：无。
- **完成定义**：能够明确回答“当前有哪些状态、各自从哪里写入、从哪里读取”。

### T2：定义统一任务状态契约

- **目标**：给回测与参数优化定义统一的生命周期状态和对外字段约定。
- **输入**：T1 差异清单。
- **输出**：统一状态契约说明，包括 `pending/running/completed/failed/cancelled` 的进入条件和返回字段。
- **依赖**：T1。
- **完成定义**：文档中已明确状态集合、状态流转、错误字段、进度字段与取消后的预期查询结果。

### T3：补充任务生命周期契约测试

- **目标**：用测试锁定统一状态契约，避免后续收敛时回归。
- **输入**：T2 状态契约。
- **输出**：覆盖创建、运行、完成、失败、取消、查询不存在任务等场景的测试。
- **依赖**：T2。
- **完成定义**：核心生命周期路径均有自动化断言，不再依赖人工解释状态语义。

### T4：收敛参数优化任务状态来源

- **目标**：明确 DB 与进程内 `_tasks` 的边界，减少进程内状态成为唯一真相源的情况。
- **输入**：T2 状态契约、T3 契约测试。
- **输出**：参数优化任务状态来源收敛改动，以及对应的查询/取消行为说明。
- **依赖**：T2、T3。
- **完成定义**：查询与取消行为不再仅依赖进程内字典才成立，重启后的状态语义有说明。

### T5：收敛回测任务取消与查询语义

- **目标**：明确回测任务中“持久化状态”和“本地运行句柄”的责任边界。
- **输入**：T2 状态契约、现有回测执行实现。
- **输出**：取消成功、取消失败、已完成不可取消、查询结果一致性的收敛方案。
- **依赖**：T2、T3。
- **完成定义**：回测任务取消与查询语义和文档保持一致，测试可验证。

### T6：梳理 CI blocker / advisory 分类表

- **目标**：明确哪些检查项应该阻断，哪些只做信息提示。
- **输入**：`.github/workflows/ci.yml`、`.github/workflows/nightly.yml`。
- **输出**：CI 分类表，至少覆盖单元测试、集成测试、文档链接、依赖同步、安全扫描。
- **依赖**：无。
- **完成定义**：每一项检查都能归类为 blocker 或 advisory，且理由明确。

### T7：调整关键 workflow 的阻断策略

- **目标**：去掉关键门禁中的伪放行配置。
- **输入**：T6 分类表。
- **输出**：更新后的 workflow 配置与 summary 输出逻辑。
- **依赖**：T6。
- **完成定义**：关键测试失败会真实导致 CI 失败，summary 中能分辨 blocker 与 advisory。

### T8：补充门禁说明与交付记录

- **目标**：让团队和后续代理能快速理解本轮门禁策略变化。
- **输入**：T7 结果。
- **输出**：门禁说明文档、变更摘要、必要的执行示例。
- **依赖**：T7。
- **完成定义**：后续维护者无需重新读 workflow 全文，也能理解本轮 CI 策略收口结果。

### 推荐执行顺序

`T1 -> T2 -> T3 -> T4/T5 -> T6 -> T7 -> T8`

其中：

- **先文档与测试，后状态收敛**
- **先分类，后加严 CI**
