## 4. 当前仍然存在的核心问题

### 4.1 P0：前端安全与关键交互链路仍有明显缺口

#### 发现

1. `StrategyPage.vue` 仍直接 `v-html="renderedReadme"` 渲染 Markdown HTML，未见净化逻辑。
2. `BacktestPage.vue` 仍通过 WebSocket URL 查询参数传递 token：`?token=...`。
3. `exportUtils.ts` 的 CSV 导出仅处理逗号/引号/换行，未处理公式注入前缀（如 `=`, `+`, `-`, `@`）。
4. `BacktestPage.vue` 中若干关键异步路径仍存在静默失败：
   - `runBacktest()` 的 `catch` 只复位 `loading`，不提示错误
   - `onMounted()` 的初始化请求未做显式错误处理
   - 回退轮询 `pollResult()` 为固定 1 秒，无退避策略

#### 风险

- README 或策略说明一旦包含恶意 HTML，前端存在 XSS 风险。
- Token 暴露在 URL 中，会进入浏览器历史、代理日志与监控采集链路。
- CSV 结果导出后在 Excel / WPS 中可能触发公式执行。
- 关键回测链路失败时用户只看到“没反应”或弱提示，影响可用性与排障效率。

#### 结论

这一组问题同时覆盖**安全性**与**用户可感知主路径**，应列为本轮最高优先级工作包。

---

### 4.2 P0：参数优化执行模型仍然混合，后续演进风险高

#### 发现

`param_optimization_service.py` 仍同时使用：

- 进程内 `_tasks` 全局字典
- `threading.Lock`
- 后台线程
- 独立事件循环线程
- `ProcessPoolExecutor`
- DB 持久化状态

虽然相比早期版本已经增加了 DB 落库和取消同步，但当前模型依旧是**“本地状态 + 线程 + 进程池 + 持久化”混合运行**。

#### 风险

- 行为边界难以解释，后续维护者很难判断“哪个状态才是真相源”。
- 取消、失败、重试、崩溃恢复语义不够统一。
- 在多 worker / 多实例 / 更复杂调度场景下，后续改造成本仍然偏高。
- 该文件本身体量大、职责密集，修复任意一个点都容易牵连其它路径。

#### 结论

参数优化模块现在不是“不能用”，而是**可用但脆弱**；最适合本轮做的是：

- 先统一状态语义
- 再拆执行器边界
- 最后补契约测试

---

### 4.3 P0：测试可信度仍有明显弱信号

#### 发现

测试目录中仍能看到以下模式：

- `assert resp.status_code in [200, 404, 500]`
- `except Exception: pass`
- 一部分测试以“只要不炸就算过”为主
- 若干 websocket / task 测试通过吞异常结束循环

这说明虽然测试数量不少，但**断言强度**与**失败信号质量**仍不稳定。

#### 风险

- 测试绿并不等于行为正确。
- 重构、收敛状态模型或加强安全时，回归保护会明显不足。
- 未来执行迭代计划时，容易出现“改完一轮，测试没报错，但问题还在”的情况。

#### 结论

如果不先修测试可信度，后续执行工作包 A/B/C 都会变成高风险改动。

---

### 4.4 P1：基础设施基线仍不够生产化

#### 发现

1. `MemoryCache` 仍是裸 `dict`：
   - 无并发保护
   - 淘汰策略本质仍是 FIFO
2. `database.py` 仍：
   - 直接 `create_all`
   - 启动时 `ensure_database_ready()` 自动建表并创建默认管理员
   - 未见 Alembic 迁移体系
   - 未见连接池参数治理
3. 本轮未发现 `alembic.ini` 和 migration `versions/` 目录。

#### 风险

- 多并发访问下，内存缓存行为不稳定。
- 数据库结构演进不可追踪，发布和回滚成本高。
- 启动时隐式建表/初始化用户，会模糊“开发便捷”与“生产约束”的边界。

#### 结论

这部分不一定先于前端安全/测试修复，但必须进入本轮计划，否则项目会长期停留在“可开发、难规范交付”的状态。

---

### 4.5 P1：遗留实盘链路仍暴露在主路由体系中

#### 发现

`api/router.py` 当前仍注册：

- 主实盘路由：`/live-trading`
- 旧链路路由：`/live-trading-crypto`

说明实盘领域仍然存在**并存链路**。

#### 风险

- 调用方和维护者很难快速判断哪条是当前主链路。
- 文档、测试、监控与用户行为可能被分流到不同实现。
- 后续修 bug 或补功能时容易重复劳动或修错地方。

#### 结论

这属于典型的“不是今天就爆炸，但会持续制造维护成本”的问题，建议在本轮作为架构收口项推进。

---

### 4.6 P1：CI 安全扫描仍是 advisory，不是 blocker

#### 发现

`.github/workflows/ci.yml` 中：

- Bandit 仍以 advisory 方式运行
- Safety 仍以 advisory 方式运行
- CI summary 也明确将 backend security 标为 informational

#### 风险

- 安全扫描结果“可见但不阻断”。
- 团队容易误判为“已经有安全门禁”。
- 随着仓库继续演进，安全债务会持续累积。

#### 结论

如果本轮要做安全修复，CI 也必须同步升级，否则问题仍会重新流入主干。

### 4.7 P2：文档与迭代索引仍然依赖人工同步

#### 发现

- `docs/INDEX.md` 当前只收录到迭代125。
- `docs/iterations/README.md` 当前只收录到迭代125。
- 说明文档目录仍然依赖人工维护，而不是“新增迭代即被纳入索引”的机制。

#### 结论

文档治理不是本轮最高风险，但应作为交付收口动作一起完成，避免这份新计划写完后又变成“孤立文档”。

## 5. 迭代126目标

本轮建议聚焦五个目标：

1. **修复前端主链路的安全与可观测性问题**。
2. **收敛参数优化任务执行模型，降低并发与状态复杂度**。
3. **提升测试可信度，让后续改动具备真实回归保护**。
4. **补齐基础设施基线（缓存、迁移、数据库启动语义）**。
5. **清理遗留实盘链路并同步文档索引**。

## 7. 建议执行顺序

建议按以下顺序推进：

`A → C → B → E(安全门禁部分) → D → E(遗留链路与文档部分)`

原因如下：

- 先修前端主链路安全和错误展示，用户马上受益。
- 再修测试可信度，给后续状态模型改造提供护栏。
- 然后处理参数优化这个当前最复杂的运行时模块。
- 紧接着把安全扫描升级为 blocker，防止同类问题回流。
- 最后做基础设施基线与遗留链路收口。

## 8. 风险与控制措施

| 风险 | 控制措施 |
|------|----------|
| 回测 WebSocket 鉴权改造影响前后端联调 | 先保留兼容层，再切换默认实现 |
| 参数优化模块重构时引入并发回归 | 先补契约测试，再拆实现 |
| 测试收紧后短期失败数上升 | 将其视为真实问题暴露，不视为回退 |
| Alembic 接入影响现有数据库初始化 | 先生成 baseline migration，再切换启动逻辑 |
| 遗留实盘链路移除后影响历史调用 | 先标记 deprecated 并做映射说明，再移除 |

## 11. 结论

当前项目已经不是“功能缺失型”问题，而是明显进入了**工程质量、运行时语义和交付可信度**主导的阶段。

相较于继续堆新功能，迭代126更适合做的是：

- 优先堵住前端安全与主链路体验问题
- 把参数优化这类高复杂度模块重新整理清楚
- 先让测试真正可信，再做结构性重构
- 同步把 CI 和文档索引纳入治理闭环

其中，参数优化这条主线已在后续实现中完成一轮关键收口：

- `/api/v1/optimization/submit` 与 `/api/v1/optimization/submit/backtest` 已成为权威入口
- legacy `/api/v1/backtests/optimization/grid` 与 `/api/v1/backtests/optimization/bayesian` 已降级为 deprecated compatibility proxy
- grid / bayesian 已统一到任务式提交、进度、结果、取消和持久化状态模型

完成这一轮后，项目会更适合进入下一阶段的稳定迭代执行。
