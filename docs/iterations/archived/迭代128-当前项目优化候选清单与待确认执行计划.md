## 3. 已确认不应再重复作为本轮主目标的事项

以下内容在当前仓库中**已经明显推进**，不建议继续按旧文档中的高优先级问题重复投入：

### 3.1 策略 README HTML 渲染已做净化

`StrategyPage.vue` 当前已经引入 `DOMPurify`，`renderedReadme` 最终会经过净化后再渲染。

结论：README 渲染 XSS 已不再是本轮最高优先级问题。

### 3.2 回测 WebSocket token 不再通过 URL 传递

`BacktestPage.vue` 当前通过 `sec-websocket-protocol` 传递 token；`deps.py` 中对应的 WebSocket token 提取逻辑也基于子协议头。

结论：旧文档里“token 仍在 URL query 中”的结论已过时。

### 3.3 回测页基础错误提示与轮询退避已有补强

`BacktestPage.vue` 当前已经：

- 对提交失败给出提示
- 对初始化失败给出提示
- 为轮询提供基础退避
- 在 WebSocket 失败时回退到轮询

结论：这一部分已不属于“静默失败严重”的状态，后续更适合做结构收敛，而不是继续零散补 toast。

### 3.4 CSV 公式注入防护已存在

`exportUtils.ts` 中已实现 `protectCsvFormula()`，会对以 `=`, `+`, `-`, `@`, `\t`, `\r` 开头的字符串进行保护。

结论：CSV 导出已不再是当前最值得优先处理的安全项。

### 3.5 内存缓存和数据库基线已有初步治理

当前已确认：

- `MemoryCache` 已使用 `RLock`
- 读取时会 `move_to_end()`，具备基础 LRU 语义
- `database.py` 中数据库启动已改为通过显式开关控制自举行为，而不是无条件自动建表/建默认管理员
- 仓库结构中已存在 `alembic/` 目录

结论：数据库与缓存不是“完全缺失治理”，但仍有进一步统一空间。

## 4. 当前更值得投入的优化方向

## 4.1 P0：回测主链路存在“重复入口 + 重复协议”问题

### 发现

当前回测链路存在明显重复：

1. REST 入口同时存在：
   - 旧入口：`/api/v1/backtest/*`（`backtest.py`）
   - 新入口：`/api/v1/backtests/*`（`backtest_enhanced.py`）
2. WebSocket 入口同时存在：
   - 根入口：`main.py` 中的 `/ws/backtest/{task_id}`
   - API 入口：`backtest_enhanced.py` 中的 `/api/v1/backtests/ws/backtest/{task_id}`
3. 两个 WebSocket 入口的实现语义并不完全一致：
   - `main.py` 的实现偏“鉴权 + 建连 + ping/pong”
   - `backtest_enhanced.py` 的实现偏“轮询任务状态并推送进度/结果”
4. 前端当前只使用根级 `/ws/backtest/{task_id}`，说明 API 路由下的 WebSocket 很可能已经不是主入口，但仍保留在代码中。

### 风险

- 维护者很难判断哪条链路才是权威实现。
- 后续修复回测实时推送问题时，容易只修到其中一条入口。
- 测试、文档、前端和后端实现可能继续围绕两套协议扩散。

### 结论

这已经不是简单的“兼容层保留”，而是**核心主链路仍存在双入口、双实现语义**的问题，应列为本轮最高优先级之一。

---

## 4.2 P0：参数优化能力实际上分裂成两套实现体系

### 发现

当前参数优化存在两套明显不同的能力面：

1. `backtest_enhanced.py` 通过 `OptimizationService` 暴露：
   - `/optimization/grid`
   - `/optimization/bayesian`
2. `optimization_api.py` 通过 `param_optimization_service.py` 暴露：
   - `/submit`
   - `/progress/{task_id}`
   - `/results/{task_id}`
   - `/cancel/{task_id}`
3. 这两套能力的运行方式差异较大：
   - 一套偏“直接返回结果”的同步/半同步式优化
   - 一套偏“创建任务、查询进度、可取消”的异步任务式优化
4. `param_optimization_service.py` 虽然已经拆出多个辅助模块，但仍是当前后端高复杂度热点；与此同时，`OptimizationService` 还在持续存在并对外暴露。

### 风险

- 调用方很难判断“哪个才是推荐的优化入口”。
- 两套实现会持续复制状态语义、错误语义和测试成本。
- 未来如果继续增强优化能力，会在两套体系之间重复劳动。

### 结论

参数优化当前最值得做的不是“再补一个功能”，而是**先统一能力面与权威入口**。这应列为本轮最高优先级之一。

### 当前状态更新

该项已完成彻底收口：

1. 权威入口已统一为 `/api/v1/optimization/submit` 与 `/api/v1/optimization/submit/backtest`。
2. `backtest_enhanced.py` 中的 `/api/v1/backtests/optimization/grid` 与 `/api/v1/backtests/optimization/bayesian` 仍保留，但已作为 deprecated compatibility proxy 代理到统一任务式主链路。
3. `optimization_service.py` 及 `OptimizationService` 类已被彻底删除；其有价值的测试逻辑已迁移为权威 helper 的直接单测。
4. `param_optimization_service.py` 为唯一的参数优化服务实现，不再依赖任何 legacy 模块。
5. 参数优化的提交、进度、结果、取消语义现已统一到任务式实现与持久化状态模型之下。

## 4.3 P1：`BacktestPage.vue` 仍然承担过多编排职责

### 发现

`BacktestPage.vue` 当前约 600+ 行，集中承担了：

- 策略选择与参数填充
- 提交回测
- WebSocket 建连与心跳
- 回退轮询
- 结果拉取与列表刷新
- 页面初始化
- 删除、取消、跳转等 UI 逻辑

而 `backtest` store 目前只封装了很薄的一层 API 调用。

### 风险

- 回测页任何一点修改都容易牵动整页状态。
- 与 WebSocket 相关的问题难以独立测试和复用。
- 若后续继续增加任务恢复、重连、断线提示等能力，页面复杂度会继续上升。

### 结论

这不是紧急缺陷，但已经是明显的**可维护性瓶颈**。建议作为第二优先级工作包推进。

## 4.4 P1：文档真相源已经出现明显漂移

### 发现

本轮核对中已经能看到多处“旧结论落后于当前代码”的情况，例如：

- 迭代126中仍把 WebSocket token URL 传递列为问题，但当前代码已经切到子协议
- 迭代126中仍把 README HTML 未净化列为问题，但当前 `StrategyPage.vue` 已接入 `DOMPurify`
- `project-context.md` 中部分 API 示例仍围绕旧版 `/api/v1/backtest/*` 路径描述
- 迭代127中的个别判断也已和当前代码不完全一致

### 风险

- 继续以旧文档为准会导致错误优先级判断。
- AI 代理、维护者和后续开发任务可能从不同文档读取到不同结论。
- 迭代文档会越来越像“历史快照”，而不是执行入口。

### 结论

文档问题不一定是本轮最紧急的工程问题，但它已经开始直接影响后续决策质量，建议纳入本轮收口项。

## 4.5 P1：回测执行模型仍然明确依赖“当前 API 进程拥有执行权”

### 发现

`backtest_service.py` 当前已经把任务状态持久化到数据库，但代码注释和取消逻辑仍明确表明：

- 执行仍由当前 API 进程持有
- 取消只有在任务运行于当前 API 进程时才真正有效
- 数据库存的是任务状态，但并不代表执行已完全脱离进程内运行时

### 风险

- 与文档中的分布式/多实例想象存在距离。
- 后续如果部署模型从单实例向多实例演进，取消、恢复、调度语义会再次成为瓶颈。
- 容易造成“状态持久化了，所以误以为天然支持多实例”的误判。

### 结论

这不是必须立刻改造成分布式调度系统，但至少应把**当前边界写清楚并决定后续方向**，否则会持续成为架构认知负债。

## 4.6 P2：遗留与兼容层仍偏多，建议做一轮轻量清点

### 发现

### 结论

这部分不应早于主链路收口，但适合作为本轮尾部工作包清理。

## 7. 建议执行顺序

建议按以下顺序推进：

`A → B → C → D → E`

原因如下：

- A 直接解决当前最核心的主链路重复入口问题。
- B 解决参数优化的能力分裂问题，避免继续沿两套体系演化。
- C 在主入口稳定后再做前端编排收敛，收益最高。
- D 用来把结论同步回文档，避免刚整理完又再次漂移。
- E 作为尾部收口，清理遗留兼容层与架构边界问题。

## 8. 风险与控制措施

| 风险 | 控制措施 |
|------|----------|
| 收口回测入口可能影响现有前端或脚本调用 | 先定义权威入口，再保留兼容层过渡 |
| 参数优化统一时引入接口回归 | 先补契约测试，再做入口裁剪 |
| 回测页重构影响当前交互体验 | 优先抽离内部编排，不先改 UI 结构 |
| 文档校准需要修改较多描述 | 只优先修正与主链路直接相关的陈述 |
| 遗留入口移除过快影响未知调用方 | 先盘点、标记、观察，再最终移除 |

## 11. 结论

基于当前真实代码状态，当前项目最值得继续优化的地方，已经从“基础安全修补”转向：
