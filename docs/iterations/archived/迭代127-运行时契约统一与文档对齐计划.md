# 迭代127 - 运行时契约统一与文档对齐计划

## 1. 背景

本轮审查基于 2026-03 当前仓库中的**真实代码、现有迭代文档与文档索引**交叉核对，重点不是重复迭代126已经提出的问题，而是识别：

- 哪些问题在迭代126之后已经**明显推进或部分收口**
- 当前项目还剩下哪些**更值得优先投入**的优化点
- 哪些地方已经出现**代码、接口、启动语义与文档之间的不一致**

本轮输出目标是形成一份**可以直接转为执行任务**的新迭代文档，作为后续 story / sprint 的输入。

## 3. 与迭代126相比，已确认推进的事项

以下问题在当前仓库中**不应再按旧结论重复定级**：

### 3.1 回测 WebSocket 鉴权前端已完成第一阶段收口

`BacktestPage.vue` 当前已经：

- 不再通过 URL 查询参数拼接 token
- 改为通过 `sec-websocket-protocol` 子协议传递 token
- 对 WebSocket 失败增加回退轮询逻辑
- 对轮询使用基础退避策略

结论：前端主链路已经从“明显不安全”推进到“基本可接受但后端兼容层仍未完全收口”的阶段。

### 3.2 回测页错误提示与初始化失败提示已有补强

`BacktestPage.vue` 当前已经：

- 在 `runBacktest()` 失败时显式提示错误
- 在 `onMounted()` 初始化失败时显式提示错误
- 对轮询最终失败给出明确用户提示

结论：这一部分已经不再属于“静默失败严重”的状态，后续重点应转向**错误语义统一**而不是单纯补 toast。

### 3.3 CSV 公式注入防护已经落地

`exportUtils.ts` 中已存在 `protectCsvFormula()`，会对以 `=`, `+`, `-`, `@`, `\t`, `\r` 开头的字符串进行保护。

结论：CSV 导出不再是本轮最高优先级风险点。

### 3.4 内存缓存和迁移基线已具备初步基础

当前仓库中已确认：

- `MemoryCache` 已引入 `RLock`
- `MemoryCache` 读取时会 `move_to_end()`，已具备基础 LRU 语义
- `src/backend/alembic.ini` 存在
- `src/backend/alembic/versions/` 目录存在

结论：这一部分不是“完全缺失”，但**运行时启动语义、迁移使用方式和文档表述仍未完全对齐**。

## 4. 当前仍然存在的核心问题

### 4.1 P0：WebSocket 运行时契约仍未完全统一

#### 发现

1. `deps.py` 中 WebSocket 鉴权仍保留 `query_params.get("token")` 兼容回退。
2. `main.py` 的 WebSocket 入口依赖 `get_websocket_current_user()` 返回 `(current_user, accepted_subprotocol)` 这一兼容型结构，说明当前仍是**双通道鉴权语义**。
3. `websocket_manager.py` 仍是全局可变连接表，连接、断开、广播和任务消息发送均直接操作共享状态。
4. `websocket_manager.py` 中“连接成功”消息会通过 `send_to_task()` 广播到同任务全部连接，而不是明确区分“给新连接的握手确认”和“给所有订阅者的状态广播”。

#### 风险

- 鉴权方式表面上已经切换，但后端仍接受旧入口，导致安全策略难以真正收口。
- WebSocket 协议语义不够单一，后续联调、排障和测试设计成本高。
- 全局连接管理虽然当前可用，但状态边界和广播边界不够清晰，未来容易出现“行为没错但语义混乱”的问题。

#### 结论

这一问题已经从“前端显性安全缺陷”转为**后端契约与连接管理语义不统一**，应作为本轮最高优先级工作包处理。

---

### 4.2 P0：API 表面与真实能力之间仍有分裂

#### 发现

1. `api/router.py` 当前同时注册：
   - `/backtest`
   - `/backtests`
2. `api/router.py` 仍保留旧链路：`/live-trading-crypto`。
3. 多个可选路由仍通过 `try/except ImportError: pass` 静默注册。
4. `main.py` 中 `/` 与 `/info` 的 feature 列表是静态声明，不一定反映当前路由是否真的可用。

#### 风险

- 调用方很难快速判断哪条接口是权威入口。
- 某个可选模块导入失败时，系统可能只是“悄悄少了一组路由”，但没有足够可观测性。
- 文档、监控、前端调用与后端实际能力之间容易产生偏差。

#### 结论

当前 API 的主要问题不是“缺功能”，而是**权威入口不够清晰、降级行为不够可见**。这类问题已经影响架构可维护性，应列为 P0。

---

### 4.3 P1：数据库启动语义、迁移体系与文档表述仍未完全一致

#### 发现

1. `database.py` 中 `ensure_database_ready()` 仍可能：
   - 根据 `DB_AUTO_CREATE_SCHEMA` 执行 `create_all`
   - 在启动时执行 `create_default_admin()`
2. `docs/INDEX.md` 中“数据库初始化指南”明确写有“**不再自动在启动时执行**”的表述。
3. 这说明“文档中的启动语义”和“代码中的默认行为”之间仍存在不一致。
4. 虽然 Alembic 基线已存在，但应用启动阶段仍保留较强的隐式自举行为。

#### 风险

- 开发环境和生产环境的数据库行为边界不清晰。
- 默认管理员创建行为如果没有显式开关与审计说明，容易成为运维和安全治理盲点。
- 迁移体系虽然存在，但如果启动逻辑仍大量依赖隐式 `create_all`，团队会难以形成统一的 schema 演进纪律。

#### 结论

数据库层现在的主要问题不是“没有迁移”，而是**迁移存在、运行时也能自举、文档还另有表述**，需要统一成单一契约。

---

### 4.4 P1：参数优化模块仍然是当前后端最高复杂度热点

#### 发现

`param_optimization_service.py` 仍然同时包含：

- 运行时任务缓存
- 线程锁
- 独立事件循环线程
- `ProcessPoolExecutor`
- 子进程执行
- DB 状态持久化
- 结果聚合与进度构造

相比迭代126，本轮并未发现这部分出现结构性收口。

#### 风险

- 该模块继续维持“多套并发/状态机制叠加”的状态。
- 任何一个小改动都可能影响取消、失败、进度查询和结果聚合。
- 当接口层和运行时契约需要继续收紧时，这个模块会持续成为高回归风险点。

#### 结论

参数优化模块仍应纳入本轮，但建议放在 WebSocket / API 契约统一之后推进，避免多个高复杂度改动并行发生。

---

### 4.5 P1：测试治理的重点应从“泛化批评”转向“运行时契约覆盖”

#### 发现

本轮快速检索显示，测试目录中仍能找到少量：

- `except Exception: pass` 的清理型代码
- 依赖 `pass` 的占位式 app / stub

但更重要的是，当前仓库中仍缺少足够显式的契约测试去覆盖以下高价值行为：

- WebSocket 只接受权威鉴权方式时的行为
- WebSocket 连接管理器的连接、断开、广播边界
- 路由注册失败时的可观测性
- 数据库启动模式与默认管理员初始化模式
- `/backtest` 与 `/backtests` 的职责边界

#### 风险

- 即使“测试数量很多”，也未必能保护最关键的运行时语义。
- 后续做契约收口时，如果没有针对性的回归测试，容易出现接口没坏但语义漂移。

#### 结论

本轮测试工作的重点，应该从“继续扫弱断言”转为**给关键运行时契约补护栏**。

---

### 4.6 P2：文档、索引与项目上下文的真相源仍然分散

#### 发现

1. `docs/INDEX.md`、`docs/iterations/README.md`、`project-context.md` 分别承担索引、迭代汇总、项目上下文角色。
2. 当前这些文档的部分描述已经出现与代码现状不同步的迹象。
3. 这类偏差并不一定是单点错误，更像是**缺少“谁是权威真相源、如何更新”的机制**。

#### 风险

- 后续继续新增迭代或进行架构调整时，文档会越来越像“历史快照”，而不是执行参考。
- AI 代理、开发者和维护者可能从不同文档读到不同结论。

#### 结论

文档问题不属于本轮最高风险，但必须作为交付收口项一起推进，否则这份新迭代文档也会很快再次过时。

---

## 5. 迭代127目标

本轮建议聚焦五个目标：

1. **统一 WebSocket 鉴权和连接管理契约**。
2. **收敛 API 权威入口，提升降级与缺失能力的可观测性**。
3. **统一数据库启动、迁移、自举与默认管理员初始化语义**。
4. **继续拆解参数优化高复杂度实现，降低状态与并发耦合**。
5. **以运行时契约测试和文档同步作为本轮交付闭环**。

## 7. 建议执行顺序

建议按以下顺序推进：

`A → B → E(契约测试部分) → C → D → E(文档同步部分)`

原因如下：

- 先统一 WebSocket 契约，能最快降低前后端联调和运行时语义分裂。
- 再收敛 API 权威入口，避免后续测试和文档继续围绕双入口扩散。
- 紧接着补契约测试，为数据库和参数优化改造建立护栏。
- 然后统一数据库启动与迁移语义。
- 最后再推进参数优化这个高复杂度模块的进一步拆分与收口。

## 8. 风险与控制措施

| 风险 | 控制措施 |
|------|----------|
| WebSocket 鉴权收口影响历史前端或脚本 | 先定义兼容窗口，再决定是否彻底移除旧入口 |
| API 权威入口调整影响现有调用方 | 保留兼容期，并在文档中标记推荐入口 |
| 数据库自举逻辑调整影响本地开发体验 | 区分开发模式与生产模式，并提供显式脚本入口 |
| 参数优化重构引入并发回归 | 先补状态契约测试，再拆实现 |
| 文档同步仍靠人工维护导致再次漂移 | 将索引更新纳入每轮迭代收口动作 |

## 11. 结论

当前项目已经明显进入**运行时契约、接口权威性、启动语义和文档真相源统一**主导的阶段。

相较于继续横向扩功能，迭代127更值得做的是：

- 先把 WebSocket 这条核心实时链路的契约统一清楚
- 再把 API 权威入口和降级可观测性收紧
- 随后统一数据库启动、自举与迁移语义
- 最后在测试和文档层把这些规则固化下来

完成这一轮后，项目会更适合继续进入可预测、可验证、可交付的稳定演进阶段。
