# 迭代123-当前项目优化与迭代计划

## 1. 背景

`backtrader_web` 已经具备较完整的产品形态：后端基于 FastAPI，前端基于 Vue 3，覆盖策略管理、回测、参数优化、模拟交易、实盘交易、监控告警等多个模块。但从当前代码与文档现状来看，项目仍存在几个明显的“继续堆功能之前必须先收口”的问题：

- 任务执行模型在不同模块之间并不统一。
- 实盘与网关管理职责集中度过高，演进成本持续上升。
- 一部分接口和服务仍保留占位实现或 TODO，容易让功能“看起来有，实际上不完整”。
- CI、文档索引、项目扫描结果与真实代码状态存在一定漂移。

本次迭代不追求一次性大重构，而是输出一份**可执行、可分批推进、带优先级和验收口径**的优化计划，作为接下来若干轮稳定性与工程质量治理的统一入口。

## 3. 当前项目最需要优化的方向

### 3.1 P0：任务执行与状态管理模型仍然分裂

当前任务相关能力已经比早期版本更成熟，但仍然没有形成统一的运行时模型：

- `BacktestService` 已经把任务状态落库，但实际执行仍由当前 API 进程持有本地 runner 与取消句柄。
- `param_optimization_service.py` 同时使用：
  - 进程内全局 `_tasks`
  - 后台线程
  - `ProcessPoolExecutor`
  - 可选 DB 持久化
- 两类任务的“状态源头”“取消语义”“重启恢复语义”还没有完全统一。

这会直接带来几个问题：

- 多 worker / 多实例时，行为难以严格预测。
- 测试需要同时覆盖 DB、线程、进程池与本地状态，复杂度高。
- 后续要接入更稳健的队列或调度器时，迁移成本会继续变大。

**结论**：任务执行模型应被视为当前最高优先级改进项之一。

---

### 3.2 P0：CI 质量门禁存在“看起来跑了，实际上没卡住”的漏口

当前 CI 已经具备 lint、测试、安全扫描、文档检查等基本能力，但仍存在几个明显风险点：

- `.github/workflows/ci.yml` 中 `bandit`、`safety`、集成测试使用了 `|| true`
- `nightly.yml` 中 `pip-audit`、`npm audit` 也保留了 `|| true`

这意味着：

- 报告能生成，但失败不一定阻断合并。
- 团队容易把“有扫描步骤”误认为“已有强门禁”。
- 当项目规模继续增长时，质量问题会以“可见但不可执行”的方式积累。

**结论**：CI 需要区分“信息性任务”和“强阻断任务”，并把真正关键的质量项变成硬门禁。

---

### 3.3 P0/P1：实盘实例与网关管理职责集中度过高

`src/backend/app/services/live_trading_manager.py` 已经膨胀为超大服务文件，承担了过多职责：

- 实例 CRUD 与持久化
- 进程生命周期管理
- 启停状态同步
- 网关连接与断连
- 手工连接逻辑
- 预置网关参数定义
- 健康状态汇总
- 运行中实例与网关关联关系维护

即使当前功能可用，这种结构仍然会持续放大几个问题：

- 任何一个小改动都容易影响多个维度。
- 单元测试需要构造复杂上下文，维护成本高。
- 新增交易所、网关模式或运行方式时，文件还会继续膨胀。

**结论**：需要把实盘实例管理拆成清晰的子职责模块，而不是继续在单文件里累计逻辑。

---

### 3.4 P1：仍有若干“占位实现/未完成接口”暴露在主链路附近

当前仓库里仍能看到若干尚未闭环的功能点：

- `realtime_data_service.py` 中历史数据获取仍是空实现返回。
- `api/realtime_data.py` 中实时推送仍带有“临时 polling / TODO 集成”的痕迹。
- `api/comparison.py` 的 share 逻辑仍未实现。
- `api/backtest_enhanced.py` 中策略信息仍有“TODO: Get from strategy table”。
- `tests/test_monitoring_service.py` 里明确写明某分支“TODO not implemented”。

这些点未必都要马上补成完整功能，但至少应明确分类：

- 是立即补全。
- 是显式降级并在接口层声明限制。
- 是暂缓实现并从 UI/API 暂时隐藏。

**结论**：本轮计划应把“占位逻辑清单化”，避免继续把未完成逻辑保留在对外路径上。

---

### 3.5 P1：服务层耦合与性能热点仍有收敛空间

从当前代码可以看到一些仍值得继续治理的点：

- `ComparisonService` 对多个 backtest 结果做校验与读取时存在重复调用 `get_result()` 的模式。
- `StrategyService` 仍以目录扫描 + 逐个读取 `config.yaml`/代码文件为主，随着策略数量增加，冷启动和列表性能仍可能成为热点。
- 若干服务通过直接实例化其他服务完成协作，边界还不够清晰，不利于替换与测试。

这些问题未必会立刻造成故障，但会在以下场景中持续放大：

- 策略模板数继续增长
- 多用户并发访问模板与结果列表
- 后续把服务做成更细粒度模块时

**结论**：本轮建议把“边界收敛 + 热点性能优化”放到第二优先级，不抢 P0，但要进入计划。

---

### 3.6 P1/P2：文档与项目事实存在漂移

当前文档体系非常丰富，但也暴露出两类问题：

- `docs/INDEX.md`、`docs/iterations/README.md` 与现有迭代文档不完全同步。
- `docs/project-scan-report.json` 中项目根路径仍指向旧目录，说明项目事实快照已发生漂移。
- 一些报告文档反映的是某次阶段性状态，不再完全代表当前主干实际情况。

这会导致：

- 新成员或后续 AI 代理读取文档时得到过期信息。
- 运维、测试、规划文档之间互相引用时逐步失真。

**结论**：文档治理不应只做“写更多文档”，而应建立**文档与代码事实同步机制**。

---

## 4. 迭代123目标

本迭代建议聚焦以下四个目标：

- **目标1**：统一任务执行与状态管理语义，为回测与优化建立一致的运行时边界。
- **目标2**：拆分实盘实例与网关管理职责，降低核心服务复杂度。
- **目标3**：清理占位实现与未闭环接口，减少“功能已暴露但能力未完成”的路径。
- **目标4**：把 CI、文档索引和项目基线同步起来，提升交付可信度。

## 5. 建议工作包

## 工作包A：统一任务执行与状态模型

### 目标

收敛回测任务与参数优化任务的执行模型，明确：

- 谁是任务状态的唯一事实来源
- 谁负责持有运行时句柄
- 取消、失败、恢复、重试的语义是什么

### 建议内容

- 为回测与优化定义统一的任务生命周期状态图。
- 约定 DB 为持久化真相源，进程内状态只做运行期缓存。
- 把优化任务从“全局 `_tasks` + 线程 + 进程池”的混合模型进一步收敛到更清晰的执行器边界。
- 明确多 worker / 多实例下哪些语义已支持，哪些语义暂不支持。
- 为取消、失败、部分完成、重启恢复补充契约测试。

### 交付物

- 任务执行模型说明文档
- 回测/优化统一任务状态约定
- 关键接口与测试补强

### 验收标准

- 文档明确说明任务状态来源与取消语义。
- 回测与优化的任务状态字段、进度查询、错误返回结构趋于一致。
- 进程重启或任务取消时，不再依赖单一进程内字典作为唯一真相源。

---

## 工作包B：实盘与网关管理解耦

### 目标

降低 `LiveTradingManager` 的职责密度，建立可维护的模块边界。

### 建议内容

将当前单体管理器拆成至少以下几个方向：

- **实例存储层**：负责实例数据持久化与读取
- **进程监督层**：负责启动、停止、存活探测、启动后同步
- **网关生命周期层**：负责 gateway runtime 创建、连接、断连、健康检查
- **预置配置层**：负责 presets 与 editable_fields 定义

同时评估是否继续长期使用 JSON 文件作为实盘实例主存储，或逐步抽象为仓储接口，为未来迁移到 DB 做准备。

### 交付物

- 拆分后的服务设计
- 精简后的 manager 主协调器
- 针对实例生命周期与 gateway 生命周期的更细测试

### 验收标准

- 单个服务文件不再承担实例持久化、网关连接、预置定义、进程管理全部职责。
- 新增交易所或新 gateway 类型时，不需要在单个超大文件中同时修改多个责任区。
- 实盘实例启停、健康查询、手工连接行为可由独立测试覆盖。

---

## 工作包C：占位能力清理与实时链路补全

### 目标

把当前已暴露但未完整实现的功能做一次系统清点与处理。

### 建议内容

- 梳理实时数据、历史数据、comparison 分享、增强回测策略元数据等占位点。
- 对每一项给出处理策略：
  - 立即实现
  - 显式降级
  - 暂时隐藏
- 对实时数据链路优先补齐“最小可用版本”，至少避免 API 返回看似成功但实际无能力。
- 对 comparison 分享等非核心路径，若短期不做，建议在接口或文档中标明未开放状态。

### 交付物

- 占位逻辑清单
- 每项占位能力的处置决策
- 已补全或已降级的接口说明

### 验收标准

- 主路径不再保留“返回空结果但无说明”的占位实现。
- 对外暴露接口的能力边界清晰可解释。
- TODO/FIXME 在主业务路径中的数量明显下降。

---

## 工作包D：CI 质量门禁与安全扫描治理

### 目标

把“报告型检查”和“阻断型检查”分层，减少伪门禁。

### 建议内容

- 对 `bandit`、`safety`、`pip-audit`、`npm audit`、集成测试重新分类：
  - 必须阻断合并
  - 仅生成报告但需在 summary 中明确标识为 advisory
- 移除关键质量步骤中的无条件 `|| true`。
- 为 integration test 建立明确失败信号，而不是始终放行。
- 对 nightly 审计结果建立留痕与处理流程说明。

### 交付物

- 调整后的 CI 分层策略
- 关键 workflow 更新
- 质量门禁说明文档

### 验收标准

- 必须通过的测试与扫描在 CI 中真正影响合并结果。
- Advisory 项与 blocking 项边界清晰。
- CI 失败原因可从 summary 快速定位。

---

## 工作包E：文档与项目事实同步治理

### 目标

建立“文档索引、迭代记录、项目快照”与真实仓库状态的一致性。

### 建议内容

- 更新 `docs/INDEX.md` 与 `docs/iterations/README.md`，保持新增文档可发现。
- 重新生成或修正项目扫描结果，确保根路径、模块数量、迭代数量不漂移。
- 为关键状态性文档加上“生成时间 / 适用范围 / 数据来源”。
- 明确哪些文档是长期有效规范，哪些文档只是阶段性报告。

### 交付物

- 更新后的索引与迭代入口
- 项目基线快照修正
- 文档分层约定

### 验收标准

- 新增迭代文档可从索引快速进入。
- 项目快照不再引用旧路径或过期状态。
- 核心文档与阶段性报告的定位清晰。

## 7. 建议排期

### 第一阶段：稳定性与交付闸门收口

建议先完成：

- 工作包A：统一任务执行与状态模型
- 工作包D：CI 质量门禁与安全扫描治理
- 工作包E 的最小部分：索引与项目快照修正

**目标**：先把“系统行为是否可信、交付是否可靠”这两个根问题收紧。

### 第二阶段：核心服务减负与占位点清理

建议推进：

- 工作包B：实盘与网关管理解耦
- 工作包C：占位能力清理与实时链路补全

**目标**：降低后续开发和线上排障成本，减少核心模块继续失控膨胀。

### 第三阶段：性能与边界细化

建议推进：

- 服务层边界整理
- 策略扫描/回测结果/对比分析的性能优化
- 可观测性与运行基线指标补齐

**目标**：为后续功能扩展留出空间，而不是持续在高复杂度地基上叠加功能。

## 8. 子迭代拆分建议

为降低单次改动面过大带来的回归风险，建议将本轮计划拆为三个连续子迭代执行。

- [迭代123-A：任务执行与CI闸门收口计划](迭代123-A-任务执行与CI闸门收口计划.md)
- [迭代123-B：实盘与网关管理解耦计划](迭代123-B-实盘与网关管理解耦计划.md)
- [迭代123-C：实时链路与占位能力补全计划](迭代123-C-实时链路与占位能力补全计划.md)

| 子迭代 | 核心主题 | 主要产出 | 完成标志 |
|------|------|------|------|
| 迭代123-A | 任务执行模型 + CI 闸门 | 统一任务状态语义、清理伪门禁、补齐契约测试 | 任务状态/取消/失败语义清晰，关键 CI 不再无条件放行 |
| 迭代123-B | 实盘与网关管理解耦 | 拆分 `live_trading_manager`，明确实例/网关/进程职责 | 实盘主服务复杂度下降，关键行为可独立测试 |
| 迭代123-C | 实时链路与占位能力补全 | 清理 TODO 能力、补齐降级说明或最小可用实现 | 对外暴露接口不再存在“看似可用、实际空实现”的主路径 |

建议不要并行推进三项，因为它们共享同一批核心模块，串行推进更利于控制回归面。

## 9. 依赖关系与执行顺序

推荐执行顺序如下：

1. **先做 123-A**
   - 原因：任务执行语义与 CI 闸门决定后续改动是否容易验证、是否容易在回归时快速发现问题。
2. **再做 123-B**
   - 原因：实盘与网关改造复杂度最高，必须在验证环境更可信的前提下推进。
3. **最后做 123-C**
   - 原因：占位能力与实时链路补全更适合在核心架构边界稳定后收口，避免补完后又被前面重构打散。

其中依赖关系可简化为：

- `123-A` 是 `123-B` 的前置条件。
- `123-B` 不是 `123-C` 的绝对前置条件，但建议先于 `123-C` 完成。
- 文档与索引治理可以穿插进行，但项目快照修正建议在 `123-A` 收口后同步更新一次。

- **测试证据**：通过的测试/校验命令与结果摘要

## 11. 主要风险与控制措施

### 风险1：任务模型调整引发已有回测/优化流程回归

**控制措施**：

- 先补契约测试，再动执行模型。
- 先统一状态语义，再调整执行器内部结构。
- 避免在同一批次内同时重构回测与优化的全部实现细节。

### 风险2：实盘与网关改造影响线上运行稳定性

**控制措施**：

- 保留旧行为兼容层，采用分层替换而不是一次性重写。
- 优先拆分内部职责，不先改对外 API 形态。
- 对实例启停、网关复用、健康检查单独做回归验证。

### 风险3：CI 门禁收紧后暴露大量历史欠账，影响开发节奏

**控制措施**：

- 先把 blocker 与 advisory 分层，不要一次把所有扫描都升为硬阻断。
- 优先提升真正影响可交付性的门禁，如测试失败、关键安全失败、依赖同步失败。
- 对暂时无法阻断的项给出整改期限和留痕机制。

### 风险4：占位功能补全范围失控

**控制措施**：

- 只处理对外主路径可见的占位点。
- 对每个点先决定“补全 / 降级 / 隐藏”，再进入实现。
- 不把实时链路补全扩展成新的大功能开发。
