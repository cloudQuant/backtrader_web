## 1. 工作区列表页（WorkspaceListPage） [P0]

### 1.1 页面布局

**导航栏区域:**
- 左侧标题："策略研究"
- 右侧操作按钮组:
  - `+新建工作区` — 打开 CreateWorkspaceDialog
  - `删除工作区` — 删除选中工作区(需确认)
  - `卡片/表格` — 视图切换按钮组 (el-radio-group)

**内容区域:**
- **卡片模式**: 使用 el-row + el-col 网格布局，每个工作区一张卡片
- **表格模式**: 使用 el-table 展示工作区列表

### 1.2 工作区卡片（WorkspaceCard）

**卡片内容:**
```
┌────────────────────────────┐
│ 📁 黄金策略研究     [idle] │  ← 名称 + 状态标签
│                            │
│ 多策略多周期黄金回测       │  ← 描述（最多2行，溢出省略）
│                            │
│ 策略单元: 6  已完成: 4     │  ← 统计信息
│                            │
│ 创建: 2026-03-15 15:00     │  ← 时间信息
│ 更新: 2026-03-15 16:30     │
│                            │
│ [编辑]           [删除]    │  ← 操作按钮

```

**交互:**
- 点击卡片体 → 进入工作区详情 (`/workspace/:id`)
- 编辑按钮 → 打开 CreateWorkspaceDialog（编辑模式）
- 删除按钮 → 确认删除
- 状态标签颜色: idle=info, running=warning, completed=success, error=danger

### 1.3 CreateWorkspaceDialog

**字段:**
| 字段 | 组件 | 必填 | 说明 |
|------|------|------|------|

**表格特性:**

- 运行状态列使用 el-tag 彩色标签

- 双击行打开策略单元详细设置
- 右键菜单支持快捷操作
- 表头可拖动调整列宽

**运行状态颜色:**
- idle → `type="info"` (灰色)
- queued → `type="warning"` (橙色)
- running → `type=""` (蓝色) + 动画
- completed → `type="success"` (绿色)
- failed → `type="danger"` (红色)
- cancelled → `type="warning"` (橙色)

### 2.4 策略单元行为规范

**运行流程:**
1. 用户选中N个单元 → 点击“运行选中”
2. 前端调用 `POST /workspace/{id}/run` 传入 unit_ids
3. 后端为每个单元组装 BacktestRequest，调用 `BacktestService.run_backtest()`（复用现有引擎）
4. 前端通过轮询 `GET /workspace/{id}/run-status` 刷新状态（3-5秒） [P0]
5. 前端实时更新表格中的 run_status / 已用时间 / 剩余时间
6. 完成后，后端提取关键指标写入 `unit_backtest_results.metrics_snapshot`（引用模式），更新 run_count
7. 前端刷新表格

> **WebSocket 增强** [P1]: 后续可替换轮询为 WebSocket 推送，连接失败时自动回退到轮询。

**并行运行:**
- 与串行运行的区别: 使用多线程/多进程同时执行
- 并行度由工作区设置 `parallel_threads` 控制
- 前端同时显示多个单元为 running 状态

**停止操作:**
- 调用 `POST /workspace/{id}/stop` 传入 unit_ids
- 后端取消对应的 asyncio 任务
- 单元状态变为 cancelled

## 3. 弹窗交互规范

### 3.1 新建策略单元（CreateUnitDialog） [P0]

**流程:**
1. 选择新建方式: 批量 或 叠加
2. 输入单元组名
3. 选择商品（从数据源列表选择，或手动输入代码）
4. 选择策略模板（从 strategyStore.templates 选择）
5. 配置周期和时间范围
6. 确定 → POST 创建

**批量模式:** 为每个选中的商品创建一个策略单元
**叠加模式:** 创建一个策略单元，但关联多个商品

**策略模板选择器:**
- 复用现有 `strategyStore.templates` 数据源
- 选择后自动填充默认参数

### 3.2 数据源管理（DataSourceDialog） [P0]

**功能:**
- 显示当前单元的商品列表（支持添加/删除多个商品）
- 配置周期设置（周期+N值）
- 配置范围设置（样本数 或 起始/结束日期）
- 复权方式、分割方式、数据范围
- "批量应用" 将设置应用到多个选中单元

### 3.3 策略单元设置（UnitSettingsDialog） [P0]

**功能分区:**
- **保证金/利率**: 保证金方式、多空保证金率、无风险利率
- **公式交易设定**: 初始资金
- **手续费**: 费率方式、开仓/平仓/平今费率
- **滑点**: 滑点单位、滑点值
- **周期/范围覆盖**: 可选覆盖数据源中的周期和范围设置
- **复权/数据范围**: 覆盖数据源设置

**操作按钮:**
- `设为默认` — 保存当前设置为工作区默认
- `恢复默认` — 恢复为工作区默认

### 3.4 策略应用设置（StrategyParamsDialog） [P0]

**功能:**
- 显示策略公式的参数列表（从策略模板读取）
- 每个参数可启用/禁用
- 可编辑参数值
- 支持添加/删除/上移/下移参数

**数据流:**
1. 选择策略单元 → 读取 `unit.params`
2. 如果为空，从策略模板加载默认参数
3. 编辑后保存 → PUT `/workspace/{id}/units/{unitId}/params`

### 3.5 切换商品 [P2 预留]

> **范围控制**: 本迭代只预留按钮和禁用态，不承诺完整能力。

**功能:**
- 打开一个商品选择器对话框
- 选择新商品后，更新选中单元的 symbol/symbol_name
- 同时更新数据源配置中的商品

### 3.6 分组更名（GroupRenameDialog） [P1]

**重命名方式（参考TBQuant截图）:**
- **自定义**: 直接输入新组名
- **使用公式名**: 用策略名作为组名
- **使用代码**: 用品种代码作为组名
- **使用名称**: 用品种名称作为组名
- **使用分类**: 用分类标签作为组名
- **替换**: 对现有组名做字符串替换

**批量逻辑:** 对所有选中单元的 group_name 执行重命名

### 3.7 单元重命名（UnitRenameDialog） [P1]

与分组更名类似，但修改的是单元在工作区内的显示名称。
重命名选项同分组更名（自定义/公式名/代码/名称/分类/替换）。

## 5. 状态刷新集成

### 5.1 轮询方案 [P0 首版]

进入 WorkspaceDetailPage 时启动轮询，离开时停止:
```typescript
// 每 3-5 秒调用 GET /workspace/{id}/run-status
const pollInterval = setInterval(() => fetchRunStatus(), 3000)
onUnmounted(() => clearInterval(pollInterval))
```

### 5.2 WebSocket 增强 [P1 可选]

进入 WorkspaceDetailPage 时尝试建立 WebSocket 连接:
```
ws://host/ws/workspace/{workspace_id}?token=xxx
```
连接失败时自动回退到轮询模式。

### 5.3 消息类型

```typescript
// 单元运行进度
interface UnitProgressMessage {
  type: 'unit_progress'
  unit_id: string
  task_id: string
  progress: number      // 0-100
  message: string
  elapsed_time: number  // 秒
  estimated_remaining: number  // 秒
}

// 单元运行完成
interface UnitCompletedMessage {
  type: 'unit_completed'
  unit_id: string
  task_id: string
  metrics: { total_return: number, sharpe_ratio: number, ... }
}

// 单元运行失败
interface UnitFailedMessage {
  type: 'unit_failed'
  unit_id: string
  task_id: string
  error: string
}

// 优化进度
interface OptimizationProgressMessage {
  type: 'optimization_progress'
  unit_id: string
  total: number
  completed: number
  progress: number
}

// 报告计算完成
interface ReportCompletedMessage {
  type: 'report_completed'
  report_id: string
}
```

### 5.4 前端处理

```typescript
// WorkspaceDetailPage.vue
function handleWsMessage(msg: any) {
  switch (msg.type) {
    case 'unit_progress':
      // 更新表格中对应单元的进度
      updateUnitProgress(msg.unit_id, msg.progress, msg.elapsed_time, msg.estimated_remaining)
      break
    case 'unit_completed':
      // 更新状态为completed，刷新指标
      updateUnitStatus(msg.unit_id, 'completed', msg.metrics)
      break
    case 'unit_failed':
      // 更新状态为failed
      updateUnitStatus(msg.unit_id, 'failed')
      ElMessage.error(`单元 ${msg.unit_id} 运行失败: ${msg.error}`)
      break
    case 'optimization_progress':
      // 更新优化进度
      break
    case 'report_completed':
      // 刷新报告数据
      break
  }
}
```
