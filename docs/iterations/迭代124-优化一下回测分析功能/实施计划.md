# 迭代124 - 实施计划（优化版）

## 1. 文档目标

本计划基于 `需求.md` 的初始诉求进行重构，目标不是一次性完整复制 TBQuant，而是在 **不破坏现有回测/优化架构** 的前提下，分阶段交付一个可落地、可联调、可验收的“工作区驱动回测分析”能力。

本计划重点解决原计划中的 4 个问题：

1. **范围过大**  
   初始计划把工作区、策略单元、批量运行、优化结果、组合报告、完整 TBQuant 指标、文件导入导出等都压在一个迭代中，研发风险过高。

2. **与现有架构衔接不够清晰**  
   当前项目已经有 `BacktestPage.vue`、`OptimizationPage.vue`、`BacktestService`、`optimization_api.py`、`BacktestTask`、`BacktestResultModel`、`OptimizationTask` 等现成能力，实施计划必须明确复用边界，避免重复建设。

3. **验收口径偏“截图对齐”**  
   研发执行需要的是“功能是否可用、接口是否稳定、状态流是否闭环、数据是否正确”，而不是单纯“看起来像”。

4. **缺少分期与降级策略**  
   对大量按钮功能、完整指标体系、图表、组合报告没有做主次划分，不利于真实交付。

## 2. 实施原则

### 2.1 复用优先

- 运行回测复用现有 `src/backend/app/services/backtest_service.py`
- 参数优化复用现有 `src/backend/app/api/optimization_api.py` 与 `param_optimization_service`
- 现有页面 `BacktestPage.vue`、`OptimizationPage.vue` 保留，不做破坏式改造
- 新增“策略研究/工作区”作为增强入口，不替代原入口

### 2.2 先闭环，再丰富

先实现研发可闭环的主链路：

- 工作区创建
- 策略单元管理
- 运行选中单元
- 查看单元结果状态
- 基础优化结果接入

再逐步补充：

- 高级导入导出
- 完整 TBQuant 指标集
- 组合报告高级能力
- 图表增强与自定义字段

### 2.3 单迭代控制复杂度

本迭代建议只承诺 **P0 + P1 的基础能力**，P2 作为“可顺延项”。

### 2.4 向下兼容

- 现有 `/backtest`、`/optimization` 路由保持可用
- 现有 `BacktestTask` / `BacktestResultModel` / `OptimizationTask` 不重构
- 新增数据表尽量只承载“工作区维度的组织关系与快照索引”

## 5. 核心设计修订

## 5.1 数据模型修订建议

原计划直接新增多张“结果表”，复杂度偏高。优化后建议如下：

### 必须新增

- `Workspace`
- `StrategyUnit`
- `WorkspaceRunRecord` 或 `StrategyUnitRunRecord`
- `WorkspaceReport`

### 可合并/降级

- `UnitBacktestResult` 不建议存储完整回测结果副本  
  建议优先存储：
  - `unit_id`
  - `backtest_task_id`
  - 关键指标快照
  - 引用时间

  详细权益曲线、交易明细仍从现有 `BacktestResultModel` 读取。

- `UnitOptimizationResult` 不建议一开始自建复杂结果表  
  建议优先存储：
  - `unit_id`
  - `optimization_task_id`
  - 配置快照
  - 排名第一参数
  - 最近一次结果摘要

  完整优化结果仍优先从现有优化任务结果读取或做轻量缓存。

### 这样做的收益

- 减少重复数据
- 降低迁移脚本复杂度
- 降低结果一致性维护成本
- 便于直接复用现有回测/优化任务体系

## 5.2 状态流修订建议

`StrategyUnit.run_status` 建议统一为：

- `idle`
- `queued`
- `running`
- `completed`
- `failed`
- `cancelled`

`Workspace.status` 不建议单独维护复杂状态机，建议通过子单元聚合计算：

- 任一单元 `running/queued` → 工作区 `running`
- 全部 `completed` → 工作区 `completed`
- 存在 `failed` 且无运行中 → 工作区 `error`
- 其他 → `idle`

## 5.3 接口边界修订建议

工作区 API 只负责：

- 工作区与策略单元管理
- 将策略单元映射为现有 `BacktestRequest`
- 调用现有回测/优化服务
- 记录工作区维度的引用关系与状态

不负责：

- 重写回测指标计算逻辑
- 重新实现优化计算引擎

## 阶段3：运行编排与状态闭环（预计 3 天）

3. 调用现有 `BacktestService.run_backtest()`
4. 建立单元与 `backtest_task_id` 的关联
5. 任务完成后回填单元状态、运行次数、最近耗时、关键指标快照

- 优先采用 **轮询可用、WebSocket增强** 的方案

### 验收标准

## 9. 风险与应对

## 9.1 范围风险

- **风险**：按钮过多、截图要求过细，导致前端大量时间消耗在“像不像”而不是“能不能用”  
- **应对**：按 P0 / P1 / P2 分级，先闭环主链路

## 9.2 数据冗余风险

- **风险**：重复存储回测/优化结果，后续一致性难维护  
- **应对**：优先做任务引用 + 关键指标快照，不做全量复制

## 9.3 并行资源风险

- **风险**：并行回测/优化可能压垮本地机器或服务实例  
- **应对**：后端增加并发上限与队列控制

## 9.4 指标一致性风险

- **风险**：TBQuant 指标口径多，短期难完全对齐  
- **应对**：本迭代先定义“基础指标集”，高级指标后补

## 9.5 WebSocket 复杂度风险

- **风险**：工作区维度聚合推送比单任务推送复杂  
- **应对**：先保证轮询闭环，再增强 WebSocket
