## 2. 二次验收总体结论

**结论：本次修复后，迭代124整体完成度较上一轮明显提升，但仍不能判定为“完全开发完成”。建议结论为：`大部分主链路已补齐，但仍存在若干未闭环实现、配置未生效项和新发现 bug，需要继续修复后再做最终验收。`**

与上一轮相比，本次已经确认修复的内容包括：

- 工作区运行接口已从“阻塞直到回测完成”改为“提交任务后立即返回”
- `last_task_id` 已在提交后立即回写，停止任务逻辑比之前更可用

## 4.3 运行链路仍有一个新的关键缺陷

### 并行运行时，后台轮询更新是串行的

`run_units(parallel=True)` 虽然提交任务时用了：

- `asyncio.gather(*[_submit_single(u) for u in units])`

关键代码：

- `src/backend/app/services/workspace_service.py:431-435`

但是提交完成后，后台轮询函数 `_background_poll_units()` 是：

- `for unit_id, task_id in submitted:`
- 每次 `await self._poll_task_completion(...)`
- 等前一个轮询完成，再处理下一个

关键代码：

- `src/backend/app/services/workspace_service.py:887-952`

这会导致：

- 虽然多个任务是并行提交的
- 但完成状态更新是串行等待的

实际风险：

- 如果第一个任务很慢，第二个任务即使早已完成，也要等前一个轮询结束后才更新状态
- 并行任务的完成状态在前端上会滞后
- 后续 `last_run_time` 记录也可能失真

这属于本次复验发现的**新 bug**。

## 5. Bug 清单（基于二次验收）

## 5.1 高优先级 Bug

### Bug-1：并行运行状态后台更新是串行的，可能导致状态滞后和耗时统计失真

- 位置：`src/backend/app/services/workspace_service.py:887-952`
- 现象：`_background_poll_units()` 逐个 `await _poll_task_completion()`，而不是并行轮询
- 影响：
  - 并行提交的多个回测任务，状态更新会排队
  - 先完成的任务也可能迟迟不更新为 `completed`
  - `last_run_time` 统计可能不准确
- 修复建议：
  - 按 unit 启动多个后台 polling task
  - 或对 `submitted` 再次使用 `asyncio.gather()` 并行轮询

### Bug-2：`timeframe / timeframe_n / bar_count` 仍未进入真实回测执行配置

- 位置：
  - 映射存在：`src/backend/app/services/workspace_service.py:874-885`
  - 但消费缺失：`src/backend/app/services/backtest_service.py:334-353`
- 现象：回测请求对象有字段，但回测服务没有实际写入这些配置
- 影响：单元配置与真实回测执行仍可能不一致
- 修复建议：
  - 在 `BacktestService` 中把这些字段真正写入策略运行所依赖的配置文件或运行参数

### Bug-3：提交优化时固定 `n_workers=4`，忽略线程设置

- 位置：`src/frontend/src/components/workspace/OptimizationConfigDialog.vue:238-242`
- 现象：线程设置弹窗保存的 `n_workers` 不会被提交优化时使用
- 影响：用户配置与实际执行不一致
- 修复建议：
  - 从 `unit.optimization_config.n_workers` 读取并提交
  - 或在优化参数弹窗显式合并线程设置

### Bug-4：优化参数提交后会覆盖原 `optimization_config`，导致部分配置丢失

后端提交优化后直接执行：

- `unit.optimization_config = {"param_ranges": ..., "n_workers": ..., "task_id": ...}`

关键代码：

- `src/backend/app/services/workspace_service.py:567-573`

这会覆盖前端原先保存的：

- `objective`
- `max_display`
- `initial_cash_wan`
- `calculation_method`
- `param_layers`

影响：

- 用户刚保存的优化配置在提交后会部分丢失
- 列表页中依赖 `optimization_config.objective` 的显示也可能异常

修复建议：

- 后端更新 `optimization_config` 时应基于原配置 merge，而不是整体覆盖

### Bug-5：报告删除语义错误，可能误删底层运行指标快照

- 位置：`src/backend/app/services/workspace_service.py:847-861`
- 现象：删除报告实际上会清空所有单元 `metrics_snapshot`
- 影响：
  - 删除报告可能导致运行结果快照丢失
  - 影响报告之外的其他展示能力
- 修复建议：
  - 如果要删除的是“报告”，应引入独立报告对象
  - 如果当前只能清视图，应改名为“清空报告缓存/快照”，并提示强风险
