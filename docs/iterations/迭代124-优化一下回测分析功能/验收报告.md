## 2. 总体结论

**结论：迭代124目前不能判定为“完全开发完成”，建议判定为“部分完成，不能整体验收通过”。**

- **阶段1（工作区壳子与列表页）**：基本完成

## 3.3 阶段3：运行编排与状态闭环

**结论：不通过**

### 已完成项

- 运行/停止/状态查询 API 已存在
  - `src/backend/app/api/workspace_api.py:273-315`
- 前端已接入运行选中、并行运行、停止选中、轮询状态
  - `src/frontend/src/stores/workspace.ts:152-205`
  - `src/frontend/src/components/workspace/WorkspaceUnitsTab.vue:378-397`
- 回测完成后会尝试回填 `metrics_snapshot`
  - `src/backend/app/services/workspace_service.py:435-460`

### 关键问题

#### 1. 运行接口是“阻塞到任务完成后才返回”，不是真正的异步启动

`run_units()` 内部在提交回测后，直接调用 `_poll_task_completion()` 等待任务结束：

- 提交回测：`src/backend/app/services/workspace_service.py:417-419`
- 阻塞轮询完成：`src/backend/app/services/workspace_service.py:421-424`
- `_poll_task_completion()` 轮询实现：`src/backend/app/services/workspace_service.py:865-879`

这会导致：

- 前端点击“运行选中任务”后，请求会卡住直到回测完成
- 不符合实施计划中“启动后通过轮询/状态流查看进度”的设计目标
- 用户无法获得真正的“任务已提交、状态异步变化”的体验

#### 2. “停止选中任务”大概率无法在真正运行中生效

当前 `last_task_id` 是在任务完成轮询之后才写回：

- `src/backend/app/services/workspace_service.py:428-435`

但停止逻辑依赖 `last_task_id` 去调用取消：

- `src/backend/app/services/workspace_service.py:516-523`

这意味着在任务真正运行期间：

- 单元可能还没有可用的 `last_task_id`
- 停止接口即使被调用，也未必能取消正在执行的回测

因此“终止选中任务”这一核心功能目前**不可靠**。

#### 3. 策略单元配置没有完整映射到回测请求

`_build_backtest_request()` 只映射了：

- `strategy_id`
- `symbol`
- `start_date`
- `end_date`
- `initial_cash`
- `commission`
- `params`

代码位置：

- `src/backend/app/services/workspace_service.py:847-862`

而 `BacktestRequest` 本身也不包含 `timeframe`、`timeframe_n`、`bar_count` 等字段：

- `src/backend/app/schemas/backtest.py:22-31`

这意味着：

- 策略单元表格里的“周期”配置不会真正影响回测执行
- `bar个数` 等字段也没有进入运行链路
- 当前“工作区单元配置”和“实际回测执行参数”并未完全一致

#### 4. “剩余时间”只是前端估算展示，不是真实剩余时间

- 代码位置：`src/frontend/src/components/workspace/WorkspaceUnitsTab.vue:538-547`

当前只是拿 `last_run_time` 做简单估算，并没有真实进度基础，因此不能视为真正完成了“剩余时间正确显示”。

## 4. Bug 清单

## 4.1 高优先级 Bug

### Bug-1：运行接口阻塞到任务完成，导致工作区运行闭环不成立

- 位置：`src/backend/app/services/workspace_service.py:417-424, 865-879`
- 现象：点击运行后，接口不是立即返回任务信息，而是一直等待到回测结束
- 影响：
  - 前端无法正确表现“已提交 -> 排队 -> 运行中 -> 完成”
  - 也会影响停止操作与多任务调度
- 修复建议：
  - `run_units()` 只负责提交任务并立即返回 `task_id`
  - 状态更新由后台任务、轮询或事件推送完成

### Bug-2：停止任务能力不可靠，运行中很可能取消不了

- 位置：
  - 任务 ID 回填：`src/backend/app/services/workspace_service.py:428-435`
  - 取消逻辑：`src/backend/app/services/workspace_service.py:516-523`
- 现象：`last_task_id` 在完成后才写回，停止接口依赖 `last_task_id`
- 影响：用户点击“终止选中任务”时，大概率无法取消真实运行中的任务
- 修复建议：
  - 在提交回测任务后立即写回 `task_id`
  - 运行状态更新与任务 ID 回写要在任务启动阶段完成，而不是完成阶段

### Bug-3：策略单元周期等配置没有进入真实回测请求

- 位置：
  - 工作区映射：`src/backend/app/services/workspace_service.py:847-862`
  - 回测请求结构：`src/backend/app/schemas/backtest.py:22-31`
- 现象：`timeframe`、`timeframe_n`、`bar_count` 等没有映射到回测请求
- 影响：策略单元配置与真实回测执行不一致
- 修复建议：
  - 扩展回测请求或在回测服务配置生成阶段补充这些字段映射

### Bug-4：工作区内无法真正发起优化任务

- 位置：
  - API 已定义：`src/frontend/src/api/workspace.ts:99-101`
  - 但页面未调用
  - 配置弹窗只保存单元配置：`OptimizationConfigDialog.vue:174-186`
- 现象：用户只能保存优化参数，不能启动优化
- 影响：阶段4主链路未完成
- 修复建议：
  - 在 `WorkspaceUnitsTab` 或 `OptimizationConfigDialog` 增加“提交优化”动作
  - 组装 `UnitOptimizationRequest` 调用 `workspaceApi.submitOptimization()`

### Bug-5：组合报告缺少后端创建/删除/重算能力

- 位置：`src/backend/app/api/workspace_api.py:404-414`
- 现象：只有 `GET /report`，没有创建、删除、重算接口
- 影响：阶段5无法验收通过
- 修复建议：
  - 增加 `POST /report`、`DELETE /report`、`POST /report/recalculate` 等接口
  - 若当前不做持久化，也至少应完成“按配置重算”的服务接口
