## 4. 本轮新确认的关键问题

## 4.1 高优先级问题

### Bug-1：优化结果接口与前端读取字段不一致，优化结果页可能无法显示真实结果

后端优化结果响应由 `_build_results_response()` 构造，返回的是：

- `rows`
- `best`
- `total`
- `completed`
- `status`

关键代码：

- `src/backend/app/services/param_optimization_service.py:657-694`
- `src/backend/app/api/workspace_api.py:354-365`

但前端 `WorkspaceOptimizationTab.vue` 读取的是：

- `results.results`

关键代码：

- `src/frontend/src/components/workspace/WorkspaceOptimizationTab.vue:409-416`

这意味着：

- 即便后端已有结果，前端也可能把 `resultRows` 置为空
- 页面会出现“暂无优化结果”的错误判定
- 后续表格、折线图、面积图、最佳参数、保存结果等功能都会受影响

这是本轮最关键的新缺陷，足以导致阶段4不能通过。

### Bug-2：页面展示的“最佳参数”与实际应用的“最佳参数”不保证一致

当前存在三层不一致：

- 后端结果响应固定按 `annual_return` 倒序排序
  - `src/backend/app/services/param_optimization_service.py:669-673`
- 前端表格又会按本地 `sortKey / sortDir` 二次排序，默认是 `sharpe_ratio desc`
  - `src/frontend/src/components/workspace/WorkspaceOptimizationTab.vue:352-376`
- 点击“应用最佳参数”时，前端固定传 `result_index = 0`
  - `src/frontend/src/components/workspace/WorkspaceOptimizationTab.vue:584-599`
- 后端 `apply_best_params()` 则直接从原始 `db_task.results[0]` 取结果
  - `src/backend/app/services/workspace_service.py:660-696`

这会导致：

- 用户页面上看到的“第1条最佳结果”
- 后端真正应用的“第0条原始结果”

**不一定是同一条记录。**

进一步地，前端优化参数里设置的 `objective` 也没有进入结果排序与最佳参数应用逻辑，因此“优化目标”目前并没有真正主导“最佳参数”的选择。

### Bug-3：优化参数保存会覆盖线程配置，导致配置丢失

`OptimizationThreadDialog.vue` 是 merge 保存：

- `src/frontend/src/components/workspace/OptimizationThreadDialog.vue:66-74`

但 `OptimizationConfigDialog.vue` 在保存和提交优化前，发送的是覆盖式 `optimization_config`：

- `src/frontend/src/components/workspace/OptimizationConfigDialog.vue:181-188`
- `src/frontend/src/components/workspace/OptimizationConfigDialog.vue:226-235`

后端 `update_unit()` 也不会 merge JSON 字段：

- `src/backend/app/services/workspace_service.py:261-264`

结果是：

- 线程设置中的 `mode / timeout` 会被覆盖掉
- `n_workers` 的持久化也存在被覆盖风险
- 用户看到“已保存的线程配置”并不稳定
