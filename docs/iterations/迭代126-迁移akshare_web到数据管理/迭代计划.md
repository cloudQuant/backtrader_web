# 迭代126 - 迁移akshare_web到数据管理

## 1. 文档目标

本计划基于 `初始需求.md` 重构，目标不是简单把 `akshare_web` 的页面和代码整体复制进来，而是在 **不破坏 backtrader_web 现有数据管理能力** 的前提下，完成一轮可落地、可联调、可验收的功能迁移。

本计划重点解决原计划中的 5 个问题：

1. **范围定义不够清晰**
   原需求强调“五个模块功能保持不变”，但没有区分“功能等价迁移”和“代码/页面 1:1 复制”的边界。

2. **缺少对当前项目真实基线的校准**
   当前 `backtrader_web` 已经存在 `/data` 路由、`DataPage.vue`、`/api/v1/data/kline` 和基于 `main.py` lifespan 的启动流程，迁移方案必须与这些现实结构兼容。

3. **迁移对象与落点缺少映射关系**
   研发需要知道 `akshare_web` 中每个模型、接口、页面，最终要落到本项目哪个目录、哪个命名空间。

4. **验收口径偏粗**
   仅写“功能保持不变”不足以指导开发，必须明确：哪些能力必须闭环、哪些可以降级、哪些可以后续迭代补齐。

5. **缺少开发前置决策项**
   包括数据库命名、历史数据是否迁移、调度器初始化策略、权限模型适配、接口脚本目录组织等，都需要在编码前冻结。

## 2.0.1 用户与价值

本迭代面向两类用户：

- **数据工程师 / 管理员**：需要在一个后台中统一管理数据采集脚本、定时任务、执行记录、数据表和接口定义，避免维护两个独立后台。
- **量化研究员**：本迭代中**仅作旁观用户**，当前回测工作流**不受任何影响**。

核心价值：

1. **统一后台入口**：把 `akshare_web` 和 `backtrader_web` 合并为一个部署，减少运维成本。
2. **为后续迭代铺路**：本迭代完成后，akshare 的数据治理能力以标准化形式落户到 `backtrader_web`，**下一个迭代**可以在此基础上实现"回测 / 参数优化可选择 akshare 数据源"。

## 2.0.2 本迭代做与不做

| 类别 | 内容 |
|------|------|
| **做** | 5 个模块功能等价迁移、父子路由改造、调度器接入、**两库 MySQL 基建**（`backtrader_web` + `akshare_data`）、SQLite 业务数据迁移、`akshare_web` 管理库数据迁移、最小权限 |
| **复用（不重建）** | `akshare_data` MySQL 数据仓库及其中已有行情数据 |

## 3.3 数据库引擎差异与处理方案（关键）

### 3.3.1 目标状态：运行时只保留两个 MySQL 数据库

 本轮方案调整后，最终运行时只保留以下两个数据库：

| # | 数据库 | 引擎 | 用途 | 对应环境变量 |
|---|--------|------|------|--------------|
| 1 | `backtrader_web` 主库 | **MySQL** | 用户、回测任务、参数优化、现有业务表，以及迁移后的 akshare 管理表 | `DATABASE_URL` |
| 2 | `akshare_data` 数据仓库 | **MySQL** | akshare 脚本采集的行情时序数据（表动态创建） | `AKSHARE_DATA_DATABASE_URL` |

 这意味着：

 - 当前 SQLite 只作为**迁移源**，不再作为长期运行数据库
 - 原 `akshare_web` 管理库也只作为**迁移源**，迁移完成后不再作为运行时依赖
 - `akshare_data` 继续保留为独立数仓，不搬入主库

### 3.3.2 为什么从三库改成两库

 调整后的好处是：

 1. **统一主业务数据库**：认证、回测、参数优化、akshare 管理能力都落在 `backtrader_web` MySQL，运维与备份更简单
 2. **淘汰 SQLite 运行依赖**：避免本地文件库在并发、迁移、备份与环境一致性上的限制
 3. **保留数仓独立性**：`akshare_data` 仍专注存放大体量行情时序数据，避免主业务库膨胀
 4. **便于后续对接回测**：回测系统以后只需要从主库发现元数据，再去 `akshare_data` 读取数据

 ### 3.3.3 本迭代的数据迁移范围

 本迭代明确把数据迁移纳入 P0 范围：

 - 若 `backtrader_web` MySQL 数据库不存在，先创建数据库
 - 将当前 SQLite 中仍在使用的业务表结构与数据迁移到 `backtrader_web` MySQL
 - 将原 `akshare_web` 管理库中的管理类表结构与数据迁移到 `backtrader_web` MySQL
 - `akshare_data` 继续复用，不迁移其中的动态行情表
 - 完成迁移校验后，应用运行配置切换为 MySQL，不再依赖 SQLite

 ### 3.3.4 表命名策略（共享主库下重新冻结）

 因为 `backtrader_web` 与 akshare 管理数据会落在**同一个 MySQL 库**中，表命名必须重新冻结：

 | 策略项 | 结论 |
 |--------|------|
 | 主业务表 | 保持 `backtrader_web` 当前表名，不加前缀 |
 | akshare 管理表 | 统一增加 `ak_` 前缀，例如 `ak_data_scripts`、`ak_scheduled_tasks` |
 | akshare 遗留用户数据 | 不与主 `users` 表合并；若需保留，迁入 `ak_legacy_users` 或迁移日志归档 |
 | 数据仓库表名 | `akshare_data` 中保持动态创建，不预先规划 |

 这样做的原因：

 - 避免与主业务表冲突
 - 明确区分“主业务表”和“akshare 管理表”
 - 为后续维护、备份、统计提供清晰命名边界

 ### 3.3.5 用户模型与历史数据映射

 迁移完成后，新系统统一使用 `backtrader_web` 主库中的 `users` 表。

 处理策略：

 - 新建的 akshare 管理表中的 `user_id` / `operator_id` 统一引用主库 `users.id`
 - 历史 `akshare_web.users` 不作为新系统登录表继续使用
 - 历史任务和执行记录迁移时，优先按用户名/邮箱等规则映射到主库用户
 - 无法映射的历史记录做降级处理：写入默认迁移用户、空值，或在迁移报告中登记待人工处理项

 重点是：**迁移后运行时只认一套用户体系**。

 ### 3.3.6 建库、建表与迁移脚本策略

 本轮不再采用“只复用现有表结构”的策略，而是改为：

 - 用 `DATABASE_URL` 指向 `backtrader_web` MySQL
 - 通过 Alembic / 初始化脚本在 `backtrader_web` MySQL 中建立当前主业务 schema 与 akshare 管理表 schema
 - 使用一次性迁移脚本：
   - `SQLite -> backtrader_web MySQL`
   - `akshare_web -> backtrader_web MySQL`
 - 迁移脚本输出行数对账、失败明细和回滚说明
 - `AKSHARE_DATA_DATABASE_URL` 保持独立，应用运行时继续直接访问 `akshare_data`

 ### 3.3.7 切换原则

 数据切换建议分四步执行：

 1. 创建 `backtrader_web` MySQL 与目标 schema
 2. 迁移 SQLite 业务数据
 3. 迁移 `akshare_web` 管理库数据
 4. 应用切换到 MySQL 配置并做回归验证

 在最终验收前，SQLite 与旧 `akshare_web` 数据库可保留为只读回退源，但不再作为正式运行依赖。

## 3.4 迁移约束

 本次规划需遵守以下约束：

  1. **复用现有认证、路由、基础设施**，不新建独立应用。
  2. **保留当前 `/data` 入口与市场数据能力**，不能以迁移为名回退已有功能。
  3. **代码分层遵循项目现有规则**：`models / schemas / services / api` 分离。
  4. **业务能力等价优先于代码逐行复制**，允许为适配当前项目做必要重构。
  5. **SQLite 与 `akshare_web` 管理库数据迁移纳入首批开发范围**，不能留到上线后再补。
  6. **akshare 数据模型必须具备前向兼容能力**，为后续迭代对接回测数据源预留空间（详见总体方案 6.4）。

## 5.1 前端策略

将当前单页 `DataPage.vue` 改造为“数据管理容器页 + 子页面”结构：

```text
/data                     -> 重定向到 /data/market

/data/tables              -> 数据表列表
/data/tables/:id          -> 数据表详情
```

页面组织建议：

- 新增 `DataLayout.vue` 作为 `/data` 父页面
- 顶部导航以 `el-tabs` 或路由驱动的页签导航实现
- `DataMarketPage.vue` 承接当前 `DataPage.vue` 的已有内容
- akshare 迁移页面单独拆分，避免将所有逻辑堆回一个超大页面

## 5.2 后端策略

后端 API 继续归属 `/api/v1/data` 命名空间，但代码文件用 `akshare_` 前缀区分：

- `app/api/akshare_scripts.py`
- `app/api/akshare_tasks.py`
- `app/api/akshare_executions.py`
- `app/api/akshare_tables.py`
- `app/api/akshare_interfaces.py`

推荐 API 路径：

- `/api/v1/data/scripts`
- `/api/v1/data/tasks`
- `/api/v1/data/executions`
- `/api/v1/data/tables`
- `/api/v1/data/interfaces`

这样做的原因：

- 对前端调用方语义直观
- 与当前 `/api/v1/data/kline` 共属同一业务域
- 代码实现上可通过多个 router 共用 `/data` 前缀，避免把所有逻辑塞进单个 `data.py`

## 5.3 数据模型策略

由于迁移后 `backtrader_web` 主业务表与 akshare 管理表位于同一 MySQL 库中，akshare 管理表必须统一增加 `ak_` 前缀：

- `ak_data_scripts`
- `ak_data_tables`
- `ak_data_interfaces`
- `ak_interface_categories`
- `ak_interface_parameters`
- `ak_scheduled_tasks`
- `ak_task_executions`

说明：

- **代码模型名** 可以保持业务语义，如 `DataScript`、`ScheduledTask`
- **数据库表名** 使用 `ak_` 前缀，与主业务表共库时避免冲突
- 原 `akshare_web` 管理库的数据迁移到这些新表中，而不是继续依赖旧库原表

## 5.4 调度器策略

由于当前项目已经有 `main.py` lifespan 启动流程，本次建议：

- 在应用启动时初始化 akshare 调度器
- 在应用关闭时优雅停止调度器
- 调度器服务独立封装，不与现有回测/优化运行器耦合

首版优先保证：

- 任务新增后可注册到调度器
- 启停状态和数据库状态一致
- 手动触发会生成执行记录

## 7.1 权限矩阵（最小方案）

| 角色 | 市场数据 | 执行记录 | 数据接口 | 定时任务 | 数据表 | 接口管理 |
|------|----------|----------|----------|----------|--------|----------|
| 普通登录用户 | ✅ 读 | ✅ 读 | ✅ 读 | ✅ 读 | ✅ 读 | ❌ 不可见 |
| 管理员（`Role.ADMIN` 或等价管理员权限） | ✅ 全部 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ✅ 全部 |

实现要点：

- 当前项目 `User` 模型没有 `is_admin` 字段，且现有 `get_current_user` 返回 `TokenPayload`；因此本迭代必须补齐“加载当前数据库用户 + 管理员校验”依赖
- 首版不扩展复杂 RBAC，只复用现有角色/权限体系中的管理员能力作为数据管理写权限入口
- 前端按钮可见性与后端接口权限**双重校验**
- 所有写操作（CRUD、启停、手动执行、扫描注册、失败重试）必须校验管理员权限
- 执行失败、手动执行等敏感操作记录操作人 ID 与时间

## 8. 风险与应对

## 8.1 认证与用户模型差异

- 风险：`akshare_web` 与当前项目用户模型和权限边界不同。
- 应对：统一使用 `backtrader_web` 现有认证体系，任务/执行记录中的用户字段以当前用户表为准，并补齐“数据库用户加载 + 管理员判定”依赖，避免沿用文档里错误的 `is_admin` 假设。

## 8.2 调度器运行一致性风险

- 风险：APScheduler 与 API 进程生命周期耦合，重启后任务状态可能与数据库不一致。
- 应对：启动时做任务重建或状态校准；首版先支持单实例部署假设。

## 8.3 数据采集脚本依赖复杂度

- 风险：`akshare_web` 的 `data_fetch` 目录可能包含当前项目未安装或未启用的依赖。
- 应对：先做依赖盘点，再按脚本分批迁移，不要求所有脚本首版一次打满。

## 8.3.1 接口全量扫描性能与稳定性风险

- 风险：`interface_loader` 直接扫描整个 `akshare` 模块，若在应用启动时执行，可能带来明显启动耗时与不可控日志噪音。
- 应对：首版冻结为“显式初始化 / 刷新”命令，不在主应用启动路径里自动全量扫描。

## 8.4 页面体量膨胀风险

- 风险：若继续把所有功能堆进 `DataPage.vue`，后续维护成本会迅速升高。
- 应对：必须拆分为布局页 + 业务子页，避免单文件持续膨胀。

## 8.5 历史数据迁移风险

- 风险：SQLite 与 `akshare_web` 两路数据同时迁移，存在字段不一致、用户映射失败、脏数据与对账复杂度。
- 应对：迁移脚本必须提供行数对账、失败明细、重跑能力与回退说明，并把用户映射失败项单独输出。

## 11. 结论

本次迭代应被定义为：

**在保留现有市场数据能力的前提下，将 `akshare_web` 的数据治理能力整合进 `backtrader_web` 数据管理域的一次结构化迁移迭代。**

它的核心不是“照搬旧系统”，而是：

- 统一入口
- 保持业务能力等价
- 复用当前项目架构
- 为后续研发提供清晰的迁移边界、目录落点和验收标准

后续开发应以本计划和配套设计文档为准执行，避免边做边改、范围失控。
