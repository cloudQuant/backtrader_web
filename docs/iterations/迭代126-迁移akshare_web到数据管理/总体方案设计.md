# 迭代126 - 总体方案设计

## 1. 文档目标

本设计文档用于回答以下问题：

- 为什么本次迁移不能直接“整包复制” `akshare_web`
- 迁移后能力在 `backtrader_web` 中的放置位置是什么
- 前后端、数据库、调度器各自如何落地
- 哪些能力必须保留，哪些实现允许重构

---

## 2. 设计原则

## 2.1 统一入口，避免双系统并存

目标是把 `akshare_web` 的数据治理能力整合进 `backtrader_web`，而不是在同一套部署中维护两个后台入口。

因此：

- 侧边栏继续只有一个 `数据管理` 入口
- 进入后通过顶部导航切换各子模块
- 原有市场数据能力保留并成为子模块之一

## 2.2 业务等价优先，不追求代码逐行迁移

需求强调“五个功能保持不变”，这里应理解为：

- 保持业务语义与核心操作闭环不变
- 保持用户能完成原来同类工作
- 不要求后端文件、前端组件、状态管理方式完全一致

## 2.3 基于当前项目真实架构扩展

当前项目已存在：

- `src/frontend/src/views/DataPage.vue`
- `src/frontend/src/router/index.ts`
- `src/backend/app/api/data.py`
- `src/backend/app/main.py`
- `src/backend/app/models/`、`schemas/`、`services/` 的成熟分层

因此本次迁移必须沿用这些结构，而不是把 `akshare_web/app/*` 原样塞入。

---

## 3. 当前系统基线

## 3.1 前端基线

当前 `backtrader_web` 的 `数据管理` 是一个单页：

- 文件：`src/frontend/src/views/DataPage.vue`
- 路由：`/data`
- 内容：
  - 股票数据查询
  - 期货持仓查询

这说明后续迁移的正确方向是：

- 拆出布局页
- 把当前页面降为 `市场数据` 子页
- 再补充 akshare 的其余子页

## 3.2 后端基线

当前后端已有：

- `/api/v1/data/kline`
- `router.py` 中的 `/data` 可选路由注册
- `main.py` 中基于 lifespan 的启动与关闭流程

这说明：

- 新增 akshare API 最合适的挂载方式，是继续放在 `/api/v1/data/*`
- 调度器适合挂在 lifespan 初始化过程中
- 不适合额外再起一个独立 FastAPI 子应用

```text
                 │            backtrader_web FastAPI            │
                 └────────────────┬───────────────┬─────────────┘
                                  │               │
                      ┌───────────▼───────┐   ┌───▼────────────────┐

                      │ MySQL（主业务库） │   │ MySQL（数仓）      │

                      │                   │   │ data_database.py   │
                      │ - users           │   │ - <动态创建的      │
                      │ - 回测 / 优化     │   │   行情时序表>      │
                      │ - 市场数据相关表  │   │   （由 akshare     │
                      │ - ak_data_scripts │   │   脚本运行时建表）│
                      │ - ak_data_tables  │   │                    │
                      │ - ak_scheduled_   │   │                    │
                      │   tasks           │   │                    │
                      │ - ak_task_        │   │                    │
                      │   executions      │   │                    │
                      │ - ak_data_        │   │                    │
                      │   interfaces      │   │                    │
                      └───────────────────┘   └────────────────────┘

```

运行时两套 engine / session 完全独立：

- **主库 session**：用于现有业务与迁移后的 akshare 管理元数据
- **akshare 数仓库 session**：用于读写行情数据表（数据表预览、数据采集写入）
- **禁止跨库 JOIN**；需要组合的数据通过应用层聚合
- 迁移后 akshare 管理表中的用户字段统一引用主库 `users.id`

## 4.4 配置项

运行时环境变量（在 `.env.example` 中提供默认示例）：

```bash

AKSHARE_DATA_DATABASE_URL=mysql+aiomysql://root:password@127.0.0.1:3306/akshare_data

# 调度时区（沿用源项目默认）
AKSHARE_SCHEDULER_TIMEZONE=Asia/Shanghai

# 脚本扫描根目录
AKSHARE_SCRIPT_ROOT=app/data_fetch/scripts

# 接口初始化模式：manual / refresh
AKSHARE_INTERFACE_BOOTSTRAP_MODE=manual
```

迁移脚本配置建议额外提供：

```bash
# 迁移阶段读取旧 sqlite
LEGACY_SQLITE_DATABASE_URL=sqlite+aiosqlite:///./backtrader.db

# 迁移阶段读取旧 akshare_web 管理库
```

其中：

- `DATABASE_URL` 与 `AKSHARE_DATA_DATABASE_URL` 是**运行时必须项**
- `AKSHARE_SCHEDULER_TIMEZONE`、`AKSHARE_SCRIPT_ROOT`、`AKSHARE_INTERFACE_BOOTSTRAP_MODE` 是 akshare 数据治理模块的关键运行配置
- `LEGACY_*` 仅在迁移脚本执行阶段使用，不进入正式运行依赖

## 6. 数据模型设计原则

## 6.1 模型范围

迁移到 `backtrader_web` 主库中的核心 akshare 管理实体如下：

| 模型类名 | 表名（所在库：`backtrader_web`） |
|----------|----------------------------------|
| `DataScript` | `ak_data_scripts` |
| `DataTable` | `ak_data_tables` |
| `DataInterface` | `ak_data_interfaces` |
| `InterfaceCategory` | `ak_interface_categories` |
| `InterfaceParameter` | `ak_interface_parameters` |
| `ScheduledTask` | `ak_scheduled_tasks` |
| `TaskExecution` | `ak_task_executions` |

数据仓库 `akshare_data` 中的表**动态创建**，本迭代不预定义具体模型，仅提供通用的"表元数据查询"和"数据预览"能力。

## 6.2 数据库命名策略（已调整）

**本轮调整后，akshare 管理表统一使用 `ak_` 前缀进入 `backtrader_web` 主库。**

调整原因：

1. `backtrader_web` 现有业务表与 akshare 管理表迁移后会共处同一 MySQL 库
2. `users`、任务表、接口表等需要与主业务表清晰隔离，避免命名冲突
3. 统一 `ak_` 前缀后，迁移脚本、备份与运维排查都更直观

在 `backtrader_web` 主库中，akshare 管理表建议如下：

| 表名 | 位置 | 说明 |
|------|------|------|
| `ak_data_scripts` | `backtrader_web` | 脚本定义 |
| `ak_data_tables` | `backtrader_web` | 数据表元数据（记录 `akshare_data` 中有哪些表） |
| `ak_scheduled_tasks` | `backtrader_web` | 定时任务 |
| `ak_task_executions` | `backtrader_web` | 执行记录 |
| `ak_data_interfaces` | `backtrader_web` | 接口定义 |
| `ak_interface_categories` | `backtrader_web` | 接口分类 |
| `ak_interface_parameters` | `backtrader_web` | 接口参数定义 |
| `ak_legacy_users` | `backtrader_web` | 可选：归档旧 akshare 用户资料，不作为登录表 |

## 6.3 用户与权限适配

`akshare_web` 原系统有自己的 `users` 表，与本项目用户体系**不兼容**；迁移后正式运行只保留 `backtrader_web.users`。

处理方式：

1. **新代码统一读取和写入** `backtrader_web.users`
2. akshare 管理表中的 `user_id` / `operator_id` 在迁移后指向主库 `users.id`
3. 旧 `akshare_web.users` 不作为运行时登录表；如需保留，迁入 `ak_legacy_users` 或输出迁移归档
4. 历史任务和执行记录迁移时，优先按用户名 / 邮箱映射到主库用户
5. 无法映射的历史记录，写入默认迁移用户、空值或迁移异常清单，不阻塞主流程
6. 展示用户名时，统一从 `backtrader_web.users` 查询

### 6.3.1 当前项目权限基线与首版落地方式

本项目当前真实基线与文档原假设有两个关键差异：

1. `User` 模型没有 `is_admin` 字段，不能按 `is_admin=True` 落地权限
2. 现有 `app.api.deps.get_current_user` 返回的是 `TokenPayload`，不是带角色信息的数据库 `User` 实体

因此本迭代的首版落地方式必须调整为：

- 新增数据管理专用依赖：从 JWT `sub` 读取用户 ID，再到主库加载当前 `User`
- 管理员判定统一走 `Role.ADMIN` 或等价管理员权限，不新增第二套 `is_admin` 判定
- 所有写接口统一依赖该管理员校验；只读接口依赖已认证用户
- 前端仅做可见性优化，**不承担安全边界**

### 6.3.2 遗留数据的处理建议

本迭代后，旧 `akshare_web.users` 的处理建议是：

- 若业务需要追溯，可迁入 `ak_legacy_users` 归档表
- 若仅用于迁移映射，可输出 CSV / 日志作为迁移审计件
- 正式运行链路不再依赖旧用户表

### 6.3.3 用户 ID 类型差异（迁移实现约束）

还需明确一个实现约束：

- `akshare_web.users.id` 是整数主键
- `backtrader_web.users.id` 是 UUID 字符串

因此：

- `ak_scheduled_tasks.user_id`、`ak_task_executions.operator_id` 等字段必须按主库用户类型建模
- 迁移脚本必须显式做 “legacy int user id -> current uuid user id” 映射
- 无法映射项必须进入迁移报告，不能静默写错类型或跳过

## 6.4 前向兼容与数据字典（重要）

> 本迭代虽然不对接回测，但 akshare 采集的数据**最终**将作为回测与投研的**可选数据源之一**。
> 关键洞察：`ak_data_tables` 表本身就是**数据字典的雏形**，记录了数据仓库中每张行情表的元信息。

### 6.4.1 数据字典：`ak_data_tables` 的双重角色

`ak_data_tables` 同时承担两个角色：

1. **管理角色**（本迭代）：展示 akshare 采集了哪些表、什么时候更新的、状态如何
2. **数据字典角色**（下一个迭代）：回测引擎通过它发现可用数据、判断数据覆盖范围

因此本迭代必须保证：

- 每次数据采集成功后，`ak_data_tables` 记录的 `table_name`、`row_count`、`last_update_time`、`data_start_date`、`data_end_date`、`last_update_status` **必须准确**
- 记录失败时，`last_update_status=failed` 必须明确写入
- 数据表被人工删除时，`ak_data_tables` 中的记录也要同步清理

这是一个**DoD 级别的硬性要求**，不能放宽。

### 6.4.2 兼容迁移与前向兼容字段并存

本轮因为要在 `backtrader_web` 主库中**新建** akshare 管理表，而不是直接复用旧 `akshare_web` 原表，所以允许在兼容迁移的前提下补充少量前向兼容字段。

建议：

- `ak_data_tables` 在承接源字段基础上，补充 `symbol_raw`、`symbol_normalized`、`market`、`asset_type`、`metadata`
- `ak_data_interfaces` 补充 `extra_config`
- 迁移脚本对历史记录按“可推导则补齐、不可推导则留空”的原则处理

原因：

- 本轮本来就要创建 `ak_` 前缀新表并做数据搬迁
- 这些字段能降低下一个迭代再改表的概率
- 对历史记录允许空值，不会阻塞本轮迁移

### 6.4.3 时间字段规范

沿用源项目约定：

- 管理表的 `created_at` / `updated_at`：UTC 时间（源项目已这样存）
- `data_tables.last_update_time`：数据采集完成时刻
- `data_tables.data_start_date` / `data_end_date`：数据覆盖的业务日期范围

本迭代**不改**这些字段的类型或语义。

### 6.4.4 数据仓库表命名（由采集脚本决定）

`akshare_data` 数据库中的表名由 akshare 采集脚本决定，本迭代不约束。
只要求：

- 每张采集出来的表必须在 `data_tables` 中有对应元数据记录
- 表名保持可识别（例如 `stock_daily_600000`、`fund_daily_000001`）

下一个迭代对接回测时，再考虑是否需要统一命名规范。

### 6.4.5 本迭代不落实、留给后续的事

以下能力**不在本迭代做**：

- 品种注册表（`symbol_registry`）
- 回测数据读取 API
- 数据版本管理
- 数据质量日志
- 多数据源统一抽象
- 数据字典可视化浏览器

### 6.4.6 下一个迭代对接回测的"契约"

下一个迭代对接回测时，回测引擎会：

1. 从 `data_tables` 查询"有哪些表、时间覆盖范围"
2. 根据用户选择的品种/时间范围，构造 SQL 查询 `akshare_data` 中对应的表
3. 把查询结果转换为回测引擎需要的格式

因此本迭代的**关键交付**是：

- ✅ `data_tables` 元数据记录准确
- ✅ `akshare_data` 中的表可以被 SQL 查询
- ✅ 数据表 API 能返回"前 N 行预览"（这个能力是下一个迭代数据读取的基础）

只要满足以上三点，下一个迭代就能顺利对接。

## 7.2 为什么不建议全部并入 `data.py`

## 8. 调度器与运行时设计

## 8.1 目标

迁移后需要满足：

- 定时任务可在应用启动后自动恢复调度
- 启停操作与数据库状态一致
- 手动执行能产生执行记录
- 应用关闭时调度器可优雅释放资源

## 8.2 推荐实现方式

- `akshare_scheduler.py`：封装 APScheduler 实例
- `akshare_scheduler_service.py`：负责数据库任务与调度器任务同步
- 调度器默认时区显式读取 `AKSHARE_SCHEDULER_TIMEZONE`，避免隐式依赖服务器本地时区
- `akshare_interface_loader.py` 不在 `main.py` 启动流程中自动全量扫描 akshare；改为显式初始化命令或管理员操作触发
- `akshare_script_service.py` 依赖 `AKSHARE_SCRIPT_ROOT` 扫描最小脚本集，并在缺失时给出可观测错误
- 在 `main.py` 的 lifespan 中：
  - 启动阶段初始化调度器
  - 启动阶段从数据库重建启用状态任务
  - 关闭阶段停止调度器

## 8.3 风险提示

如果系统未来多实例部署，需要进一步设计：

- 调度器实例选主
- 分布式锁
- 任务幂等控制

首版可先以单实例部署为前提。

## 9. 前端交互设计原则

## 9.1 导航结构

- 数据表结构 + 数据预览

## 9.3 允许的重构

以下方面允许按当前项目风格调整：

- 布局方式
- 表单组件实现
- 状态管理方式
- 接口封装方式
- 组件拆分方式

## 9.4 权限与角色（本迭代最小方案）

| 角色 | 市场数据 | 数据接口 | 定时任务 | 执行记录 | 数据表 | 接口管理 |
|------|----------|----------|----------|----------|--------|----------|
| 普通登录用户 | ✅ 读 | ✅ 读 | ✅ 读 | ✅ 读 | ✅ 读 | ❌ 不可见 |
| 管理员 `Role.ADMIN` / 等价管理员权限 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ✅ 全部 | ✅ 全部 |

落地要点：

- **后端**：所有写操作（CRUD / 启停 / 手动执行 / 扫描注册 / 重试）接口必须校验管理员权限
- **前端**：非管理员不显示"接口管理"Tab；不显示创建 / 编辑 / 删除按钮
- **前后端双重校验**，不依赖前端隐藏作为权限控制
- **敏感操作**（手动执行任务、删除数据）记录操作人 ID 与时间戳
- **不做**：复杂 RBAC 重构、字段级权限、数据级权限（后续迭代再评估）

## 10.1 本迭代非目标

其中任何时候，都不能牺牲现有 `市场数据` 页面可用性。

## 11. 建议结论

本次迁移最合理的实施方式是：

- 保留 `数据管理` 作为统一入口
- 以父子路由承载多子模块
- 使用 `akshare_` 前缀组织新增后端代码
- 使用 `ak_` 前缀组织迁移后的 akshare 管理表
- **采用双 MySQL 运行方案**：`backtrader_web` + `akshare_data`
- 把 SQLite 与旧 `akshare_web` 管理库数据迁入 `backtrader_web` 主库，并在验收后停用 SQLite
- 基于当前项目的生命周期机制接入调度器
- 以业务等价为目标，而不是代码照搬
- **数据模型采用前向兼容设计**，为后续迭代对接回测数据源预留扩展空间

这套方案既能满足原需求，也能尽量控制后续维护成本。
