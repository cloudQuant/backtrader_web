### 背景

之前写过一个 akshare_web 的项目，主要是下载清洗 akshare 的数据，现在希望把这个项目整合进 backtrader_web 中，放到数据管理中。

### 资源

akshare_web 这个项目放在 `Documents/source_code/akshare_web` 中。

### 需求

1. 把 akshare_web 侧边栏中的 `数据接口`、`定时任务`、`执行记录`、`数据表`、`接口管理` 这几个相关的功能迁移到 backtrader_web 中的数据管理中，把原先的侧边栏按钮变成数据管理页面顶部的导航栏。
2. 希望 `数据接口`、`定时任务`、`执行记录`、`数据表`、`接口管理` 这几个功能保持不变。
3. 可以对当前项目的源代码进行相应的修改。

### 本迭代边界（产品澄清）

> 以下为产品经理基于初始需求进一步补充的范围说明，用于指导开发与验收。

1. **最终定位**：akshare 采集的数据未来将作为 `回测` 与 `投研分析` 的**可选数据源之一**（而不是唯一数据源），后续可能还会接入其他数据源。
2. **本迭代只做功能迁移**：5 个模块在新系统中可用，并落地到独立的 MySQL 存储，**暂不对接回测/参数优化引擎**。
3. **下一个迭代**再专门处理"让回测和参数优化可选择 akshare 数据源"的能力。
4. **不改现有市场数据页数据源**：`/api/v1/data/kline` 保持现状，不在本次迁移中改动。
5. **数据库引擎**：采用**两个 MySQL 数据库**方案
   - `backtrader_web`：作为新的主业务数据库，承载当前 SQLite 中的业务表，以及从 `akshare_web` 迁移过来的管理类表
   - `akshare_data`：继续作为 akshare 行情时序数据仓库
6. **数据库迁移纳入本迭代**：
   - 若 `backtrader_web` MySQL 数据库不存在，需要在本迭代创建
   - 需要把当前项目 SQLite 中仍在使用的业务表与数据迁移到 `backtrader_web` MySQL
   - 需要把原 `akshare_web` 管理库中的业务数据迁移到 `backtrader_web` MySQL
   - 迁移完成后，运行时不再依赖 SQLite
7. **表结构策略**：由于迁移后 `backtrader_web` 与 akshare 管理数据落在同一个 MySQL 库中，akshare 迁移过来的管理表需要采用明确命名空间策略（如 `ak_` 前缀）避免冲突。
