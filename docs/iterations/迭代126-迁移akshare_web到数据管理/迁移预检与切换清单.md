# 迭代126 - 迁移预检与切换清单

## 1. 文档目标

本清单用于把“文档已确认的迁移方案”落成**实际可执行步骤**，避免开发进入联调或切库阶段后再补关键决策。

适用范围：

- `SQLite -> backtrader_web MySQL`
- `akshare_web 管理库 -> backtrader_web MySQL`
- `backtrader_web` 运行时正式切换到 MySQL

---

## 2. 迁移前预检

### 2.1 环境与连接预检

- 已安装并可访问 MySQL
- `backtrader_web`、`akshare_data` 目标库所在实例可连接
- 本地 `.env` 已明确：
  - `DATABASE_URL`
  - `AKSHARE_DATA_DATABASE_URL`
  - `LEGACY_SQLITE_DATABASE_URL`
  - `LEGACY_AKSHARE_DATABASE_URL`
  - `AKSHARE_SCHEDULER_TIMEZONE`
  - `AKSHARE_SCRIPT_ROOT`
- `src/backend` 依赖已安装，包含 `aiomysql`

注意：

- MySQL root 密码只能通过本地环境变量、交互输入或受控脚本传入，**不得写入仓库文档或示例文件**
- 若需要创建数据库，可以使用本地已知 root 凭据，但执行后仍应把运行配置写回 `.env`，而不是把密码硬编码进脚本

### 2.2 源数据预检

- SQLite 文件存在且可读
- `akshare_web` 管理库可读
- 已确认 `akshare_data` 为复用目标，不做整库搬迁
- 已抽样检查以下关键表是否存在且有数据：
  - `users`
  - 当前主业务关键表
  - `data_scripts`
  - `scheduled_tasks`
  - `task_executions`
  - `data_tables`
  - `data_interfaces`
  - `interface_categories`
  - `interface_parameters`

### 2.3 结构差异预检

- 已确认 `backtrader_web.users.id` 为 UUID 字符串
- 已确认 `akshare_web.users.id` 为整数
- 已设计 legacy user -> current user 映射规则：
  - 优先用户名
  - 次选邮箱
  - 再次选人工映射表
  - 无法映射则进入失败清单

---

## 3. 备份与回退件

切库前必须形成以下回退件：

- SQLite 原文件备份
- `akshare_web` 管理库逻辑备份
- `backtrader_web` 目标库迁移前空库快照或建库脚本
- 本次迁移脚本版本号 / git 提交号
- 迁移日志输出目录

最低要求：

- 任一迁移失败后，可回到“源库仍保持原状”的状态
- 回退过程不依赖人工回忆，必须有清晰步骤

---

## 4. 执行顺序

推荐严格按以下顺序执行：

1. 创建 `backtrader_web` MySQL 数据库
2. 在 `backtrader_web` 中执行 Alembic / schema 初始化
3. 跑 `SQLite -> backtrader_web` 迁移 dry-run
4. 跑 `akshare_web -> backtrader_web` 迁移 dry-run
5. 修复 dry-run 中的映射失败、类型失败、唯一键冲突
6. 执行正式迁移
7. 对账关键表行数
8. 抽检用户映射、任务、执行记录、数据表元数据
9. 将应用 `DATABASE_URL` 切到 MySQL
10. 做应用启动验证与 `/data` 模块回归

---

## 5. 对账要求

### 5.1 SQLite -> 主库

至少对账：

- `users`
- 当前项目仍在使用的核心业务表
- 回测 / 参数优化 / Workspace 相关核心表

### 5.2 akshare_web -> 主库

至少对账：

- `data_scripts -> ak_data_scripts`
- `scheduled_tasks -> ak_scheduled_tasks`
- `task_executions -> ak_task_executions`
- `data_tables -> ak_data_tables`
- `data_interfaces -> ak_data_interfaces`
- `interface_categories -> ak_interface_categories`
- `interface_parameters -> ak_interface_parameters`

### 5.3 对账输出

迁移脚本至少输出：

- 源表行数
- 目标表行数
- 跳过行数
- 失败行数
- 失败原因摘要

---

## 6. 切库后验证

### 6.1 应用启动验证

- 后端在 `DATABASE_URL=mysql+.../backtrader_web` 下可正常启动
- 主库连接成功日志可见
- `akshare_data` 连接成功或降级日志可见
- 不依赖 SQLite 文件也能启动

### 6.2 功能验证

- 登录正常
- `/data` 默认跳转 `/data/market`
- 市场数据页功能零回归
- `/data/scripts` 能看到迁移后的脚本
- `/data/tasks` 能看到迁移后的任务
- `/data/executions` 能看到迁移后的执行记录
- `/data/tables` 能查看元数据与预览

### 6.3 管理能力验证

- 管理员权限可写
- 普通用户写操作返回 403
- 脚本扫描 / 接口初始化可显式触发
- 手动执行后生成新执行记录

---

## 7. 回退条件

出现以下任一情况，应停止切库并执行回退：

- 关键业务表对账不一致且原因未查清
- 用户映射失败比例超过预期阈值
- 应用切到 MySQL 后无法正常登录
- `/data/market` 现有功能回归失败
- 迁移后 akshare 管理数据出现明显关联错乱
