# 迭代116：开发环境校验与可复现治理

## 1. 迭代目标

本迭代聚焦于“开发环境和文档入口的一致性”，目标是把环境问题前置到启动前，而不是等到运行测试或启动服务时才暴露。

## 2. 背景

在迭代115的验收过程中，前端问题已经可以通过代码修复收口，但后端验收暴露了另一个更基础的问题：

1. 当前机器上的 `backtrader` 被导入为 namespace package
2. 导入后的模块缺少 `Analyzer`
3. 测试在导入应用阶段就失败，而不是在业务逻辑断言阶段失败

这说明项目需要一个显式的“开发环境健康检查”入口。

## 3. 本次交付

本迭代包含以下交付物：

1. 新增 `scripts/verify-dev-env.sh`
2. 将 README / 安装文档 / 开发文档统一到该脚本入口
3. 将后端安装方式统一为 `pip install -e ".[dev,backtrader]"`
4. 将前端安装方式统一为 `npm ci`
5. 为 `backtrader missing Analyzer` 增加专门的排查说明

## 4. 脚本检查项

`scripts/verify-dev-env.sh` 会检查：

1. `node` 是否存在
2. Node 主版本是否为 `20`
3. `python` 是否存在
4. Python 版本是否满足 `3.10+`
5. 是否存在 `.nvmrc`
6. 是否存在 `src/backend/pyproject.toml`
7. `backtrader` 是否可导入
8. 导入后的 `backtrader` 是否包含 `Analyzer`

## 5. 验收标准

以下条件满足时，可认为迭代116完成：

1. 新机器按 README 或安装文档操作时，会先执行环境校验脚本
2. 文档不再引导开发者使用旧的 `requirements.txt` 安装路径
3. 环境异常时，脚本能给出可读错误，而不是让开发者自己猜
4. 针对 `backtrader` 错误导入的排查步骤已经进入文档主路径

## 6. 建议使用方式

```bash
git clone <repo>
cd backtrader_web
./scripts/verify-dev-env.sh
```

如果校验失败，先修复环境，再安装依赖和运行测试。
