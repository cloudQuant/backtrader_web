# 迭代118：验收缺陷修复与交付闸门加固计划

## 1. 迭代目标

本迭代聚焦两类问题：

1. 收口迭代116、迭代117在验收中确认未完成或实现错误的部分
2. 增加最小但有效的交付闸门，避免“文档写了、代码提交了、启动却失败”的问题再次出现

本迭代不追求大规模重构，而是先把系统恢复到“可启动、可验证、文档与实现一致”的状态，再为后续架构演进打基础。

## 2. 背景与当前结论

基于本轮验收，当前项目存在以下明确问题：

1. `src/backend/app/main.py` 当前存在语法错误，`py_compile` 无法通过，属于阻断级问题
2. `scripts/verify-dev-env.sh` 虽然拆出了“阶段 1 / 阶段 2”提示，但仍是单次执行流程，文档也仍然要求在安装依赖前直接运行，和实现不一致
3. `docs/INSTALLATION.md`、`docs/DEVELOPMENT.md`、`README.en.md` 仍包含旧启动路径、旧数据库初始化路径和顺序错误的命令
4. `src/backend/app/db/session_provider.py` 和 `src/backend/app/db/sql_repository.py` 当前实现存在导入错误和事务语义错误，不能视为可用的 request-scoped session / unit-of-work 骨架
5. `src/backend/app/services/backtest_manager.py` 只把任务状态放进数据库，但实际执行仍由 Web 进程 `asyncio.create_task` 触发，取消逻辑仍依赖进程内状态，不能视为“任务执行模型完成解耦”
6. 数据库初始化虽然新增了显式脚本，但 `src/backend/app/db/database.py` 里的旧初始化入口和相关测试、文档还没有完全迁移
7. 对外文档和 OpenAPI 描述中仍然存在“100% complete”“Async Task Queue”“1200+ tests, 100% coverage”这类不可靠陈述
8. 当前 CI 缺少“后端入口可导入/可编译”的显式冒烟步骤，导致这类阻断问题可能在开发阶段被忽略

## 3. 本次迭代不包含的内容

以下事项不在本次范围内：

1. 全量引入 Celery、Dramatiq、RabbitMQ、Kafka 等完整任务平台
2. 一次性将全部 repository、service、API 迁移到 unit-of-work
3. 全量替换现有文档体系
4. 大规模 UI 重做

## 4. 故事拆解

### Story 118.1：修复后端启动阻断问题，并增加入口冒烟校验

**问题**

当前 `src/backend/app/main.py` 处于不可导入状态，属于阻断级缺陷。只要入口文件不可解析，其他 117 改造一律没有验收意义。

**涉及文件**

1. `src/backend/app/main.py`
2. `src/backend/tests/test_main.py`
3. `.github/workflows/ci.yml`

**实现要求**

1. 修复 `main.py` 的语法问题，去掉误入的重复文本块
2. 保留 Story 117.3 的目标，不要重新引入应用启动时的隐式建表/建管理员行为
3. 清理 OpenAPI 描述中的旧说法和夸大陈述
4. 新增最小后端入口冒烟校验，至少覆盖：
   - `python -m py_compile app/main.py`
   - `python -c "from app.main import app; print(app.title)"`
5. 将上述校验接入 CI 或本地标准自检脚本

**验收标准**

1. `src/backend/app/main.py` 可被 `py_compile`
2. `from app.main import app` 可成功导入
3. OpenAPI 描述不再出现明显失真的“100% 完成”类文案

**自测命令**

```bash
cd src/backend
python -m py_compile app/main.py
python -c "from app.main import app; print(app.title)"
pytest -q tests/test_main.py
```

**预计工时**

0.5 到 1 人天

### Story 118.2：把环境校验真正收口为“安装前 / 安装后”双入口

**问题**

当前 `scripts/verify-dev-env.sh` 只是同一个脚本里的两段输出，仍然会在一次执行中直接走到依赖检查；而文档仍然把它放在依赖安装前作为单一步骤调用，和预期流程不一致。

**涉及文件**

1. `scripts/verify-dev-env.sh`
2. `README.en.md`
3. `docs/INSTALLATION.md`
4. `docs/DEVELOPMENT.md`
5. `docs/BACKTRADER_IMPORT_TROUBLESHOOTING.md`
6. `docs/TROUBLESHOOTING.md`

**实现要求**

1. 将脚本改为显式模式，例如：
   - `./scripts/verify-dev-env.sh --preinstall`
   - `./scripts/verify-dev-env.sh --postinstall`
2. `--preinstall` 只检查系统级条件：
   - Node
   - Python
   - `.nvmrc`
   - `pyproject.toml`
   - `package.json`
3. `--postinstall` 再检查项目依赖：
   - `backtrader`
   - `backtrader.Analyzer`
   - `fastapi`
   - `sqlalchemy`
4. 所有文档统一到同一调用方式
5. 清理文档中旧的 `init_db()` 启动路径和重复/顺序错误的安装步骤

**验收标准**

1. 新机器可以只跑 `--preinstall` 并成功通过系统检查
2. 安装依赖后可以再跑 `--postinstall`
3. README、安装文档、开发文档中的校验步骤一致
4. `docs/INSTALLATION.md` 中不再出现重复段落和旧初始化命令

**自测命令**

```bash
./scripts/verify-dev-env.sh --preinstall
./scripts/verify-dev-env.sh --postinstall
rg -n "verify-dev-env|asyncio.run\\(init_db\\)|python -c \"from app.db.database import init_db" README.en.md docs
```

**预计工时**

0.5 到 1 人天

### Story 118.3：完成数据库初始化路径迁移，并清理旧入口

**问题**

117 已经引入 `src/backend/scripts/init_db.py`，但旧的 `init_db()` / `create_default_admin()` 仍在 `database.py` 中保留原语义，相关测试和文档也仍然围绕旧入口组织，导致运行契约不清晰。

**涉及文件**

1. `src/backend/app/db/database.py`
2. `src/backend/scripts/init_db.py`
3. `src/backend/tests/test_db.py`
4. `docs/DATABASE_INIT.md`
5. `docs/INSTALLATION.md`
6. `docs/DATABASE.md`

**实现要求**

1. 明确区分以下职责：
   - session/engine 提供
   - 建表
   - 创建默认管理员
2. 若保留 `database.py` 中的方法，需重命名或注释清楚其用途，避免与“应用启动初始化”混淆
3. 将测试从“应用启动自动初始化”迁移为“显式初始化行为”
4. 文档统一到 `src/backend/scripts/init_db.py`
5. 开发环境与生产环境初始化路径分开说明

**验收标准**

1. 应用启动路径中不再隐式创建表或管理员
2. 数据库初始化仅通过显式命令触发
3. `tests/test_db.py` 与当前初始化契约一致

**自测命令**

```bash
cd src/backend
pytest -q tests/test_db.py
python scripts/init_db.py --help
```

**预计工时**

1 到 1.5 人天

### Story 118.4：重做 request-scoped session / unit-of-work 骨架，使其真正可用

**问题**

当前 `session_provider.py` 存在错误导入，`sql_repository.py` 中外部 session 的生命周期处理也不正确。现在这部分更像“概念草稿”，还不是可运行实现。

**涉及文件**

1. `src/backend/app/db/session_provider.py`
2. `src/backend/app/db/sql_repository.py`
3. `src/backend/tests/test_db.py`
4. 任选一条真实业务链路

**实现要求**

1. 修复 `session_provider.py` 的导入和类型定义
2. 明确 repository 的两种模式：
   - 自管 session
   - 使用外部 session，但不自行关闭/提交外部 session
3. 只有 repository 自建 session 时才负责 commit/close
4. 至少在一条真实链路中接入该能力，例如：
   - 认证与 refresh token 写入链路
   - 回测任务创建与状态更新链路
5. 补单测，验证：
   - 同一事务内多次 repository 调用
   - 异常回滚
   - 外部 session 不被 repository 提前关闭

**验收标准**

1. `session_provider.py` 与 `sql_repository.py` 可正常导入
2. 至少一条业务链路使用同一事务边界
3. 新增测试能证明外部 session 模式真实生效

**自测命令**

```bash
cd src/backend
python -m py_compile app/db/session_provider.py app/db/sql_repository.py
pytest -q tests/test_db.py
ruff check app/db/session_provider.py app/db/sql_repository.py tests/test_db.py
```

**预计工时**

1.5 到 2.5 人天

### Story 118.5：把回测执行模型收口到“状态持久化 + 执行边界明确”的第一阶段

**问题**

当前 `BacktestExecutionManager` 只解决了“状态查询落库”，但没有解决“谁执行任务”的根问题。Web 进程仍通过 `asyncio.create_task` 发起长任务，取消逻辑也仍依赖进程内 `self._running_processes`。

**涉及文件**

1. `src/backend/app/services/backtest_manager.py`
2. `src/backend/app/services/backtest_service.py`
3. `src/backend/tests/test_backtest_service.py`
4. `src/backend/tests/test_backtest_api.py`
5. `docs/STORY_117_2_IMPLEMENTATION.md`

**实现要求**

1. 明确区分“任务状态管理”和“任务执行器”
2. 即使暂时不引入独立 worker，也要把执行器抽象单独封装
3. Web 层不能再把“数据库持久化”表述为“已经完成多实例解耦”
4. 取消逻辑需要明确说明能力边界：
   - 仅可取消当前进程持有的执行器
   - 或新增可恢复/可标记的任务状态机
5. 文档中将 117.2 的完成度表述改为真实状态
6. 为后续 worker 化预留入口，例如 CLI/runner 接口

**验收标准**

1. `BacktestExecutionManager` 的职责只限于状态管理
2. `BacktestService` 不再把当前实现描述成完整的多实例任务系统
3. 回测任务相关测试与当前真实运行模型一致

**自测命令**

```bash
cd src/backend
pytest -q tests/test_backtest_service.py tests/test_backtest_api.py
ruff check app/services/backtest_manager.py app/services/backtest_service.py
```

**预计工时**

1.5 到 3 人天

### Story 118.6：补齐交付闸门，防止“语法错误 / 文档漂移 / 入口失真”再次进入主线

**问题**

当前问题说明仅靠现有 lint/test 还不够，需要更明确的交付门禁。

**涉及文件**

1. `.github/workflows/ci.yml`
2. `scripts/verify-dev-env.sh`
3. 关键入口文档

**实现要求**

1. 在 CI 中增加后端入口编译/导入冒烟

   - 端口
   - 环境校验命令
   - 数据库初始化命令
3. 对外文档只保留可验证事实，移除：
   - `100% Complete`
   - `1200+ tests, 100% coverage`
   - `Async Task Queue` 这类与现状不一致的陈述
4. 把验收命令沉淀到一个统一 checklist 文档或脚本中

**验收标准**

1. 入口文件语法错误可在 CI 第一时间暴露
2. README / 安装文档 / 开发文档中的关键命令不再互相冲突

**预计工时**

## 5. 推荐执行顺序

建议按以下顺序推进：

1. `Story 118.1` 修复后端入口阻断
2. `Story 118.2` 完成环境校验和文档入口对齐
3. `Story 118.3` 收口数据库初始化路径
4. `Story 118.4` 修复 session / repository 骨架
5. `Story 118.5` 收口回测执行模型第一阶段
6. `Story 118.6` 把闸门补进 CI 和验收流程

这样安排的原因：

1. 先恢复基本可运行状态
2. 再统一入口和运行契约
3. 最后处理中期架构骨架和防回归闸门

## 6. 风险提示

1. `Story 118.4` 如果处理不当，容易引入 session 提前关闭、重复 commit 或脏事务问题
2. `Story 118.5` 涉及回测主链路，必须补回归测试，不能只靠文档说明
3. `Story 118.3` 修改数据库初始化路径时，要同步更新现有测试，否则会出现“代码新契约、测试旧契约”继续分裂

## 7. 迭代完成定义

满足以下条件时，可视为迭代118完成：

1. 后端主入口可编译、可导入、可启动
2. 安装前 / 安装后环境校验路径明确且文档一致
3. 数据库初始化只通过显式入口触发
4. request-scoped session / unit-of-work 骨架可运行，且至少落地一条真实事务链路
5. 回测执行模型的能力边界被真实实现和真实描述
6. CI 可以在第一时间拦截语法错误、导入错误和关键文档漂移

## 8. 总工时预估

约 5.5 到 10 人天，建议由 1 名中级工程师主导，必要时安排 1 名初级工程师协助文档、脚本和测试收口。
