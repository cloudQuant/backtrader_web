# 迭代117：运行时稳定性与交付一致性提升计划

## 1. 迭代目标

本迭代聚焦项目从“本地可开发”走向“稳定可交付”的关键缺口，目标是减少运行时隐患、统一系统事实来源，并为后续部署和多实例运行做准备。

## 2. 背景

在迭代115和迭代116之后，项目已经完成了文档基线、测试基建和环境自检的第一轮治理，但继续分析代码后，仍然有一些更高优先级的问题尚未处理：

1. 回测任务仍依赖进程内全局状态和 `asyncio.create_task`，多实例或重启场景不可靠。
2. 数据库初始化仍在应用启动阶段直接 `create_all` 并创建默认管理员，生产边界不清晰。
3. `SQLRepository` 仍以“每个方法自己开 session / 自己 commit”为主，跨仓储事务难以表达。
4. 文档、README 和系统自检入口已经改善，但“启动路径”和“真实运行契约”仍未完全统一。
5. 系统状态说明中仍包含部分“完成度式陈述”，与当前实际工程状态不完全一致。

## 3. 适用对象

- 中级工程师主导实现
- 初级工程师可参与子任务，如脚本、文档、测试补齐
- 需要 1 名熟悉 FastAPI / SQLAlchemy / 异步任务模型的 reviewer

## 4. 本次迭代不包含的内容

以下事项不在本次实现范围内：

1. 全量引入 Celery / Dramatiq / Kafka 等完整分布式任务平台
2. 一次性把所有 repository、service、API 全部重写
3. 大规模重做前端页面设计
4. 引入复杂的容器编排或云原生部署体系

## 5. 故事拆解

### Story 117.1：把环境校验从“单阶段脚本”改为“安装前 / 安装后”两阶段流程

**问题**

当前 `scripts/verify-dev-env.sh` 在文档中被放在安装前执行，但脚本本身要求当前 Python 环境已经可以导入 `backtrader`。这对一台刚 clone 项目的新机器并不成立。

**受影响位置**

1. `scripts/verify-dev-env.sh`
2. `README.en.md`
3. `docs/INSTALLATION.md`
4. `docs/DEVELOPMENT.md`

**改进方案**

将环境检查拆成两个阶段：

1. `preinstall`：仅检查系统级前置条件
   - `node`
   - Node 主版本
   - `python`
   - Python 主次版本
   - `.nvmrc`
   - `pyproject.toml`
2. `postinstall`：检查项目依赖是否真的可用
   - `backtrader` 导入
   - `backtrader.Analyzer`
   - 可选的 `fincore`
   - 可选的最小 import smoke test

**验收标准**

1. 一台未安装项目依赖的新机器可以先跑“安装前检查”
2. 安装完依赖后可以再跑“安装后检查”
3. 文档对两阶段检查的时机说明明确

**预计工时**

0.5 到 1 人天

### Story 117.2：将回测任务从进程内状态抽离到持久化任务模型

**问题**

当前回测任务仍依赖：

1. 进程内 `_running_tasks`
2. 进程内 `_running_processes`
3. 进程内 `_user_task_count`
4. `asyncio.create_task`

这意味着：

1. 服务重启后内存状态丢失
2. 多 worker / 多实例场景下状态不一致
3. 任务取消、恢复、统计都依赖当前进程活着

**受影响位置**

1. `src/backend/app/services/backtest_service.py`
2. `src/backend/app/models/backtest.py`
3. `src/backend/app/api/backtest*.py`
4. `src/backend/app/websocket_manager.py`

**改进方案**

先做“轻量可演进”的第一阶段，不直接引入重型队列：

1. 新增任务调度抽象层，例如 `BacktestExecutionManager`
2. 将任务状态以数据库为主、内存为辅
3. 明确“提交任务 / 领取任务 / 执行任务 / 完成任务 / 取消任务”的状态机
4. 预留 worker 进程入口，例如脚本或 CLI
5. Web 进程只做提交与查询，不承担长期执行

**验收标准**

1. Web 进程不再直接依赖 `_running_tasks` 作为唯一事实源
2. 任务状态在服务重启后仍可查询
3. 取消逻辑和并发统计不只依赖当前进程内字典

**预计工时**

2 到 4 人天

### Story 117.3：收紧数据库生命周期，停止在应用启动阶段隐式建表和创建管理员

**问题**

当前 `init_db()` 在应用启动时直接执行：

1. `Base.metadata.create_all`
2. 自动创建默认管理员

这会把“应用启动”和“基础设施初始化”混在一起，不利于生产部署，也让权限边界模糊。

**受影响位置**

1. `src/backend/app/db/database.py`
2. `src/backend/app/main.py`
3. `src/backend/.env.example`
4. 部署文档

**改进方案**

1. 将“建表/迁移”与“启动应用”解耦
2. 引入显式 bootstrap 命令
3. 默认管理员只允许通过显式初始化命令创建
4. 为后续 Alembic 引入留出接口

**验收标准**

1. 应用启动时不再隐式创建默认管理员
2. 基础设施初始化有明确命令入口
3. 文档中明确区分“开发便利行为”和“生产行为”

**预计工时**

1 到 2 人天

### Story 117.4：引入 request-scoped session / unit-of-work 雏形

**问题**

当前 `SQLRepository` 仍主要以“每个方法自己创建 session、自己 commit”为单位工作，这在简单 CRUD 下没问题，但跨多个仓储的业务流程难以保证同一事务边界。

**受影响位置**

1. `src/backend/app/db/sql_repository.py`
2. `src/backend/app/db/database.py`
3. 各 service 中跨多个 repository 的流程

**改进方案**

第一阶段不需要全量替换全部业务，只做骨架：

1. 提供可注入的 session provider
2. 允许 repository 接收外部 session
3. 为关键服务引入 unit-of-work 雏形
4. 选 1 到 2 条跨仓储链路先落地

**优先链路建议**

1. 认证与 refresh token 链路
2. 回测任务与回测结果写入链路

**验收标准**

1. 至少一条关键业务链路可在同一事务中完成
2. repository 不再强制每个方法都独立提交
3. 新旧调用方式可以共存，避免一次性大重构

**预计工时**

1.5 到 3 人天

---

### Story 117.5：清理系统状态说明和工程事实漂移

**问题**

当前部分文案仍带有“100% 完成”“1200+ tests, 100% coverage”“Async Task Queue”等陈述，但和仓库真实实现并不完全一致。这类问题不会立刻导致功能错误，但会误导开发、验收和部署。

**受影响位置**

1. `src/backend/app/main.py`
2. `README.en.md`
3. `docs/TECHNICAL_DOCS.md`
4. `docs/PROJECT_COMPLETION.md`
5. 其他对外入口文档

**改进方案**

1. 删除不准确的完成度宣称
2. 只保留可验证的事实
3. 对测试数量、覆盖率、任务模型等信息改成“按当前验证结果描述”
4. 为未来自动生成版本/构建信息预留位置

**验收标准**

1. 对外文档和 API 描述不再包含明显无法验证的“完成度宣称”
2. 运行模型描述与当前实现一致
3. 关键入口文档之间不互相矛盾

**预计工时**

0.5 到 1 人天

## 6. 推荐执行顺序

建议按以下顺序推进：

1. `Story 117.1` 两阶段环境校验
2. `Story 117.5` 工程事实清理
3. `Story 117.3` 数据库生命周期治理
4. `Story 117.4` request-scoped session / unit-of-work 雏形
5. `Story 117.2` 回测任务持久化改造

这样安排的原因：

1. 先统一环境和事实来源，减少后续改造噪音
2. 再收紧数据库生命周期和事务边界
3. 最后处理对运行时影响最大的任务执行模型

## 7. 风险提示

1. `Story 117.2` 和 `Story 117.4` 都涉及后端核心执行链路，必须补回归测试
2. `Story 117.3` 可能影响现有开发便利性，需要同步文档
3. 任何数据库初始化流程调整都要避免破坏现有 SQLite 开发体验

## 8. 迭代完成定义

满足以下条件时，可视为迭代117完成：

1. 安装前/安装后校验路径清晰可用
2. 系统状态描述与真实实现一致
3. 启动流程不再隐式做生产敏感动作
4. 至少一条关键业务链路拥有明确事务边界
5. 回测任务执行模型不再完全依赖单进程内存状态

## 9. 总工时预估

总计约 **5.5 到 11 人天**。

建议拆成多个 PR 分阶段合并，不要一次性把五个 story 混在一起。
